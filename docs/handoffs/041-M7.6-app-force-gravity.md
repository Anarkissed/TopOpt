### Data model (pure value type, headlessly tested) — `Sources/TopOptFlows/ForceModel.swift` (new)
The M7.6 force/gravity state layered over the M7.5 `SelectionModel` (which owns
group identity/faces/colour), keyed per-group on the group `UUID`. Faithful to
the prototype's `S`, `setGravity`, `mkAnchor`/`mkLoad`, `loadDir`, `fmtW`, the
weight scrub, and the `opt`/`hint` sync block:
- `gravity` — the tapped floor-face outward normal, normalized, stored in **model
  space** (D2); `gravityFace`; `phase` (setup/edit); `settleRotation` (a
  `simd_quatf` rotating gravity → world −Y). `enterGravitySetup()` = the chip's
  "Change" (re-enters setup, keeps the vector).
- `GroupKind` (`.pending` / `.anchor` / `.load(direction:weightKg:)`) with
  `makeAnchor` / `makeLoad` / `setDirection` / `setWeight` (clamped 0.1–500 kg) /
  `sync(groups:)` pruning removed groups. A fresh group is `.pending` (D3).
- `WeightUnit` (kg/lbs, storage always kgf, D5); `formattedWeight` (proto `fmtW`:
  <10 → 1 decimal, else rounded); `scrub(kg:byPoints:)` (0.05-per-point display-
  unit step, clamped).
- `directionVector` (gravity → −Y, push → −n, pull → n, D4); `forceNewtons`
  (kgf × 9.80665) and `loadForceVectorNewtons` (the vector M7.7 hands the core
  traction path, D7); `arrowTipAtApplicationPoint` (D6 into-face test).
- `anchorCount` / `loadCount` / `hasPending` / `totalLoadKg`, `canOptimize`
  (edit phase ∧ ≥1 anchor ∧ ≥1 load ∧ no pending) and `optimizeSummary` (the
  proto sub-label cascade), `panelKindLabel`, and `tint(for:)` (anchor green).

### Bridge wiring (calls M7.6-core's signatures) — `Sources/TopOptFlows/LoadCaseTagger.swift` (new)
Turns the declared load case into the M7.6-core bridge calls: every anchor/load
face is TAGGED (`tag_step_face`, anchor→Fixture / load→Load) **and** MASKED
(`mask_step_face(.frozenSolid, depth N)`) as a passive N-voxel shell so the
optimizer cannot remove the BC surface (D7 / MOD AC). Pending groups are rejected
up front (`.pendingGroup`) so a run never half-applies. Both bridge calls are
injected (default = real `TopOptKit.tagStepFace` / `TopOptKit.maskStepFace`), so
the mapping is spied headlessly. This does not run the solver and does not apply
the traction itself — `topopt::traction_loads` is not on the Swift bridge, so the
load force vector is carried forward for M7.7's background run (see NOT-done).

### Small tested helpers
- `ViewerMesh.faceNormal(_:)` (ViewerMesh.swift) — the area-weighted outward
  normal of a B-rep face, feeding gravity setup from a tapped face. Nil for a
  faceless (STL) mesh.
- `SelectionModel.clearActive()` (SelectionModel.swift) — deselect + drop the
  empty group, for committing an Anchor / "Change" gravity (proto `activeId=null`).

  the part does not visibly rotate onto a floor yet. Reduced-motion snap is

- **Did NOT draw in-scene force arrows (D6).** The convention
  (`arrowTipAtApplicationPoint`) and the world force vector are computed + tested,
  but nothing is rasterized in the stage. The weight/direction are shown on the
  bottom control cluster instead.
- **Did NOT float the chip/snap-row/weight pill at the projected group centroid**
  ("beside a selection"). They render in a fixed bottom-centre cluster for the
  active group. Faithful floating needs a live camera→screen projection seam from
  `MetalMeshView` into SwiftUI overlays, which does not exist (the MTKView
  self-drives its camera without SwiftUI layout updates). That seam + the arrows +
  the settle are the coherent renderer follow-up.
- **Did NOT tint anchor faces green in the 3D highlight.** The Selections panel
  and `ForceModel.tint(for:)` use anchor green, but `MetalMeshView`'s per-vertex
  tint still uses the group palette (M7.5). Wiring the force-model tint into the
  renderer is a small follow-up I left out to avoid disturbing the tested id-pass
  GPU glue.
- **Did NOT handle empty-space taps to drop the active pending group.** The viewer
  tap seam only fires on a face hit, so tap-in-space (proto: drop the pending
  group) is unavailable; the chip/snap-row ✕ button covers removal instead.
- **Did NOT run the LoadCaseTagger from Optimize / did NOT plumb traction.**
  Optimize toasts the ready load case; actually applying tag+mask on a background
  queue and feeding `traction_loads` (not on the Swift bridge) into a run is M7.7.
  `LoadCaseTagger` is the seam M7.7 calls; it is fully tested standalone here.
- **Did NOT remove the now-unused `WorkspaceTool` enum** (SelectionModel.swift).
  The tool *segment UI* is deleted (D1); the enum is inert and left in place to
  keep the diff minimal (no test references it). A later cleanup can delete it.
- **Did NOT persist gravity/roles/weights across imports or launches.** State is
