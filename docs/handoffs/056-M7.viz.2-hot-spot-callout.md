# Handoff 056 — M7.viz.2: hot-spot callout (the single worst-stress point)

## Task
M7.viz.2 "Hot-spot callout. Find and label the single highest-stress point on the
displayed variant with its actual value and its margin (value ÷ yield). A tappable
marker that frames the worst point, so the user is not left hunting the red zone.
Consumes the same von Mises field." (TRACK app / M7-VIZ block; the topmost unchecked
task there.)

## What I did
Tests first (rule 3), then implementation. All changes are `/app/`-only, in
`app/TopOptKit/Sources/TopOptFlows/` + its test target — no `/core/`,
`tests/fixtures/**`, `materials.json`, `rules.json`, `ci/`, `ARCHITECTURE.md`,
`ROADMAP.md`, or `Package.swift`. No new dependency (`simd`/`SwiftUI` ship with the
OS). No C++ changed. Builds directly on M7.viz.1 (merged): reuses the existing
`StressField` (von Mises field consumption), the yield the viz.1 scale is keyed to
(`ResultsModel.yieldStrengthMPa`), and the M7.8 stress overlay plumbing — nothing
re-implemented, no new bridge field, interface freeze respected.

Files modified:
- **`ResultsModel.swift`**
  - `StressField.peak() -> (valueMPa, index, position)?`: locates the field's argmax
    (the single highest-stress voxel), returning its value (MPa), its flat field
    index, and the world position of that voxel's **center**. The index→(i,j,k)
    decomposition inverts the core layout `index = (k*ny + j)*nx + i` exactly (plane
    = nx*ny; k = idx/plane; j = (idx%plane)/nx; i = (idx%plane)%nx), and the center is
    `origin + (i+0.5, j+0.5, k+0.5)*spacing` — the exact inverse of the existing
    `value(at:)` sampler. Returns nil for an empty field or an all-zero field (no
    stress → nothing to call out). Ties resolve to the lowest index (strict `>`).
  - `HotSpot` (new value type) + `ResultsModel.hotSpot`: the displayed (selected)
    variant's worst point — `valueMPa`, `margin` = **value ÷ yield** (the task's
    definition: fraction of the material limit reached at the worst point; 1.0 = at
    yield, > 1 = over), `position` (world center for the marker), `fieldIndex`
    (== the field argmax), and preformatted `valueLabel` ("24 MPa") / `marginLabel`
    ("44% of yield" / "at/above yield (N%)"). The yield is the SAME
    `yieldStrengthMPa` the viz.1 scale is keyed to. `hotSpot` is nil when the
    selected variant has no stress field, and it tracks the tab selection (each
    variant carries its own `vonMisesField`).
  - `hotSpotMarginLabel(fractionOfYield:hasYield:)`: below yield → "N% of yield"; at
    or above → the honest "at/above yield (N%)" (matching viz.1's "at/above yield"
    language); no material limit → plain "peak stress" (no %, since there is no yield
    to compare against).
- **`ResultsScreen.swift`**: a tappable hot-spot marker (device QA). It projects
  `hotSpot.position` to screen via the viewer's published `CameraProjection`
  (`onProjection` wired into the results `MetalMeshView`, the same M7.6 mechanism the
  workspace force-arrow overlays use), draws a ring that frames the worst point, and
  toggles a small callout with the value + margin on tap. Shown only while the stress
  overlay is on and the point projects in front of the camera; the ring/callout tint
  goes red when the peak is at/above yield. Pixels/placement are maintainer device QA
  (the M7 /app/ standard); the located point + value/margin arithmetic are headlessly
  tested on `ResultsModel`.

  worktree here had (a) a populated git-ignored `occt-frameworks.generated.json` and
  (b) a stale prebuilt `vendor/TopOptCore.xcframework`. Symptoms:
  1. The populated JSON made the **macOS** build try to link the iOS-only
     `vendor/occt-ios/*.xcframework` slices → "no library for this platform was found".
     The documented macOS-test path needs that JSON **absent** (OCCT-free manifest,

     `Package.swift` content (NOT on the JSON), I also had to bust the cache the way
     `build_occt_ios.sh` does: `rm -rf ~/Library/{Caches/,}org.swift.swiftpm` +
     `~/Library/Developer/Xcode/DerivedData/TopOptKit-*` + `touch Package.swift`.
  2. The stale core threw `simp_optimize: max_iterations must be >= 1` (the M7.mma.1
     `SimpUpdater updater` field shifted the `SimpOptions` layout — old lib + new
     headers). Fix: `./app/scripts/build_core.sh` to rebuild the xcframework against
     the worktree core. After both fixes the full suite is green.
  I left the worktree in the documented default (JSON absent, cache busted) so the
  maintainer's macOS test path is clean; `git status` shows only my three source files
  and `Package.swift` is UNMODIFIED (mtime bump only).
- **`hotSpot` / `peak()` are computed each access** (cheap: one O(N) scan of the field
  on `peak`). The marker reads `hotSpot` only while the stress overlay is on. No
  memoization needed; if a device profile shows a scrub-with-stress hitch, memoize on
  `selectedIndex` alongside the existing `stressTints` note in `ResultsScreen`.
- **`margin` here is value ÷ yield (higher = worse), NOT the safety-factor `margin`**
  used elsewhere (`ResultVariantVM.worstCaseMargin`, where larger = safer). This is the
  task's explicit definition for the callout ("its margin (value ÷ yield)"). Kept
  distinct + documented so the two are not conflated.
- **`peak()` decomposes the index assuming `values.count == nx*ny*nz`** (the core's
  invariant, same assumption `value(at:)` already makes). A short/ragged field would
  give a wrong (i,j,k); production always supplies the full grid-sized field.
