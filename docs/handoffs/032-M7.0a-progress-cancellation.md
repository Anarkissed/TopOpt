## What I did
Tests first (rule 3), then the two hooks and the driver forwarding. No CMake
changes, no new dependencies, no /app/ (M7.0a is explicitly "pure core").

### API — `core/include/topopt/simp.hpp`
- `SimpOptions` gains `progress`
  (`std::function<void(int iteration, double compliance, double change)>`)
  and `cancel` (`const std::atomic<bool>*`), both absent/null by default.
  Documented contract: `progress` fires once per COMPLETED OC iteration with
  the 1-based global iteration number (monotone across M6.3 projection
  stages), the start-of-iteration compliance and the design change — the same
  values recorded in `history`, so no extra computation exists on the path
  when it is absent (the "zero overhead" reading; the only added per-iteration
  cost is one null check each for callback and flag). `cancel` is polled once
  per OC iteration at its START (so a request is honoured before the next
  FEA solve).
- `SimpOptimizeResult` gains `cancelled`. A cancelled result is valid and
  self-consistent: the loop stops, the usual final consistency solve still
  runs, so `physical_density == filter(design)` (projected at the last
  stage's beta when projecting; mask pins applied in the masked overload) and
  `compliance == compliance(physical_density)` hold exactly as for a
  completed run; `converged` is false; `history` has `iterations` entries.
  Header includes gained `<atomic>` + `<functional>`.

### Loop — `core/src/simp/simp.cpp` (both `simp_optimize` overloads)
The identical two-line hook in each overload's stage loop: the cancel poll at
the top of the OC iteration (break inner loop, then break the stage loop),
and the `progress` invocation right after the history push. Nothing else in
the loop moved; with both hooks absent the iteration body is unchanged.

### Driver — `core/include/topopt/pipeline.hpp`, `core/src/simp/minimize_plastic.cpp`
- `MinimizePlasticOptions` gains `progress`
  (`std::function<void(std::size_t rung_index, std::size_t rung_count, int
  iteration)>`) and `cancel` (same pointer type). The driver OWNS the
  optimizer's hooks: `options.simp.progress` / `options.simp.cancel` are
  overridden per rung (documented in the header) — the wrapper forwards
  (rung index [0-based, ladder order], ladder size, iteration) and drops
  compliance/change (the task's payload is exactly the triple).
- `MinimizePlasticResult` gains `cancelled`. A cancel observed during a
  rung's optimization makes that rung the REJECTED terminal rung
  (`accepted == false`, its `optimization.cancelled` true), no later rung
  runs, `stopped_on_margin` stays false (a cancel is not a margin stop), and
  the assembled report still carries exactly the accepted prefix and
  schema-validates. The cancelled rung's per-rung analysis (stress solve, V3
  suite, settings line) is SKIPPED — the caller asked to abort, and the
  half-optimized field is not shipped; those fields stay default-constructed
  (documented in pipeline.hpp).

## What I did NOT do
- **Did NOT wire progress/cancel into the CLI (`run_job` / job schema) or
  `simp_optimize_variants`.** The M3.6 variants runner copies `SimpOptions`
  verbatim, so a caller-set callback fires per variant with per-variant
  iteration numbers and no variant context — unchanged behavior, documented
  nowhere as rung-aware. The CLI job schema is maintainer-owned (DECISIONS
  2026-07-09); a headless batch driver has no progress consumer today.
- **Did NOT compute the per-rung analysis (stress solve, V3, settings) for a
  cancelled rung.** Its `v3` / `report` fields stay default-constructed
  (pipeline.hpp documents this). Rationale: a cancel aborts the run; the §7
  "every optimized output" V3 mandate is about outputs the pipeline ships,
  and a cancelled rung ships nothing (it is excluded from the JobReport).
- **Did NOT add a way to resume a cancelled run** (no warm-start from a
  partial design). The cancelled result is valid and inspectable, not
  resumable.
- **Did NOT make the cancel instantaneous.** The flag is polled once per OC
  iteration (the task's granularity), so the longest wait is one FEA solve +
  one OC update, plus the final consistency solve a cancelled run still pays
  to return a valid result. On a 128³ app-scale problem that is seconds, not
  milliseconds — if the app needs a harder abort, that is a new task.
- **Did NOT preserve caller-set `options.simp.progress` / `options.simp.cancel`
  inside minimize_plastic** — the driver overrides both (documented). Callers
  use the new `MinimizePlasticOptions` fields.
- **Did NOT add thread-safety beyond the flag.** `progress` runs on the
  optimizing thread; marshalling to a UI thread is the app's job (M7.7).
- **Did NOT run the OCCT/lib3mf-gated tests beyond the suite runs shown**
  (`export_3mf` absent locally as always; CI is the source of truth for it).
  The changed headers are compiled only by Eigen-gated TUs (simp.cpp,
  minimize_plastic.cpp, run_job.cpp), so the dependency-light build is
  untouched by this change.
- Did NOT touch /app/, ARCHITECTURE.md, any fixture byte, materials.json,
  rules.json, CI config, or any existing test's assertions/tolerances (both
  test files are purely extended). ROADMAP edit = the single M7.0a box.

## Warnings for the next run
- **M7.0b (the next task) touches `MinimizePlasticVariant` again** (von Mises
  field, variant mesh, support volume, mass). Note the cancelled-rung carve-
  out added this run: a cancelled variant's `v3`/`report` are default-
  constructed — M7.0b's new fields should follow the same rule (don't compute
  visualization payloads for a rung the user aborted).
- **The driver overrides `options.simp.progress` / `options.simp.cancel`.**
  If a future task wants raw per-iteration compliance/change in the app, it
  must extend the `MinimizePlasticOptions.progress` payload (M7.7 may want
  this for a compliance sparkline) rather than setting `simp.progress`.
- **`initial_compliance` is 0.0 when cancelled before the first iteration**
  (nothing was solved to fill it; documented in simp.hpp). Consumers that
  divide by it must guard the pre-set-cancel case.
- **Cancel latency is one OC iteration + one consistency solve** (see "did
  NOT" above) — budget UI expectations accordingly (M7.7's Cancel button).
- **The callback must not throw** (documented). A throwing callback
  propagates out of `simp_optimize` mid-loop; nothing is corrupted (locals
  unwind), but the result is lost. The Swift bridge (M7.1) should catch on
  its side of the boundary.
- **`std::function` in SimpOptions makes it non-trivially-copyable** — it
  already carried `std::vector`s, so nothing new in kind, but the M7.1
  bridge should pass SimpOptions by reference across the C++ side and build
  the callback in C++ (a Swift closure boxed via
  `std::function` is the interop-supported route).
