### Fix 2 — self-weight gravity units in the bridge (this commit)
**Reported symptom:** the app shows "No printable variant found" and rejects all
3 l-bracket variants, while `topopt-cli` accepts all 3 (margins in the thousands)
with min-feature violations reported as WARNINGS. **Reported hypothesis:** the app
hard-rejects on `min_feature_violations > 0`.

**Actual root cause (reproduced, not assumed):** a bridge units bug, NOT a
min-feature gate. I ran the real bridge path on `l-bracket.step` at res 48 and got
`acceptedCount=0`, terminal rung `margin=0.00128` — a genuine *strength-margin*
rejection. The bridge's `run_minimize_plastic` set no gravity, so the core used
the default `opts.gravity = 9.81` against the g/cm³ material density, making the
self-weight body force ~1e6× too large (the CLI folds the g/cm³→t/mm³ 1e-9 factor
into gravity: `run_job.cpp` `options.gravity = magnitude_mm_s2 *
kGramPerCm3ToTonnePerMm3`). The app path already computed acceptance exactly like
the core (`accepted_count = report.variants.size()`, margin-only) and never gated
on min-feature — the reporter's hypothesis did not match the code. Both the CLI
and the app apply the SAME margin-only policy; the app was just fed a garbage load.

**Fix:** in `bridge.cpp`, set `opts.gravity = 9810.0 * 1e-9` and
`opts.gravity_direction = {0,0,-1}` (mirrors the CLI/demo self-weight). After the
fix the same repro returns `acceptedCount=3`, margins ~1201–1277×, all accepted,
with the thin-feature warnings present (780/578/1312 regions) — CLI parity (the
margins/counts differ in exact value from the CLI's because the bridge clamps the
min-x plane while the CLI clamps the two screw holes; both are far above the 1.5×
gate, so all 3 accept either way).

## Blocked — Bug 2: workspace/project state is not persisted (recommend its own task)

**Confirmed:** the workspace's entire working state lives as `@State` /
`@StateObject` INSIDE `WorkspacePlaceholder` (`selection: SelectionModel`,
`force: ForceModel`, `viewerMesh`, and the `run`), which SwiftUI destroys when the
view leaves the hierarchy. Navigating Home ⇄ workspace (or opening a recent)
rebuilds it empty, losing gravity, groups, roles, directions, weights, and the
imported model. `AppModel` (the app-lifetime `@StateObject`) only holds
navigation + the import draft + an in-memory recents list; it does not own the
working state.

**Why this is NOT part of the M7.7 run-screen fix and should be its own task:**
fixing it properly is a structural data-model change, not a RunScreen edit — it
moves ownership of SelectionModel/ForceModel/gravity/material/model-ref out of the
transient view and adds a persistence layer that does not exist. Folding it into
this commit would balloon the diff far past the run screen and mix concerns
(worker rule: stay in scope). I did NOT fake it (no per-field UserDefaults hack —
that would silently drop the mesh + groups and rot on any model change).

**Recommended split:**
- **M7.x-persist-a — in-memory ownership (fixes the reported data loss):**
  Introduce a `ProjectModel` (`@MainActor ObservableObject`) owning the per-project
  working state (imported model ref + mesh, material/process, gravity vector +
  face, `SelectionModel`, `ForceModel`, and ideally the `RunModel` so a background
  run also survives navigation — the deeper Bug 1). `AppModel` holds
  `@Published var currentProject: ProjectModel?`; `WorkspacePlaceholder` binds to
  it instead of owning `@State`. Move the "reset selection/force" out of
  `rebuildMesh()` (which fires on every appear) into project *creation* only, so
  re-appearing restores rather than clears. Headless tests: leave + return
  preserves groups/roles/weights/gravity; a NEW import starts clean. No disk I/O —
  smaller, and it alone fixes the navigation data-loss bug.
- **M7.x-persist-b — cross-launch disk persistence (follow-up):** make
  `ProjectModel` `Codable`, persist one file per project under Application Support,
  COPY the imported model file into app storage (the security-scoped picked URL is
  not durably accessible), make `recentProjects` a real persisted list that
  re-imports from the app's copy on open (this also fixes the pre-existing
  "recents are in-memory / `open()` doesn't re-import" limitation from handoff 041
  and Addendum 1), and reload on launch with a schema version for forward-compat.

**Recommendation:** do persist-a next (fixes the reported navigation data loss and
lets Bug 1's background run survive Home), then persist-b for launch-to-launch
durability. Both warrant their own tests + review; neither belongs in the M7.7
run-screen commit.

## Addendum 3: M7.x-persist-a — workspace state survives navigation (done)

Maintainer approved doing persist-a (the in-memory half of the Bug 2 split above).
Implemented on this branch. Pure `/app/` Swift; no `/core/`, bridge, fixtures, or
existing-assertion changes. Cross-launch disk persistence is still the separate
persist-b follow-up (not started).

### What I did
Hoisted the workspace's working state out of the transient `WorkspacePlaceholder`
into a reference type owned by `AppModel`, so it outlives navigation:
- **`ProjectModel.swift` (new)** — `@MainActor ObservableObject` owning the mesh
  (`viewerMesh`, built from the import at creation), the `SelectionModel` groups,
  the `ForceModel` load case, and the project's `RunModel` (one per project, so a
  run also survives leaving/returning to the workspace). Plus identity/metadata
  (`id`, `name`, `material`, `process`, `importedFile`, `importedMesh`). No
  `Codable` yet (that's persist-b).
- **`AppModel`** — now owns `@Published private(set) var project: ProjectModel?`
  plus `projectsById: [UUID: ProjectModel]` keyed by the `RecentProject.id`.
  `continueToWorkspace()` builds the `ProjectModel` from the import draft and keys
  it; `open(recent)` restores the exact same instance when it exists this launch
  (else opens an empty project for the recent's material — the persist-b seam);
  `backHome()` keeps everything. `makeRunRequest` now reads from `project`.
- **`WorkspacePlaceholder`** — takes `project: ProjectModel` (`@ObservedObject`)
  and forwards `selection` / `force` / `viewerMesh` / `run` to it via computed
  properties with a `nonmutating set`, so ALL ~90 existing call sites are
  unchanged (they mutate the ProjectModel, which republishes). Deleted the
  `rebuildMesh()` + `.task(id: meshID)` that reset selection/force on every appear
  (the reset now happens ONLY at project creation — a new import starts clean, a
  re-open restores). The top chrome reads `project.name` / `project.material`.
- **`RootView`** — passes `model.project` into the workspace.

### Why it fixes the bug
Before, `selection`/`force`/`run`/`viewerMesh` were `@State`/`@StateObject` inside
`WorkspacePlaceholder`; SwiftUI tore them down on navigation and `rebuildMesh()`
re-zeroed them on re-appear. Now they live in an `AppModel`-owned `ProjectModel`
that persists for the launch, and the recents grid restores the exact instance.
