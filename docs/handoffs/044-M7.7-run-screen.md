### Tested core — `Sources/TopOptFlows/RunModel.swift` (new)
The run-screen state machine, exercised headlessly:
- `RunRequest` — the inputs `minimize_plastic` needs (model path, material,
  materials/rules paths, resolution, project name).
- `RunProgress` (pure) — the M7.0a callback tuple (0-based `rung`, `rungCount`,
  `iteration`) with derived `fractionComplete` / `percent` / `stageLabel`. The
  bar ramps asymptotically WITHIN a rung (the core exposes no iteration ceiling)
  and steps cleanly at each rung boundary; it never reaches 100% until the run
  resolves. Confirmed against the core: `minimize_plastic.cpp` calls
  `progress(rung, rung_count, iteration)` with `rung` 0-based over the ladder.
- `RunFailure` — `.solver(diagnostic)` (the core threw, e.g. CG non-convergence /
  the M3.1 singular-system guard) and `.allRungsRejected` (`acceptedCount == 0`),
  each with design-sheet `title` + `message`.
- `RunPhase` — idle → running → succeeded / cancelled / failed.
- `RunModel` (`@MainActor ObservableObject`) — `start(_:)` runs the optimize on a
  background queue via an injected `RunScheduler`; the progress callback publishes
  a `RunProgress` on main and reads a thread-safe `CancelToken` directly for the
  keep-going decision (so Cancel needs no cross-thread hop). `cancel()`,
  `runInBackground()` (flags + notifier), `dismissFailure()`, `reset()`.
  Classifies the terminal outcome: cancelled outcome → `.cancelled`;
  `acceptedCount == 0` → `.failed(.allRungsRejected)`; a thrown `TopOptError` →
  `.failed(.solver(message))`; otherwise `.succeeded` (outcome retained for M7.8).
- Seams: `RunScheduler` (`GCDRunScheduler` prod / `SynchronousRunScheduler`
  tests), `RunNotifier` (`SilentRunNotifier` default, `LocalRunNotifier` on
  device — see below).

### On-device "Run in Background" — `LocalRunNotifier` (in RunModel.swift)
A `RunNotifier` that, when the user backgrounds a run, takes a
`UIApplication.beginBackgroundTask` execution assertion so the in-flight run keeps
going after the user leaves the app, and posts a `UserNotifications` local
notification on completion (releasing the assertion). All system objects are
touched lazily and on the main thread, so constructing it off-device / in a
preview is harmless. Guarded `#if canImport(UIKit)` / `canImport(UserNotifications)`
so the macOS package build compiles the same file.

### Fix 2 — self-weight gravity units in the bridge (this commit)
**Reported symptom:** the app shows "No printable variant found" and rejects all
3 l-bracket variants, while `topopt-cli` accepts all 3 (margins in the thousands)
with min-feature violations reported as WARNINGS. **Reported hypothesis:** the app
hard-rejects on `min_feature_violations > 0`.

**Actual root cause (reproduced, not assumed):** a bridge units bug, NOT a
min-feature gate. I ran the real bridge path on `l-bracket.step` at res 48 and got
`acceptedCount=0`, terminal rung `margin=0.00128` — a genuine *strength-margin*
rejection. The bridge's `run_minimize_plastic` set no gravity, so the core used
the default `opts.gravity = 9.81` against the g/cm³ material density, making the
self-weight body force ~1e6× too large (the CLI folds the g/cm³→t/mm³ 1e-9 factor
into gravity: `run_job.cpp` `options.gravity = magnitude_mm_s2 *
kGramPerCm3ToTonnePerMm3`). The app path already computed acceptance exactly like
the core (`accepted_count = report.variants.size()`, margin-only) and never gated
on min-feature — the reporter's hypothesis did not match the code. Both the CLI
and the app apply the SAME margin-only policy; the app was just fed a garbage load.

**Fix:** in `bridge.cpp`, set `opts.gravity = 9810.0 * 1e-9` and
`opts.gravity_direction = {0,0,-1}` (mirrors the CLI/demo self-weight). After the
fix the same repro returns `acceptedCount=3`, margins ~1201–1277×, all accepted,
with the thin-feature warnings present (780/578/1312 regions) — CLI parity (the
margins/counts differ in exact value from the CLI's because the bridge clamps the
min-x plane while the CLI clamps the two screw holes; both are far above the 1.5×
gate, so all 3 accept either way).
