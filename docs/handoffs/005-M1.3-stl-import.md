New public headers (the API the app/plugins see, per ARCHITECTURE §3):

- `core/include/topopt/stl.hpp`: `StlFormat`, `StlError`, `StlMesh`,

  watertight; throws with a diagnostic otherwise — the §5 pipeline entry
  "import -> watertight check"). STL I/O lives in the `io/` module per §3.

Implementation:
- `core/src/io/stl.cpp`: format detection (leading `solid` token AND a literal
  `facet` keyword ⇒ ASCII, else binary — a binary header may itself begin with
  "solid", so both conditions are required); an ASCII parser (token scan pulling
  `vertex x y z` triples); a binary parser (80-byte header + LE `uint32` count +
  50 bytes/triangle, size-validated against the declared count); and vertex
  welding by **exact** coordinate match (STL writes coincident vertices
  identically, so exact welding is correct and avoids an epsilon that could
  collapse distinct nearby vertices). LE decoding is done by explicit byte
  assembly so the parser is host-endianness independent.
- `core/src/mesh/mesh.cpp`: `check_watertight` counts triangle uses per
  undirected edge (boundary = used once, non-manifold = used >2×; watertight =
  non-empty AND zero of each); `signed_volume` = divergence theorem
  (Σ dot(v0, cross(v1,v2)) / 6); `bounding_box` over vertices.

Tests — `core/tests/unit/test_stl.cpp`, registered as ctest `stl_import`
(self-contained `CHECK` harness matching `test_materials.cpp`; no test
framework, per §4). `CMakeLists.txt` injects `STL_FIXTURE_DIR` so the tests are
independent of ctest's working directory (same pattern as `MATERIALS_JSON_PATH`
from M1.2b). Assertions use the human-authored ground truth in
`core/tests/fixtures/stl/expected_values.json` as literals (fixture unchanged).

`core/CMakeLists.txt`: added `src/io/stl.cpp` + `src/mesh/mesh.cpp` to the
`topopt` library and a `test_stl` executable + `add_test`. CI runs the same

## Warnings for the next run
- `STL_FIXTURE_DIR` is a compile-time absolute path baked into `test_stl` at
  configure time (same trade-off as `MATERIALS_JSON_PATH`): if the repo is
  moved/renamed after configuring, reconfigure before running. A clean CI
  checkout always reconfigures.
- Format detection is heuristic (`solid` + `facet`). A pathological binary STL
  whose 80-byte header contains the exact ASCII bytes "solid"…"facet" would be
  misread as ASCII. No such file exists here; the size-validated binary parser
  is the backstop for normal binary files.
- `read_stl_file` parses without enforcing watertightness; `import_stl_file`
  enforces it. Downstream pipeline code (voxelizer, M1.5) should call
  `import_stl_file` (or check the report) — a raw `read_stl_file` result may be
  open.
- `check_watertight` on an empty mesh returns `watertight = false` (an empty
  mesh is not a closed solid), so `import_stl_file` rejects a 0-triangle STL.

## Blocked
**The committed `sphere_r10mm.stl` fixture's stated volume is not reproducible
from its own geometry at its own tolerance.** `expected_values.json` declares
for `sphere_r10mm.stl`:
- `mesh_volume_mm3 = 4047.044679978858`
- `volume_rel_tolerance = 1e-9`

Recomputing the divergence-theorem signed volume from the committed STL's
vertices gives **4047.04479204495** — a relative difference of **2.77e-8**,
about **28× the stated 1e-9 tolerance**.

Root cause (verified): the committed sphere vertices are stored at only ~7
significant figures (scientific notation, e.g. `8.506508e+00`); the max radius
error over the welded vertices is `|r - 10| = 5.4e-7`, i.e. ~5e-8 relative per
coordinate. A ~2.8e-8 relative volume error is exactly what that quantization
produces. The 16-digit `mesh_volume_mm3` was evidently computed from
higher-precision vertices (before the STL was written at 7 sig figs), so **no
correct importer can reproduce it from this file at 1e-9**.

This is not an importer bug: the cube fixture (exact-coordinate geometry)
reproduces its 1000 mm³ volume to 1e-9, which validates both `signed_volume`
and the parser. Only the sphere — because of its truncated on-disk vertices —
cannot match its stated value.

Per the rules I did not: edit the fixture, loosen its declared tolerance, or
loosen my own test tolerance to force a pass. Instead the sphere-volume pin is
omitted (with an explanatory comment in `test_stl.cpp` pointing here) and the
M1.3 box is left unchecked.

**Decision needed from the human maintainer (fixture discipline, AGENT_PROMPTS
§4):** reconcile the sphere fixture, by either
1. recomputing `mesh_volume_mm3` from the committed `sphere_r10mm.stl` vertices
   (→ 4047.04479204495…) so it matches the geometry at 1e-9, or
2. re-exporting `sphere_r10mm.stl` at higher vertex precision so the existing
   4047.044679978858 value is reproducible at 1e-9, or
3. setting an appropriate `volume_rel_tolerance` for the sphere consistent with
   its on-disk precision (≳ 1e-7).

Once the fixture is consistent, restoring the sphere-volume assertion in
`test_stl.cpp` and checking the M1.3 box is a one-line follow-up.
