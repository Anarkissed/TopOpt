# Handoff 059 — M7.rmin: physical mm-based density-filter radius

## Task
Make the SIMP density-filter radius a PHYSICAL length scale
(`rmin_voxels = min_feature_mm / grid.spacing`, floored at ~1.5 voxels) so the
minimum member thickness is resolution-independent, instead of the fixed voxel
count (2.5) that shrinks in mm as the grid is refined (diagnosis 060).

## Benchmark-collision investigation (the critical prerequisite) — CASE 1
The fix is BACKWARD-COMPATIBLE; V2 stays green with no fixture change. Two
independent reasons, both verified:

1. **Structural**: the Gate-V2 test (`core/tests/validation/test_gate_v2.cpp`)
   builds `SimpOptions` directly, sets `opt.filter_radius = 2.5` (kFilterRadius,
   line 417/480), and calls `topopt::simp_optimize(...)` (line 487). It NEVER
   routes through `minimize_plastic` or the bridge. The M7.rmin change lives on
   the `minimize_plastic` driver (new `min_feature_mm` field) and the bridge —
   neither of which Gate-V2 touches. So the locked benchmark radius is untouched
   by construction.
2. **Numerical**: the V2 benchmark grids are 1.0 mm-spaced (`const double h =
   1.0` in test_gate_v2). The chosen default `min_feature_mm = 2.5` yields
   `2.5 / 1.0 = 2.5` voxels EXACTLY at that spacing — numerically continuous
   with the M6.3-locked `rmin 2.5` (DECISIONS 2026-07-10). Asserted directly in
   the new test ("benchmark continuity"). No `tests/fixtures/**` change; V2's
   pinned reference values are still valid.

Result: proceeded under Case 1. `ctest` confirms gate_v2 still passes (67.25 s),
unchanged.

### Chosen default `min_feature_mm = 2.5 mm` and reasoning
- Nozzle-scale FDM minimum feature is ~1.5–3 mm; 2.5 mm sits in that band and,
  for a 0.4 mm nozzle, is ~6 extrusion widths — a reliably printable minimum
  strut.
- 2.5 mm reproduces the M6.3-locked 2.5-voxel radius exactly on the 1.0 mm
  benchmark geometry, keeping the locked decision continuous while scaling
  correctly for real parts (e.g. a 100 mm part at 64³ → spacing 1.5625 →
  rmin 1.6 voxels; at 128³ → spacing 0.781 → rmin 3.2 voxels; effective mm scale
  constant at 2.5 mm and the voxel radius GROWS with resolution, suppressing the
  thin-member proliferation the bug describes).

## What I did
Files modified (core) + the one permitted bridge line:
- `core/include/topopt/simp.hpp`: declared
  `double physical_filter_radius(double min_feature_mm, double spacing,
  double floor_voxels = 1.5)` beside `make_density_filter`, with the
  resolution-independence rationale and the floor contract documented.
- `core/src/simp/simp.cpp`: implemented it — validates all three args finite and
  > 0 (throws `std::invalid_argument`), returns
  `max(floor_voxels, min_feature_mm / spacing)`.
- `core/include/topopt/pipeline.hpp`: added `double min_feature_mm = 0.0` to
  `MinimizePlasticOptions`. `> 0` enables the physical scaling per rung; `0` (the
  default) keeps `simp.filter_radius` (voxel units) verbatim — so every existing
  direct caller/test is byte-identical.
- `core/src/simp/minimize_plastic.cpp`: validates `min_feature_mm` finite and
  `>= 0`; when `> 0`, sets each rung's `opt.filter_radius =
  physical_filter_radius(options.min_feature_mm, grid.spacing)` right after
  `SimpOptions opt = options.simp`.
- `app/TopOptKit/Sources/TopOptBridge/bridge.cpp`: the single permitted line —
  `opts.simp.filter_radius = 2.5;` → `opts.min_feature_mm = 2.5;` in
  `enable_projection`, with the comment updated to explain the physical scaling.
- `core/CMakeLists.txt`: registered `test_rmin` (Eigen-gated, like `test_simp`).
- `core/tests/unit/test_rmin.cpp`: NEW (tests-first).

Design choice: the conversion lives in `minimize_plastic` (not the bridge)
because the grid spacing is only known after voxelization — the bridge sets
options before the part is voxelized, so it can only carry the mm length scale;
core does the mm→voxel conversion where `grid.spacing` exists. `filter_radius`
stays the low-level voxel knob (unchanged meaning), so the interface freeze and
all direct callers hold.

## Warnings for the next run
- The physical radius is applied ONLY on the `minimize_plastic` path (the app/
  bridge path). Direct `simp_optimize` callers and every core test still use
  `SimpOptions.filter_radius` in VOXEL units — that field's meaning is
  unchanged. `min_feature_mm` is a driver-level override that computes it.
- The floor (1.5 voxels) means the mm length scale is NOT preserved on grids so
  coarse that `min_feature_mm / spacing < 1.5` (spacing > 1.667 mm at the 2.5 mm
  default). There the effective feature is larger than requested — intended
  (a mesh can't resolve a feature finer than its checkerboarding floor), and
  asserted in the test, but worth knowing if someone runs very coarse grids.
- `enable_projection` sets `min_feature_mm` on the `MinimizePlasticOptions` root,
  NOT on `opts.simp`. If a future edit reintroduces a per-run
  `opts.simp.filter_radius` assignment there, it will be silently overwritten
  per-rung whenever `min_feature_mm > 0`.
