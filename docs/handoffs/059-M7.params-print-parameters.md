# Handoff 059 — M7.params: print-parameters input screen

## Task
M7.params (M7-SHIP block, named assignment — not the topmost-unchecked default):
add the PRINT PARAMETERS modal sheet from `docs/design/PrintParams_TopOpt.dc.html`
to the current app. Capture wall loops / top shell layers / bottom shell layers /
infill % (+ layer height, captured-but-not-wired), FDM-sensible defaults, PERSIST
on the project (persist-b/c pattern), feed the values to the M5.1 settings engine

## What I did

All changes are under `/app/` (the bridge under `app/TopOptKit/Sources/TopOptBridge`
is the allowed additive-field location — it is not `/core/`). Nothing in `/core/`,
`/tests/`, or `docs/ROADMAP.md` was touched.

### 1. The captured value type — `PrintParams` (new)
`app/TopOptKit/Sources/TopOptFlows/PrintParams.swift`. A `Codable`, `Sendable`
value type holding `layerHeightMM`, `wallLoops`, `topLayers`, `bottomLayers`,
`infillPercent`, `infillPattern`.
- `PrintParams.fdmDefault` = 0.2 mm layers, 3 walls, 4 top / 4 bottom shells, 20 %
  gyroid infill (a typical desktop-FDM starting point). The sheet seeds from this
  and its "Default" button resets to it.
- `clamped()` enforces sane FDM bounds (layer 0.04–1.0 mm; walls 0–10; top/bottom
  0–15; infill 0–100 %; unknown pattern → default) — the numeric fields let a user
  type anything, so this is applied on sheet close.
- `slicerOverride: SlicerOverride` projects the user values onto the M5.1 engine's
  FDM field set (walls / top / bottom / infill / pattern). **Layer height is
  deliberately omitted** — the engine's `SlicerSettings` has no layer-height field,
  so it is captured but not wired (per the task). `SlicerOverride` is a small value
  type mirroring the core engine's FDM output so the "user overrides the
  recommendation" contract is expressible + testable in `/app/` without a `/core/`
  change. This is the seam a settings/report/export surface will consume.

### 2. Persistence on the project (persist-b/c pattern)
- `ProjectModel` gains `@Published var printParams: PrintParams = .fdmDefault`,
  written into `snapshot(savedAt:)` and read back in `init(restoring:)`.
- `ProjectSnapshot` gains `printParams: PrintParams?` — **optional**, exactly like
  the existing `minimizePlastic` / `quality` fields, so snapshots written before
  M7.params still decode (nil → `PrintParams.fdmDefault`). Schema version unchanged.
- No change to `OutcomeStore` (results blob) — print params ride in the JSON
  snapshot, which is where the lightweight working state lives.

### 3. The sheet UI + workspace entry
- `app/TopOptKit/Sources/TopOptFlows/PrintParamsSheet.swift` (new): the PRINT
  PARAMETERS sheet, styled with the DS glass tokens — title + subtitle, a "Default"
  reset, a 2-column grid (Layer height / Wall loops / Top shell layers / Bottom
  shell layers), a full-width Infill density row (accent %-readout + 0–100 step-5
  slider), an Infill pattern picker (6 patterns), and a primary Done. It edits the
  open project's `printParams` LIVE via key-path bindings.
- Presented on the workspace, matching the design's `openParams` affordance (the
  design shows print params as a workspace-reachable sheet, not a post-import gate;
  the current app flow is import → workspace, so this matches it). `RootView` renders
  it over the design's blurred scrim (mirroring the existing import overlay);
  `WorkspacePlaceholder` adds a "Print Parameters" pill paired with Optimize in the
  bottom bar (Optimize lives bottom-right in the current app, so the pair sits there
  rather than top-right).
- `AppModel` gains `printParamsSheetPresented`, `openPrintParams()` (no-op with no
  open project), and `closePrintParams()` (clamps + persists on Done and on
  scrim-dismiss — the sheet edits live, so both paths keep the edits).

### 4. Infill % threaded through the bridge (ADDITIVE)
The one field that feeds the optimizer (via the future M7.infill-margin ladder
knockdown) is plumbed app → bridge:
- `BridgeLoadCase` (in `app/.../TopOptBridge/include/TopOptBridge.hpp`) gains
  `int infill_percent = -1` (`< 0` = "no override; use the recommendation").
  **Additive + defaulted**: `run_minimize_plastic_loadcase`'s signature is
  unchanged (BridgeLoadCase is passed by const ref), so every existing caller keeps
  compiling untouched.
- `TopOptKit.minimizePlasticLoadCase` gains an `infillPercent: Int = -1` parameter
  that sets `lc.infill_percent`.
- `RunModel.RunRequest` gains `infillPercent: Int` (default `-1`), forwarded by
  `bridgeRunner`. It is part of the request identity, so an infill edit re-enables
  Optimize (correct — infill feeds the optimizer, unlike the slicer-only fields).
- `AppModel.makeRunRequest()` passes `project.printParams.infillPercent`.
- `bridge.cpp` receives `load_case.infill_percent` at the point `opts` is built and
  `(void)`-casts it with a comment marking the exact seam where M7.infill-margin
  will feed it to the core once the core options field exists (see Deferred, below).

## "Feed to the M5.1 settings engine as USER OVERRIDES" — how this landed
The app **does not currently surface the M5.1 recommended slicer settings anywhere**
— the bridge `OptimizeVariant` carries margins / mass / orientation / stress / mesh
but no walls/top/bottom/infill/pattern, and no app screen displays recommended
settings. So there is no in-app "recommended value" to override at runtime yet. What
M7.params delivers within scope:
- The user's `PrintParams` **are** the override, captured + persisted per project.
- `PrintParams.slicerOverride` expresses the override as the engine's FDM field set
  (dropping layer height) — the tested contract a settings/report/export consumer
  uses to replace the recommendation.
- `infillPercent` additionally reaches the bridge for the optimizer.
The actual merge of override-over-recommended happens where the recommendation is
produced/shown: core `recommend_settings` (out of `/app/` scope) and/or a future
settings-display/export surface (M7.9). See Deferred.

## Deferred to M7.infill-margin (NOT a blocker on this task)
This is the anticipated hand-off the task described ("so M7.infill-margin CAN
consume it"), not a Blocked stop — M7.params is complete as scoped.

**The last hop (bridge → core) awaits a `/core/` field owned by M7.infill-margin.**
`bridge.cpp` now receives `load_case.infill_percent` at the exact point `opts`
(`topopt::MinimizePlasticOptions`) is assembled in `run_minimize_plastic_loadcase`,
with a marked `(void)`-cast + comment. When M7.infill-margin adds an infill field to
`MinimizePlasticOptions` (or the equivalent knockdown input), set it there from
`load_case.infill_percent` (values 0–100; `< 0` means "no override"). No app change
is then required — the value already flows sheet → project → RunRequest → bridge.

**Two smaller follow-ups for whoever lands the core side:**
1. **STL / self-weight path.** Infill is threaded only through `BridgeLoadCase`
   (the STEP `run_minimize_plastic_loadcase` path). The self-weight
   `run_minimize_plastic` entry (STL, no BridgeLoadCase) would need the same additive
   defaulted parameter to knock down the ladder for faceless models. Left out here to
   avoid changing that signature for no consumer; add it alongside the core field.
2. **Surfacing the override / recommended settings.** The walls / top / bottom /
   pattern overrides (and `PrintParams.slicerOverride`) have no in-app consumer yet
   because the app never displays recommended slicer settings. When M7.9 (export /
   report) or a settings-display surface lands, have it read `project.printParams`
   (via `slicerOverride`) as the override of `recommend_settings`' output. The
   contract + tests already exist here.
