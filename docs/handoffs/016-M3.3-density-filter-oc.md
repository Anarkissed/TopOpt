## What I did
Added the SIMP density filter and the Optimality-Criteria design update (the two
operators ARCHITECTURE §4 pairs as the SIMP optimizer) to the existing `simp/`
module, and a validation test that drives them through a fixed-length SIMP loop
on a 2D-equivalent MBB slice, reproducing the classic 99-line MBB truss.

Public API — `core/include/topopt/simp.hpp` (extended; still Eigen-free):
- `struct DensityFilter` — a precomputed linear density filter over the grid's
  design (solid) voxels, stored in CSR form keyed by grid voxel index. Weights
  `H_ei = max(0, radius - dist(e,i))` with `dist` in **voxel units** (so `radius`
  is in voxels, ARCHITECTURE §4: "≥ 1.5 voxels"), independent of grid spacing.
  Empty voxels are excluded from every neighbourhood and map to 0.
  - `filter_density(x) -> xPhys`  (`xPhys_e = Σ_i H_ei x_i / Σ_i H_ei`, i.e.
    `xPhys = M x`).
  - `filter_sensitivity(dfdxphys) -> dfdx`  (the transpose `M^T`, so the chain
    rule `d/dx = M^T d/dxPhys` is exact; used to back-filter both the compliance
    and the volume sensitivities).
- `make_density_filter(grid, radius)` — build the filter (throws if radius ≤ 0).
- `oc_update(grid, filter, density, dcompliance, volume_fraction, move=0.2,
  density_min=1e-3)` — one classic OC design update. It filters the physical-
  space compliance sensitivity `dc/dxPhys` and the unit volume sensitivity
  through `filter` (chain rule), then advances the design by
  `x_new_e = clamp(x_e·√(−dc_e/(λ·dv_e)), [x_e−move, x_e+move] ∩ [density_min, 1])`,
  where the Lagrange multiplier λ is found by bisection so the **physical**
  (filtered) volume `Σ_e xPhys(x_new)_e = volume_fraction · n_design` (the
  constraint is enforced on the filtered density, matching the density-filter OC
  updater). Returns the grid-indexed updated design (Empty voxels 0).

Implementation — `core/src/simp/simp.cpp` (appended to the existing Eigen-gated
TU; no Eigen used by the filter/OC themselves, but they live beside
`simp_compliance` which drives the graded solver, so they stay in the same TU):
- `make_density_filter`: single pass over the grid building CSR rows
  (`row_start`, `neighbor`, `weight`, `weight_sum`); scans an integer offset cube
  of `ceil(radius)` and keeps only solid neighbours with `dist < radius`.
- `DensityFilter::filter_density` / `filter_sensitivity`: the forward average and
  its exact transpose.
- `oc_update`: bisection on λ. The root is bracketed robustly (physical volume is
  monotone-decreasing in λ; `l1 = 0`, `l2` doubled until the candidate volume
  drops to the target) so it does not depend on the sensitivity magnitude/scale.

Design choices and WHY:
- **Density filter, not sensitivity filter.** ARCHITECTURE §4 locks a "density
  filter (radius ≥ 1.5 voxels)". The classic 99-line uses a *sensitivity* filter,
  but the architecture wins (startup rule 1), so I implemented the density filter
  (the top88 / 88-line default) and the volume constraint is enforced on the
  physical density. Both reproduce the classic MBB pattern.
- **OC enforces the constraint on `xPhys`, so it takes the filter.** This makes
  the achieved *physical* volume hit the target to bisection tolerance (the test
  asserts it tightly), rather than only the raw design variables. The two halves
  are the SIMP optimizer per §4, so coupling them is appropriate.
- **`oc_update` back-filters `dc/dxPhys` and `dv` internally** (it already holds
  the filter), so the caller passes exactly what `simp_compliance` returns — one
  fewer place to get the chain rule wrong.
- **The M3.3 test runs a FIXED number of iterations (60), not to a convergence
  criterion.** ROADMAP splits "convergence criteria" and the fixture-based V2
  gate into M3.4; the `change` metric here oscillates ~0.04 forever due to
  single-voxel boundary flips (a known discrete-design behaviour), so a
  `change < tol` stop would be the wrong thing to assert at M3.3. The test
  asserts structural properties of the resulting design instead.
- `density_min = 1e-3` (the classic Sigmund floor and the M3.2 default). The
  penalized void stiffness is `~1e-9·E0`, above the M3.1 void-gate's `1e-12`
  relative threshold, so no design voxel is dropped; Jacobi-CG converges in
  ~370–700 iterations per solve on the 2D slice (no non-convergence throw).

Build — `core/CMakeLists.txt`: added `test_mbb` executable + `add_test(NAME
mbb_validation)` inside the existing `if(Eigen3_FOUND)` block (next to `simp`).
CI configures `-DTOPOPT_REQUIRE_DEPS=ON`, so it always builds/runs in CI with no
CI-file change. The filter/OC live in the already-Eigen-gated `simp.cpp`, so the

### How the MBB gate is set up and why it is trustworthy
`test_mbb` (new, `tests/validation/`) builds the classic MBB half-beam as a
60×20×1 slice (length x, height y, one voxel thick z): plane-strain BCs (fix
`u_z` everywhere), symmetry at x=0 (fix `u_x`), a roller at the bottom-right
corner (fix `u_y`), and a downward `-y` unit load at the top of the symmetry
plane. `E0=1`, ν=0.3, p=3, volfrac=0.5, filter radius 1.5, move 0.2. It runs 60
SIMP iterations of `filter → simp_compliance → oc_update` and asserts the
signature of the classic solution (values from the run below, thresholds chosen
with wide margin and clear physical meaning — no fixture, no tuned tolerance):
1. compliance drops well below the uniform start (`c_final < 0.5·c_init`; actual
   ratio **0.219**);
2. physical volume fraction matches target (`|v−0.5| < 5e-3`; actual **0.50000**);
3. design is substantially discrete (`Mnd < 0.5`; actual **0.268**, uniform=1.0);
4. genuine voids and solids formed (`min < 0.05` & `max > 0.95`; actual
   **0.001 / 1.000**);
5. mid-span I-section: solid top/bottom chords, void neutral axis, clear
   separation (`chord > 0.8`, `neutral < 0.2`, `chord−neutral > 0.6`; actual
   **1.000 / 0.002**);
6. material retained under the load and at the support (both **1.000**).

## Warnings for the next run
- **M3.4 must add the convergence criterion and the V2 fixture gate.** The
  `change` metric oscillates (~0.04) on discrete designs and will not fall below a
  small tolerance by itself; use a compliance-based / max-iteration criterion, or
  damp the OC step. The MBB *pattern* is stable from ~iteration 40 onward
  (compliance plateaus at ~200 by iter 30); the fixture compliance for V2 will be
  a converged value, so run more iterations than this test's 60 before comparing.
- **`oc_update` enforces the constraint on the FILTERED density and back-filters
  the sensitivities itself.** Pass it the raw `simp_compliance().dcompliance`
  (i.e. `dc/dxPhys`) and the *current design* `x` (not `xPhys`). Passing already-
  filtered sensitivities, or passing `xPhys` as the design, will double-filter and
  is wrong.
- **The design vector is `x`; the physical/analysis vector is `xPhys =
  filter.filter_density(x)`.** Feed `xPhys` to `simp_compliance` (and to any mesh
  extraction / property check), but keep iterating on `x`. Mixing them silently
  produces a blurrier or checkerboarded result.
- **CG cost scales with the SIMP stiffness contrast.** With `density_min=1e-3`,
  p=3, the contrast is `~1e9`; on the 2D slice Jacobi-CG still converges (~370–700
  iters at tol 1e-6). A 3D M3.4 run at higher resolution and full contrast may
  need a larger iteration cap or a looser optimization-phase tolerance to avoid a
  `fea_solve_cg` non-convergence throw — watch `CgInfo` per iteration.
- **`radius` is in VOXEL units, not model units.** `make_density_filter(grid,
  1.5)` is 1.5 voxels regardless of `grid.spacing`. The filter excludes Empty
  voxels from neighbourhoods, so on a real (holed) part the boundary weight sums
  differ from the interior — this is intended (design domain = solid voxels).
- **`mbb_validation` adds ~4 s** to the suite (60 CG solves on the 2D slice); full
  suite ~93 s locally, still well under the M2.3 "< 5 min CI budget".
