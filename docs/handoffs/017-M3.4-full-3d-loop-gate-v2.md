Public API — `core/include/topopt/simp.hpp` (extended; still Eigen-free):
- `struct SimpOptions` — loop controls: `volume_fraction`, `filter_radius`
  (voxel units, default 1.5), `move` (0.2), `max_iterations` (60), `change_tol`
  (0.01), `cg_tolerance` (1e-8, tight per the fixture's `solver_note`),
  `cg_max_iterations` (0 → Eigen default). `penalty`/`density_min` continue to
  live on `SimpParams` (not duplicated here).
- `struct SimpIteration` — one recorded step: `compliance` (of `xPhys` at the
  *start* of the step), `change` (max_e |x_new − x|), `volume_fraction`
  (achieved physical fraction after the step).
- `struct SimpOptimizeResult` — `design` (final x), `physical_density`
  (filter(x)), `compliance` (of `physical_density`), `initial_compliance`
  (uniform start), `volume_fraction` (achieved), `iterations`, `converged`,
  `history`.
- `SimpOptimizeResult simp_optimize(grid, params, bcs, loads, options)` — the
  classic min-compliance SIMP loop:
  1. `xPhys = filter.filter_density(x)`
  2. `(c, dc/dxPhys) = simp_compliance(xPhys)`
  3. `x = oc_update(x, dc/dxPhys, volfrac)`
  starting from `x = simp_uniform_density(grid, volfrac)` and stopping when
  `max_e |x_new − x| < change_tol` (→ `converged = true`) or after
  `max_iterations` (the iteration cap). The volume constraint is enforced on the
  *physical* (filtered) density by `oc_update` every step.

Implementation — `core/src/simp/simp.cpp` (appended; same Eigen-gated TU):
- `simp_optimize` validates its options (volume_fraction ∈ (0,1], move > 0,
  max_iterations ≥ 1, change_tol ≥ 0; `filter_radius` validated by
  `make_density_filter`), runs the loop above recording one `SimpIteration` per
  step, and **throws `std::runtime_error` if any penalized CG solve fails to
  converge** (so the caller never optimizes on a garbage field). It uses
  `grid.solid_count()` as the design-voxel count for the achieved-fraction
  metric.
- Reported final state is **self-consistent**: after the loop it sets
  `physical_density = filter(design)` and `compliance =
  simp_compliance(physical_density).compliance` via one final solve on the
  converged field, so `compliance == compliance(physical_density)` and
  `physical_density == filter(design)` exactly (a unit check asserts both). This
  is one OC step past the reference's printed iteration-60 value; the design has
  plateaued, so it lands within the 5% gate (see evidence — rel err ~2e-5).

Design choices and WHY:
- **Reused the M3.3 `oc_update` as-is.** Its OC exponent η is the `sqrt`
  (η = 0.5), `move = 0.2`, and it enforces the constraint on the *filtered*
  density by bisection — exactly the `formulation_locked.oc_update` in
  `benchmarks.json`. Bracketing differs (`l1=0`, `l2` doubled vs the fixture's
  `[0,1e9]`) but converges to the same root, so I did not touch it.
- **Two convergence criteria, both real.** `change_tol` (design-change stop) and
  `max_iterations` (hard cap). The Gate V2 test sets `change_tol = 0` to run the
  full 60 iterations like the reference (the `change` metric oscillates and does
  not fall below a small tol — handoff 016's warning); the unit test exercises
  the `change_tol` early-stop path separately (§8c).
- **`cg_tolerance` default 1e-8.** `benchmarks.json` `solver_note` requires
  `<= 1e-8` or the 60-iteration trajectory drifts; the 5% gate absorbs
  CG-vs-direct differences, not loose-tolerance drift. The Gate V2 test uses
  1e-8; the small unit run uses 1e-9.

### How the Gate V2 test is set up and why it is trustworthy
`test_gate_v2` (new, `tests/validation/`) reads the two benchmarks from the
immutable `tests/fixtures/benchmarks.json` (fixture dir injected via
`BENCH_FIXTURE_DIR`, same pattern as `test_fea`; a test-local minimal JSON reader
copied from `test_fea` and extended with boolean/null literals for the fixture's
`all_voxels_design: true`). Grid dims, material (E0, ν) and `volfrac` come from
the fixture; the BC/load *patterns* are constructed in code from the fixture's
textual descriptions, keyed by benchmark name:
- **`cantilever_24x8x8`:** clamp all 3 DOF on the x=0 node face; total Fz = −1
  spread equally over all (ny+1)(nz+1)=81 nodes of the x=nx face.
- **`clamped_beam_48x8x8`:** clamp all 3 DOF on *both* the x=0 and x=nx node
  faces; total Fz = −1 spread over the (ny+1)=9 nodes of the top center line
  (i=nx/2, k=nz, all j).
It runs `simp_optimize` (p=3, ρ_min=0.001, filter radius 1.5, move 0.2, 60
iterations, CG tol 1e-8, initial x=volfrac), thresholds the physical density at
0.5, and computes 6-connected components. Asserts exactly the fixture's declared
gates:
1. final compliance within `compliance_final_rel_tolerance` (0.05);
2. achieved physical volume fraction within
   `achieved_volume_fraction_abs_tolerance` (0.01) of the reference;
3. the **dominant** thresholded component spans the x=0 and x=nx faces (the
   fixture notes the cantilever's two 4-voxel tip islands are a 0.5-threshold
   artifact, so single-component enforcement is *not* asserted on the cantilever
   — that is V3/M3.5 on the cleaned mesh);
4. where the fixture pins `connected_components_at_threshold_0p5` (clamped beam:
   1), assert it exactly, and the clamped beam's dominant component also reaches
   the loaded top-center line;
5. plus a setup-correctness anchor: `compliance_first_iteration` (the uniform-
   start compliance) matches within the same 5% band (a wrong BC/load/material
   setup that happened to match the *converged* compliance would still miss the
   uniform-start value). The fixture lists it `informational_only`; asserting it
   only adds a guard, it does not relax anything.
The measured match is essentially exact (compliance rel err ~2e-5; thresholded
voxel counts 464 / 456-dominant / 3-components for the cantilever and 912 /
1-component for the clamped beam reproduce the fixture's informational values),
which independently confirms the whole chain (BC/load setup, filter, OC, graded
compliance) matches the reference formulation.

## Warnings for the next run
- **ROADMAP wording vs fixture:** M3.4 reads "3D MBB + cantilever", but
  `benchmarks.json` ships **`cantilever_24x8x8` + `clamped_beam_48x8x8`** (a
  doubly-clamped center-loaded beam — an MBB-class beam benchmark, not the
  half-symmetry MBB slice of the M3.3 `mbb_validation` test). The fixture is the
  immutable source of truth for what V2 asserts, so `test_gate_v2` runs exactly
  the two benchmarks the fixture defines. Not a blocker; flagged so the naming
  mismatch is not mistaken for a missing benchmark.
- **`simp_optimize` reports a self-consistent final state via one extra solve.**
  `result.compliance` is `compliance(filter(final_design))` (after the last OC
  update), i.e. one OC step past the reference's printed iteration-60 value. It
  matches `compliance_final` to ~2e-5 because the design has plateaued; do not
  expect exact equality with a reference that prints the pre-last-update value.
- **Gate V2 uses `change_tol = 0` to run the full 60 iterations.** If M3.5+
  drives `simp_optimize` with the default `change_tol = 0.01`, the loop will run
  to the cap anyway on these discrete designs (the `change` metric oscillates
  ~0.04). Do not assume it converges early; budget for `max_iterations` solves.
- **`gate_v2` adds ~13 s** (2 × 61 CG solves at tol 1e-8 on 24×8×8 and 48×8×8);
  full suite ~102 s locally, still well under the M2.3 "< 5 min CI budget". The
  clamped beam is the heavier of the two.
- **CG tolerance matters for the trajectory.** The fixture's `solver_note`
  requires CG tol ≤ 1e-8; `SimpOptions.cg_tolerance` defaults to 1e-8. Loosening
  it can drift the 60-iteration trajectory enough to miss the 5% gate even though
  each individual solve looks converged.
- **The test-local JSON reader now handles `true`/`false`/`null` literals** (the
  fixture uses `all_voxels_design: true`); the `test_fea` copy does not. If a
  future fixture reader is factored out, carry the literal support.
