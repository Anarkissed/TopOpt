# Handoff 039 — restore green CI: empty the committed iOS OCCT framework list

## Task
Assigned ROADMAP task was **M7.6 (Force arrows + weight)**. Per worker startup
rule 4, I ran the full test suite BEFORE changing anything and it was **RED**:
the app package failed to build on macOS. So this run's task changed to
**"restore green CI"** and M7.6 was NOT started (rule 4: "your entire run is now
'restore green CI', regardless of what ROADMAP says").

## Pre-flight (startup rule 4) — the failure that redirected the run
`./app/scripts/build_core.sh` then
`xcodebuild test -scheme TopOptKit-Package -destination 'platform=macOS'`:

```
While building for macOS, no library for this platform was found in
'.../vendor/occt-ios/TKernel.xcframework'.
  (… one per framework, 47 total …)
Testing cancelled because the build failed.
** TEST FAILED **
```

## Root cause (proven from the committed tree, not guessed)
`app/TopOptKit/Package.swift` line 36 had its **generated** iOS-OCCT framework
list committed with 47 entries:

```swift
let iosOCCTFrameworks: [String] = ["TKBO", "TKBRep", …, "TKernel"]   // 47
```

Two facts make that a broken committed state:

1. **`app/TopOptKit/vendor/` is git-ignored** (`git ls-files vendor/` → 0 files;
   `app/.gitignore`: "Vendored core + OCCT (regenerated by scripts/build_core.sh;
   never committed) TopOptKit/vendor/"). So on a fresh checkout the 47
   `binaryTarget(path: "vendor/occt-ios/<TK>.xcframework")` paths **do not
   exist** → package resolution fails.
2. Even with the frameworks present locally, the **`TopOptOCCT` product**
   (`Package.swift:139`) lists those 47 iOS-only binary targets **with no
   platform gating**, so a macOS build looks for a macOS slice in each and finds
   none → the "no library for this platform" build failure above.

Package.swift's own header documents the invariant this violated
(`Package.swift:26–29`): *"The committed value is EMPTY (OCCT-free) so a fresh
checkout and the macOS test path build with no iOS binaries declared; treat a
populated list like vendor/ (generated, not committed)."* The M7.1b handoff
(038) likewise says of this list: *"Empty by default … a tracked but regenerated
edit; do not commit it (it is to vendor/ what an object file is to source)."*

The populated list reached `main` via commit `14c234d "Update Package.swift"`
(merged in PR #38). It is a generated artifact committed by accident; the M7.1b
verification that passed did so against the **empty** list (handoff 038-M7.1b:
"macOS `swift test` (committed empty list): 75/75").

## What I did
One-line revert of the generated value to the documented empty state, exactly as
`build_occt_ios.sh` would emit on an OCCT-free tree. No other file touched.

- **`app/TopOptKit/Package.swift`** line 36:
  `let iosOCCTFrameworks: [String] = [ …47… ]` → `let iosOCCTFrameworks: [String] = []`

This is a build-manifest change only — no /core/, no Swift source, no bridge, no
fixture, no materials.json, no test, no CI config, no ARCHITECTURE.md. No new
dependency. `git diff --stat`: `1 file changed, 1 insertion(+), 1 deletion(-)`.

I did NOT platform-gate the `TopOptOCCT` product instead, because (a) the binary
targets reference git-ignored paths that are absent on a fresh checkout, so the
list *must* be empty when committed regardless of gating, and (b) gating M7.1b's
product wiring is a design change beyond a restore-green fix. Emptying the list is
the single change that both matches the documented invariant and unbreaks the
build; the maintainer repopulates it locally per-machine via the script.

## Warnings for the next run
- **To get STEP-on-iOS back, the maintainer regenerates the list locally — do NOT
  re-commit it.** With the frameworks already under `vendor/occt-ios/` (47), run
  `RELINK_ONLY=1 ./app/scripts/build_occt_ios.sh` then `./app/scripts/build_core.sh`;
  this rewrites `Package.swift` line 36 with the local framework list (a
  generated, uncommitted edit) and Xcode links + embeds them on iOS. Leave that
  edit out of any commit — it is machine-specific and breaks macOS/fresh-checkout
  builds if committed (this handoff's whole cause).
- **This regression can recur.** Any `git add -A` after running the OCCT script
  will re-stage the populated `Package.swift`. Before committing app work, check
  `git diff app/TopOptKit/Package.swift` and ensure `iosOCCTFrameworks` is `[]`.
  A committed guard (a manifest-shape assertion, or a `.gitattributes`/pre-commit
  hook keeping the generated block empty) would prevent recurrence but is a
  maintainer decision — see Blocked.
- **Run `./app/scripts/build_core.sh` before any build/test** (regenerates the
  git-ignored `vendor/TopOptCore.xcframework`). Unchanged from M7.1–M7.5.
- **Use `-scheme TopOptKit-Package` (or `swift test`)** for the test action; the
  per-product schemes are build-only.
- **M7.6 is next and is /app/-scoped.** Its testable core (weight→Newtons,
  arrows, roles/optimize-enable) lives in TopOptFlows; the SelectionModel already
  carries `GroupRole` and `WorkspaceTool.force` (inert) as M7.6 seams (handoff
  038-M7.5). The design's weight dialog / arrow chips / unit toggle / Optimize
  enablement are in `docs/design/TopOpt.dc.html` (workspace + WEIGHT DIALOG blocks).
