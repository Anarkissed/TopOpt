## What I did
Added a new `orient/` module (ARCHITECTURE §3) with the geometry-only pieces of
orientation handling: candidate build-direction generation and the support
proxy. No scoring/combining and no Gate V5 — those are M4.4. Nothing in FEA,
SIMP, mesh, voxel or IO changed; every prior test is bit-identical.

New public API — `core/include/topopt/orient.hpp` (Eigen-free, OCCT-free):
- `std::vector<Vec3> flat_face_normals(const TriangleMesh& mesh, double
  area_fraction = 0.05)`. Bins triangle normals by direction (grouped when within
  ~2.5°), accumulates facet area per group, and returns the area-weighted unit
  normal of each group whose total area ≥ `area_fraction` of the whole mesh area.
  Large planar faces are reported; curved surfaces (many small facets spread over
  many directions) are not. Throws `std::invalid_argument` on an empty mesh or
  `area_fraction` ∉ (0, 1].
- `std::vector<Vec3> orientation_candidates(const TriangleMesh& mesh)`. Returns
  the deduplicated union (directions within ~1.8° merged) of: the 26 lattice
  directions {-1,0,1}³ \ {0} normalized (the "coarse sphere sampling", which
  already contains the 6 axis-aligned directions), plus each significant
  flat-face normal of `mesh` and its opposite. All unit. Throws
  `std::invalid_argument` on an empty mesh (no faces to align to).
- `int support_overhang_voxels(const VoxelGrid& grid, const Vec3& build_dir)`.
  The support-volume proxy: counts solid voxels that would need support material
  — a solid voxel is supported when at least one of its 26 neighbours whose
  offset is within 45° of straight-down (−build_dir) is solid, OR the voxel is in
  the lowest build-direction slab (resting on the build plate). Unsupported solid
  voxels are the overhang count. `build_dir` need not be unit (normalized
  internally); throws `std::invalid_argument` on a zero-length `build_dir`.
- `constexpr double kOverhangAngleDeg = 45.0` — the self-supporting threshold.

Implementation — `core/src/orient/orient.cpp` (new TU in the always-built
library; pure C++/std, so it builds and its test runs in every configuration,

Design choices and WHY:
- **Overhang rule = 45° downward support cone with a plate-supported base band.**
  A solid voxel is supported if any neighbour within 45° of straight-down is
  solid (the standard AM self-supporting cone: directly-below + the four
  exactly-45° diagonals), OR it sits in the lowest build-coordinate slab (on the
  plate). This is what makes the metric usable for M4.4's hook: a flat extrusion
  printed flat is fully plate/below-supported (≈0 support) while the same part
  printed upright has an unsupported curved underside. A naive "count all
  downward-facing surfaces" proxy would instead count the whole flat base and
  *disfavour* the flat orientation — contradicting the V5 fixture — so the base
  band is essential, not cosmetic.
- **Exactly-45° counts as self-supporting**, so an overhang is strictly steeper
  than 45° (matches ROADMAP "> 45°"). The cone test uses `>= cos45 − 1e-9`.
- **Face-normal candidates from the mesh, not the STEP B-rep.** The Gate V5
  fixture is `hook.stl` (no B-rep), and the whole orientation path must run from
  a mesh + voxel grid. Basing `flat_face_normals` on triangle normals keeps the
  module OCCT-free and testable in every build, and "flat face" = a direction
  that accumulates a significant fraction of total area (so curved surfaces don't
  spam candidates).
- **New `orient/` module, nothing else touched.** M4.3 is geometry-only; the
  score that combines support proxy + layer-tension penalty (and reads
  `z_knockdown`) is M4.4. No speculative scoring API added here.

### How the tests are set up and why they are trustworthy
`orientation` (new, `tests/unit/test_orient.cpp`, always-built pure-C++/std,
public API only, 48 checks). Meshes/grids built in code:
- **flat_face_normals**: a correctly-wound box → exactly the 6 axis normals (each
  present, each unit); a non-cubic box → still 6; a large slanted quad with
  normal (2,0,1)/√5 plus a tiny +y sliver → only the slant clears the 5% area
  threshold (recovered as (2,0,1)/√5), and lowering `area_fraction` to 1e-6
  reports the sliver too (bite on the threshold); empty-mesh and `area_fraction`
  0 / >1 throw.
- **orientation_candidates**: an axis-aligned box → exactly 26 candidates (all
  flat-face normals already lattice dirs); the 6 axis dirs present; all unit; all
  pairwise > ~1.8° apart (dedup holds); the 26 partition as 6 face + 12 edge + 8
  corner (structural check on the sphere sample). A slanted flat face not aligned
  to any lattice dir → 26 + 2 = 28 (the slant and its opposite present) — proves
  the face path actually feeds candidates. Empty mesh throws.
- **support_overhang_voxels**: a solid 3×3×3 box → 0 overhangs for all 6 axis
  directions (base + below-support). A hand-computed post+arm shape (vertical post
  i=0,k=0..3 + horizontal arm i=0..4,k=3): built +z → **3** overhangs (the i=1
  arm voxel is supported by the post at exactly 45°, i=2,3,4 hang free — the count
  of 3 rather than 4 is the 45°-reach bite); built −z → **0** (the arm slab is the
  base, the post hangs supported from above); built +x → **0** (arm lies along the
  build direction). A floating 5×5 shelf on a central 1×1 pillar → **20** overhangs
  built +z (25 − pillar-top − 4 voxels the pillar supports at 45°) and **0** built
  −z (shelf becomes the base). Non-unit `build_dir` matches its unit form; a
  zero-length direction throws; a solid-free grid → 0.

## Warnings for the next run
- **"≤ 26 dirs" is the cap on the SPHERE SAMPLE, not on the whole candidate set.**
  The 26 lattice directions are the coarse sphere sample (and subsume the 6
  axis-aligned dirs); significant flat-face normals are added on top, so a part
  with slanted flat faces returns MORE than 26 candidates (the slant test asserts
  28). This is the grammatical reading of the ROADMAP line ("coarse sphere
  sampling (≤ 26 dirs)") and the only one compatible with also requiring
  face-normal-aligned candidates for arbitrarily-faceted parts. If M4.4 or review
  wants a hard total cap, that is a spec change (DECISIONS.md), not a bug here.
- **The support proxy exempts the lowest build-coordinate half-voxel slab as
  "on the plate".** This is what lets the flat-lying orientation score ≈0
  support. It means a voxel that floats disconnected but happens to sit in the
  lowest slab is treated as plate-supported (a proxy simplification), and for a
  tilted `build_dir` the base band is a slanted slab of 0.5·spacing thickness.
  Off-grid neighbours never count as support (only in-grid solids + the base
  band), so an overhang that reaches the exact bounding-box lateral wall with void
  below is counted correctly as unsupported.
- **The metric counts voxels, not support-material volume or contact area.** It is
  a proxy for ranking orientations (its purpose in M4.4), not a support-time
  estimate. It also does not model support columns' own footprint or accessibility.
- **`orientation_candidates` requires a non-empty mesh** (it validates via
  `flat_face_normals`), even though the 26 sphere dirs are geometry-independent —
  a part always has a mesh, so this is a misuse guard, not a limitation.
- **`support_overhang_voxels` reads only `grid.solid()` + geometry**, so it treats
  Load/Fixture/FrozenSolid the same as any solid voxel (all "solid"); it is
  orientation geometry, independent of the design mask.
- **`orientation` adds ~0.04 s** (all in-code geometry, no solves); full suite
  ~119 s locally, well under the M2.3 "< 5 min CI budget".
