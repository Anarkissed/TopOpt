## What I did
Added the Jacobi-preconditioned Conjugate Gradient iterative solver
(ARCHITECTURE §4's designated solver for voxel FEA) alongside the existing
direct solver, and verified it converges on a 64³ problem well inside the CI
time budget.

Public API — `core/include/topopt/fea.hpp` (extended; still **Eigen-free**):
- `struct CgInfo {bool converged; int iterations; double residual;}` — solver
  diagnostics (iteration count and final relative residual). Lets a test assert
  the solve actually iterated and reached tolerance, and lets later stages (M3
  SIMP, which calls this per design iteration) read convergence telemetry.
- `FeaSolution fea_solve_cg(grid, E, nu, bcs, loads, tolerance = 1e-8,
  max_iterations = 0, CgInfo* info = nullptr)` — same global linear-elastic
  system as `fea_solve`, solved with CG instead of a direct factorisation.
  `tolerance` is the CG relative-residual stop; `max_iterations <= 0` uses
  Eigen's default cap (2 × free-DOF count). Throws `std::runtime_error` if CG
  does not reach `tolerance` within the cap (guard against silently returning an
  unconverged field — CG on a consistent system converges to *a* answer
  regardless). BC/load range checks and material validation match `fea_solve`.

Implementation — `core/src/fea/assembly.cpp`:
- Extracted the assembly + BC reduction shared by both solvers into a private
  `assemble_reduced(...)` helper (anonymous namespace) that returns the reduced
  free-DOF system `K_ff u_f = rf`, the free-DOF list, the prescribed-displacement
  seed `up`, and `ndof`. This is the same code M2.2's `fea_solve` already ran
  inline (global sparse assembly over solid voxels, load scatter, Dirichlet
  reduction via the selection matrix `P` with `K_ff = P K Pᵀ` materialised
  un-chained — the Eigen-5 product quirk from handoff 011 is preserved). No
  numerical behaviour changed; `fea_solve` now calls the helper then applies its
  `SimplicialLDLT` factorisation and the unchanged LDLT-pivot singularity check.
  The `fea_assembly` (M2.2) test passing unchanged is the guard that the refactor
  is behaviour-preserving.
- `fea_solve_cg` calls the same helper, then solves with
  `Eigen::ConjugateGradient<SpMat, Eigen::Lower|Eigen::Upper,
  Eigen::DiagonalPreconditioner<double>>`. `DiagonalPreconditioner` is exactly
  `M = diag(K_ff)` — the Jacobi preconditioner named in the task. `K_ff` is
  symmetric so the solver reads both triangles. It sets the tolerance, the
  optional iteration cap, fills `CgInfo`, and throws on non-convergence.
- Added `#include <Eigen/IterativeLinearSolvers>` and `<string>` (the helper
  builds solver-name-tagged exception messages).

Build — `core/CMakeLists.txt`:
- Added `test_cg` executable + `add_test(NAME fea_cg)` inside the existing
  `if(Eigen3_FOUND)` block (next to `fea_assembly`), so it is gated on Eigen the
  same way `assembly.cpp` is. It builds its grids in code (no fixture). CI
  configures with `-DTOPOPT_REQUIRE_DEPS=ON`, so `fea_cg` always runs there;
  CI's existing `ctest` step picks it up with no CI-file change.

Tests — `core/tests/unit/test_cg.cpp` (new), ctest `fea_cg` (self-contained
CHECK harness, no framework, §4; public API only).

### Why the 64³ gate is trustworthy
The 64³ test is a fully solid cantilever (65³ = 274 625 nodes → 823 875 DOFs),
clamped on the whole x=0 face, transverse −z tip load. It asserts CG converges
to tolerance within a **4000**-iteration cap. The observed count is **578**
iterations (deterministic — same matrix, same pinned Eigen), so the cap has a
~7× margin; a pathological convergence regression would blow the cap and throw
rather than pass silently. Local Release wall time for the whole `fea_cg` binary
(all 5 test groups incl. the 64³ solve) is ~20 s at ~4.2 GB peak RSS — well
inside the < 5 min budget and CI's 16 GB.

The 64³ test only checks convergence / finiteness / load-direction sign (there
is no independent reference field for it). **Correctness** of the CG result is
anchored by two smaller tests where the answer is known independently: the exact
2×1×1 uniaxial patch field, and a 6×3×3 cantilever cross-checked DOF-for-DOF
against the direct `fea_solve`.

## What I did NOT do
- **Did not replace `fea_solve` with CG or route it through the iterative
  solver.** Both entry points exist: `fea_solve` (SimplicialLDLT, exact for the
  small M2.2 patch test and its 1e-9 tolerance) and `fea_solve_cg` (CG, for
  large grids). §4 sanctions both. The direct path and its exact test are
  untouched.
- **Did not add a rank / singularity guard to the CG path** equivalent to the
  LDLT pivot check in `fea_solve`. CG on a consistent but singular (under-
  constrained) system converges to *a* solution silently; `fea_solve_cg` only
  guards non-convergence (throws when the iteration cap is hit before
  tolerance). Every M2.3 test constrains the model fully (whole-face clamp or
  three symmetry planes), so `K_ff` is SPD. A caller that under-constrains and
  wants detection should use `fea_solve` (which has the pivot check) or add a
  null-space check later. Flagged for M2.4/M3.
- **Did not add stress / strain / von-Mises recovery** (that is M2.4) or
  reaction-force output. `fea_solve_cg` returns displacements only, like
  `fea_solve`.
- **Did not tune / expose the preconditioner choice.** It is fixed to Eigen's
  `DiagonalPreconditioner` (= Jacobi), exactly as the task specifies. No
  incomplete-Cholesky or block-Jacobi variant (not requested; §8 no speculative
  abstraction).
- **Did not handle interior-void grids** (nodes attached to no element). As
  handoff 011 noted, such a free node yields a zero `K_ff` diagonal; under CG
  the Jacobi preconditioner would divide by it. All M2.3 grids are fully solid.
  SIMP void regions (M3) will need those DOFs pinned/filtered — a later concern.
- Did not touch `/app/`, `ARCHITECTURE.md`, `materials.json`, CI config, any
  fixture, or any existing test/assertion/tolerance. The ROADMAP change is the
  single M2.3 box.

## Warnings for the next run
- **`fea_solve_cg` throws on non-convergence, not on singularity.** If M3 drives
  it on a nearly-singular SIMP system, a slow/failed converge surfaces as a
  `std::runtime_error` at the iteration cap — handle it (raise the cap, add
  regularisation, or pin void DOFs) rather than assuming a returned field is
  valid. There is no silent-garbage path *except* the consistent-singular case
  above, which no current caller hits.
- **`max_iterations = 0` means Eigen's default cap = 2 × free-DOF count.** For
  64³ that is ~1.6M iterations — effectively unbounded for a test's wall clock.
  Always pass an explicit cap when you want a bounded, fast-failing solve (the
  64³ test passes 4000). The iteration count is deterministic per matrix, so a
  cap set from an observed count (578 → 4000) is safe across CI hardware; only
  wall time varies with hardware, not the count.
- **The reduction still builds the full triplet list and `P K Pᵀ`.** Assembly
  for 64³ peaks at ~4 GB RSS (≈150 M triplets). Fine for CI's 16 GB and the
  current grid sizes, but a matrix-free CG (apply `K` element-by-element without
  assembling) will be the move if grids grow past ~128³. Not needed now (§8).
- **Node / DOF / Voigt conventions remain load-bearing** and identical to
  M2.1/M2.2 (bottom-CCW-then-top-CCW, `3*node+comp`). `assemble_reduced` is the
  single shared assembly path for both solvers now — a bug there breaks both.
- **`CgInfo` is filled even on the throwing path** (before the throw), so a
  caller catching the exception can still inspect `iterations`/`residual` if it
  holds a pointer to the struct. The non-convergence test relies on this.
- Local Eigen is homebrew; CI uses vcpkg-pinned Eigen.
  `ConjugateGradient`/`DiagonalPreconditioner` are stable across both; the 578
  iteration count may differ by a handful between versions but stays far under
  the 4000 cap.
