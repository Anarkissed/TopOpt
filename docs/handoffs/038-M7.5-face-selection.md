## Bridge-capability audit (shaped the deliverable — read this first)
The task states the bridge "already exposes … tagging entry points
(`tag_step_face` / `mask_step_face`)" and that the loop walk should "use the face
adjacency the bridge exposes; if adjacency is not exposed, file `## Blocked` …
do not widen the bridge yourself." I audited `TopOptBridge.hpp` + `TopOptKit.swift`
against `core/include/topopt/step.hpp`:

- ✅ `ImportedMesh.faceIDs` — per-triangle B-rep face id (STEP). Exposed.
- ✅ `tag_step_face(step_path, face_id, as_fixture, resolution)` — exposed → group →
  **Fixture / Load** tagging is deliverable.
- ❌ **`mask_step_face` is NOT bridged.** The core has it (M3.7 passive regions,
  `step.hpp:115`) but the bridge forwards only `tag_step_face`. So group →
  **Frozen** tagging cannot be applied → see `## Blocked`.
- ❌ **No B-rep face adjacency and no `StepSurfaceKind` are forwarded.** However
  both are *derivable* from data the bridge already exposes (`indices` + `faceIDs`),
  so the hole face-loop walk is implemented Swift-side with **no bridge change**
  (see below) rather than blocked.

Ground truth for the loop walk, from the committed fixture's expected values
(`core/tests/fixtures/demo/expected_values.json`, `test_cli.cpp:152–165`): the
l-bracket has **10 B-rep faces (8 planar + 2 cylindrical)**, and the **two Ø5
through-holes are each a single cylindrical face**. So for this fixture "select the
hole's whole face loop" = select that one cylinder face's triangles.

### Testable core (pure value types, GPU-free) — `Sources/TopOptFlows/`
- **`SelectionModel.swift`** — the design's Selections logic ported to Swift:
  `SelectionGroup` (id, editable `name`, `colorIndex` → `DS.Color.groupPalette`
  colour, ordered deduped `faces`, `faceLabel`) and `SelectionModel`
  (create/`addGroup`, `pickFaces`/`pickFace`, `rename`, `setActive`, `remove`,
  `color(forFace:)`/`faceToGroup`). Faithful to the design's `_makeGroup`
  (auto-name "Group A/B/C…" wrapping at Z; round-robin palette via a `nextColor`
  cursor) and `_pickFace` (tap-again-to-deselect; a face belongs to exactly one
  group so a new pick *steals* it from others; empty non-active groups are dropped).
  Also `GroupRole` (fixture/load/frozen) and `WorkspaceTool` (orbit/face/force with
  the design's hint copy).
- **`FaceSelection.swift`** — two pieces:
  - `FacePicker` — ray-casts a normalized tap through the `OrbitCamera` (unproject
    near/far, Möller–Trumbore, nearest hit) to the B-rep face id under it. This is
    the CPU reference for the id-buffer pass (same input = a pixel, same output = a
    face id), so a test pins a known pixel → expected face id with no GPU.
  - `FaceTopology` — the hole face-loop walk. Face adjacency (faces sharing a mesh
    edge) and curvature (a face whose triangle normals fan by > 5° is
    cylindrical/conical, i.e. a hole; a planar face's normals are constant) are
    **derived from `indices` + `faceIDs`** — no bridge widening. `loop(fromFace:)`
    walks the connected run of *curved* faces from a tapped hole face (grabbing a
    multi-face counterbore whole) and stops at the planar faces around it; tapping a
    planar face selects just that face. For a single-face hole (every l-bracket
    hole) it returns exactly that cylinder face.
- **`SelectionTagging.swift`** — `SelectionTagger` maps groups → the bridge:
  Fixture → `tag_step_face(asFixture: true)`, Load → `tag_step_face(asFixture:
  false)`, one call per face id. The bridge call is injected (defaults to
  `TopOptKit.tagStepFace`) so a spy verifies the exact (faceID, asFixture) calls.
  **Frozen groups are rejected up front** with `SelectionTagError.frozenUnsupported`
  (no partial tagging) because `mask_step_face` is not bridged — see `## Blocked`.

## What I did NOT do
- **Did NOT bridge `mask_step_face` / apply Frozen tagging.** `mask_step_face` (the
  core's M3.7 passive-region entry point) is not on the bridge surface, and widening
  the bridge is out of scope for this task ("do not widen the bridge yourself; file
  Blocked"). Fixture/Load tagging is fully wired via `tag_step_face`; Frozen is
  reported `.frozenUnsupported` and requested in `## Blocked`. **Because Frozen
  tagging is part of the M7.5 deliverable, the ROADMAP M7.5 box is left UNCHECKED.**
- **Did NOT use the core's exact `StepSurfaceKind` for the loop walk.** The bridge
  does not forward per-face `StepFaceInfo` (kind/radius), so curvature is derived
  from a triangle-normal-spread heuristic. It is exact for the l-bracket (2 clean
  cylinders vs 8 flats) and any part whose holes are visibly curved, but a fillet or
  a very coarse tessellation could fool it. Forwarding `StepFaceInfo` would make it
  robust — see `## Blocked` (a recommendation, not a hard blocker).
- **Did NOT run the viewer's Metal draw / gestures / id-pass on a device.** The
  highlight, tap, and id-pass render need a display; they compile (all three slices)
  and the id-pass readback is checked headlessly against the CPU picker, but the
  on-device look, orbit/zoom/tap *feel*, and highlight fidelity are maintainer QA
  (the M7 /app/ standard).
- **Did NOT batch multi-face tagging.** `tag_step_face` is stateless — it re-imports
  + re-voxelizes per call — so tagging a group with N faces does N import+voxelize
  round-trips. Correct but slow; a future `tag_step_faces([face_ids])` bridge call
  (or a stateful session grid, M7.7) should replace it. Not widened here.
- **Did NOT assign roles from arrows.** In the design a group with arrows is a Load
  and an arrow-less group a Fixture anchor; arrows are M7.6. The `SelectionTagger`
  takes a role-per-group closure so M7.6/M7.7 can drive it; the workspace does not
  tag yet (no Optimize until M7.7).
- **Did NOT persist selection.** `SelectionModel` is per-workspace-session `@State`,
  reset when a different part is imported. Cross-launch persistence is a later
  concern (same as recents, handoff 036).

## Blocked
Two bridge accessors are needed to *complete* M7.5; both require touching the bridge
surface, which this task forbids doing unilaterally. The M7.5 box is left unchecked
pending #1.

1. **`mask_step_face` on the bridge (required for Frozen tagging — completes M7.5).**
   The core has `topopt::mask_step_face(grid, model, face_id, mask_value,
   depth_voxels, mask)` (`core/include/topopt/step.hpp:115`) but the bridge only
   forwards `tag_step_face`. Requested minimal addition to `TopOptBridge.hpp` +
   `bridge.cpp` (mirroring `tag_step_face`, POD-only, catching core exceptions):
   ```
   int64_t mask_step_face(const std::string& step_path, int face_id,
                          int mask_value /*0=FrozenSolid,1=FrozenVoid…*/,
                          int depth_voxels, int resolution, BridgeError& err);
   ```
   `SelectionTagger` already routes `.frozen` groups to a `.frozenUnsupported` error
   at exactly the seam this would fill; wiring it is a small follow-up once the
   accessor exists.
2. **Forward per-face `StepFaceInfo` on `ImportedMesh` (recommended, for a robust
   loop walk — not blocking M7.5's l-bracket deliverable).** The core computes
   `StepModel.faces` (`StepSurfaceKind` Plane/Cylinder/Other + `cylinder_radius_mm`,
   `step.hpp:52–67`) but the bridge drops it. Exposing e.g. a parallel
   `std::vector<int32_t> face_kinds` (+ optional radii) on `ImportedMesh` would let
   the loop walk classify holes from the exact B-rep instead of the normal-spread
   heuristic, and would let the design's "tap a hole" affordance/hint key on real
   cylinder faces. The current heuristic is exact for the l-bracket, so this is a
   robustness recommendation, not a hard blocker.
