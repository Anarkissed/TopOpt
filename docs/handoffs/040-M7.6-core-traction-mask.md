### Core — uniform-traction load path (ctest)
The core had `self_weight_loads` (a body load) but **no surface-traction entry
point**, so I added one (the task's "if the core lacks a traction-assembly entry
point, add it in /core/").

- **`core/include/topopt/fea.hpp`** — declared
  `std::vector<NodalLoad> traction_loads(const VoxelGrid& grid, VoxelTag tag,
  Vec3 total_force)`. Also added a direct `#include <stdexcept>` (the file now
  names `std::invalid_argument` in a doc-comment; AGENT_PROMPTS §1 portability
  gate). Documented the exact contract: the loaded surface is every cell-face of
  a `tag` voxel whose axis-neighbour is not solid (Empty or off-grid) — the
  exposed boundary of the tagged selection; `total_force` is spread as a uniform
  traction over it; because voxels are cubic every free face has equal area, so
  each free face carries `total_force / face_count`, split 1/4 to each of its
  four corner nodes (the consistent load vector of a bilinear quad under uniform
  traction). Contributions accumulate over shared nodes, so the emitted nodal
  loads **sum exactly to `total_force`** and the load is genuinely distributed,
  never lumped at a centroid. Direction is applied uniformly to every free face
  (a fixed-direction traction — the UI's Gravity/Push/Pull). Throws
  `std::invalid_argument` when the tagged region has no exposed face.
- **`core/src/fea/assembly.cpp`** — implemented `traction_loads` next to
  `self_weight_loads` (same TU, same Eigen gate, same emit-per-non-zero-component
  style), plus two file-local helpers: `neighbour_solid` (is the axis-neighbour
  across a given cell-face solid; off-grid counts as exposed) and
  `face_corner_nodes` (the four grid nodes of a given cell-face).
- **`core/tests/unit/test_traction.cpp`** (new) + **`core/CMakeLists.txt`**
  (registered `traction_loads` inside the `if(Eigen3_FOUND)` block next to
  `fea_assembly`, since `traction_loads` lives in the Eigen-gated `assembly.cpp`).

Design choice (documented in the header): the traction spreads over ALL free
faces of the tagged voxels — the full exposed surface of the selection — not
only the faces coplanar with the picked B-rep face. The tag-based core primitive
does not carry the B-rep normal; distributing over the exposed surface is
well-defined, area-weighted, and makes the nodal sum exact. For a one-voxel
`tag_step_face` slab against a planar face this is the face itself plus the thin
perpendicular rim of the slab's edge (a small fraction for a thin slab). Noted
under "What I did NOT do" as the modelling approximation.

### Bridge — forward `mask_step_face` (app package / xcodebuild)
The core `mask_step_face` (M3.7) existed but was not on the bridge surface
(handoff 038-M7.5 Blocked #1). Forwarded it, mirroring `tag_step_face`.

- **`app/TopOptKit/Sources/TopOptBridge/include/TopOptBridge.hpp`** +
  **`bridge.cpp`** — added
  `int64_t mask_step_face(step_path, face_id, mask_value, depth_voxels,
  resolution, BridgeError&)`. POD-only, catches core exceptions → `BridgeError`,
  gated on `TOPOPT_BRIDGE_HAS_OCCT` exactly like `tag_step_face` (STEP needs
  OCCT). `mask_value` matches `topopt::MaskValue` (0=Active, 1=FrozenSolid,
  2=FrozenVoid) and is range-validated; stateless (re-imports + re-voxelizes per
  call, same pattern/limitation as `tag_step_face`).
- **`app/TopOptKit/Sources/TopOptKit/TopOptKit.swift`** — `TopOptKit.MaskValue`
  enum (`active`/`frozenSolid`/`frozenVoid`, raw values matching the core) and
  `maskStepFace(stepPath:faceID:mask:depthVoxels:resolution:)` wrapper, so
  M7.6-app can call the bridge type-safely.
- **`app/TopOptKit/Tests/TopOptKitTests/TopOptKitTests.swift`** — 3 tests
  (below).

## What I did NOT do
- **Did NOT wire the app's `SelectionTagger.frozen` path to the new bridge.**
  Handoff 038-M7.5 routes `.frozen` groups to `.frozenUnsupported`; consuming the
  new `maskStepFace` (freeze load/anchor groups as a passive shell) is M7.6-app's
  job — the ROADMAP says M7.6-app "Calls M7.6-core's bridge signatures". I
  provided the signature + wrapper + tests; the Swift force/gravity experience
  wires it. Kept the diff to the one task.
- **Did NOT plumb `traction_loads` into `run_minimize_plastic`.** The bridge's
  optimize driver still hardcodes the min-x fixture + self-weight (M7.1 seam).
  Passing a real tagged Load selection (traction) + Fixture/Frozen shell into the
  optimizer is a larger run-wiring change (M7.6-app / M7.7), not this task.
  `traction_loads` is the core entry point those will call; it is fully tested
  standalone here.
- **Did NOT restrict the traction to faces coplanar with the picked B-rep face.**
  The traction spreads over the full exposed surface of the tagged voxels
  (including the thin perpendicular rim of a one-voxel slab). This is the
  documented modelling choice; it keeps the nodal sum exact and is area-correct
  for the exposed surface. A B-rep-normal-aware restriction would need the face
  normal forwarded into the core primitive — deferred, not needed for D7's
  "uniform traction over the combined area of its selected faces".
- **Did NOT change `tag_step_face`, `self_weight_loads`, or any existing test.**
  Only additions.
- **Did NOT batch multi-face masking** (same stateless re-import-per-call cost as
  `tag_step_face`, handoff 038-M7.5). A future `tag/mask_step_faces([...])` or a
  stateful session grid (M7.7) is the fix; not widened here.
- **Did NOT run the iOS/simulator build or any on-device gate.** OCCT-free iOS
  default is unchanged (`occt-frameworks.generated.json` absent — the M7.1c
  invariant). The bridge `mask_step_face` compiles OCCT-off to the same graceful
  "requires OpenCASCADE" `BridgeError` as `tag_step_face`.
