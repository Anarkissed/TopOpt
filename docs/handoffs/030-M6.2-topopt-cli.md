## What I did
Added the M6.2 CLI: a strict job.json parser, a `run_job` driver that wires the
whole §5 pipeline (STEP import → watertight check → voxelize → geometric
fixture-face selection → tag → self-weight minimize_plastic → report + mesh
export), a thin `topopt-cli` executable over it, and the demo-job integration
test. The committed demo fixture `tests/fixtures/demo/job.json` DEFINES the
schema (its `_comment`; DECISIONS.md 2026-07-09) and the test drives it
end-to-end against `expected_values.json`'s invariants.

### Job schema — `core/include/topopt/job.hpp`, `core/src/cli/job.cpp` (always-built)
- `parse_job` / `load_job_file`: recursive-descent JSON parser (module-local,
  mirroring the materials/settings parsers — no shared "json" API, no new
  dependency) + strict schema validation: exactly the fixture's keys, unknown
  (non-underscore) keys rejected with the key named, missing required keys
  rejected, type/value rules per field (resolution integer ≥ 1; selector kind
  "cylindrical" with radius > 0; gravity direction 3 finite numbers non-zero,
  magnitude > 0; ladder in (0,1] strictly descending; margin_stop ≥ 0; simp
  block optional with only max_iterations; mesh_format "3mf"|"stl").
  Keys beginning `_` are maintainer comments, ignored at every level (the
  fixture itself carries four of them).
- `JobDescription` / `JobFaceSelector` / `JobGravity` / `JobOutput` structs;
  `kJobFaceRadiusToleranceMm = 1e-6` per the fixture's `_fixture_note`.

### Geometric face selection — `step.hpp` / `src/io/step.cpp` (extension)
`StepModel` gains `faces` (one `StepFaceInfo` per B-rep face, same TopExp order
as `triangle_face`): surface kind Plane/Cylinder/Other via `BRepAdaptor_Surface`
(sees through trimming; radius from `Cylinder().Radius()`, mm). This is what
lets the CLI implement the LOCKED selection rule (DECISIONS.md 2026-07-09:
geometric, never raw OCCT face indices). No OCCT type appears in the public
header; no new OCCT toolkit is linked (BRepAdaptor_Surface lives in the
already-linked TKBRep).

### The driver — `core/src/cli/run_job.cpp` (gated on OCCT AND Eigen)
Per job: material lookup → mesh-format capability check (fail fast, before any
heavy work) → `import_step_file` (relative `model` resolves against the job
file's directory) → selector matching over `model.faces` (each selector must
match ≥ 1 face; all matches tagged FIXTURE via `tag_step_face`) → watertight
check → `voxelize(resolution)` → Dirichlet BCs clamping every node of every
Fixture voxel → `minimize_plastic` (M5.3) → `job_report_json` written to
`<out>/<output.report>` → one exported mesh per ACCEPTED variant
(`<prefix>_<vf*100, 3 digits>.<format>`, e.g. `variant_070.3mf`), the variant's
`check_v3` cleaned mesh, via M6.1 `write_3mf_file`/`write_stl_file`.
- **Units:** the job is mm + mm/s²; materials density is g/cm³. The
  mm-MPa-consistent density unit is t/mm³ = 1e-9 g/cm³, and the margin depends
  only on the product density × gravity, so `run_job` folds the 1e-9 into the
  gravity magnitude passed to `MinimizePlasticOptions.gravity`. Reported
  stresses are then MPa (documented in job.hpp + run_job.cpp).
- 3MF path compiled under `TOPOPT_HAVE_3MF` (defined when lib3mf is found);
  without it a job requesting "3mf" throws a JobError diagnostic up front.

### The executable — `core/src/cli/main.cpp` (`topopt-cli`, gated OCCT+Eigen)
`topopt-cli run <job.json> [--out DIR] [--materials PATH] [--rules PATH]`.
Defaults: out = "."; materials/rules = the committed files (paths injected at
build time — this is the repo's dev/CI driver, not an installed product).
Exit 0 success (prints a run summary), 1 with the diagnostic on stderr for any
job/pipeline failure, 2 for usage errors.

### The fix the demo exposed — `src/simp/simp.cpp` `effective_mask`
The demo bracket is the FIRST optimizer run on a grid with Empty voxels (every
earlier SIMP test builds fully-solid boxes). The mask-aware path counted
caller-`Active` EMPTY voxels into the Active-voxel budget, so the OC volume
target `volume_fraction * n_active` was computed over the whole bounding grid
(73728 voxels) instead of the part (~17k solid): the target was unreachable,
the OC bisection saturated, and EVERY rung collapsed to the same all-solid
design (achieved vf 0.2316 = solid/grid for requests 0.7/0.5/0.3). voxel.hpp's
documented contract — "Empty voxels are never design variables, so their mask
entry is ignored" — was not honored. Fix: `effective_mask` now normalizes
Empty voxels to FrozenVoid (exactly what Empty means to every downstream
consumer: excluded from design, filter and stiffness; the analysis grid is
unchanged since Empty already contributes no element). Fully-solid grids are
bit-identical (no Empty voxels → no change), which the untouched
simp/mbb/gate_v2/property_v3/variants/passive_regions/minimize_plastic tests
confirm. One clarifying paragraph added to the masked-overload docs in
simp.hpp. Without this fix M6.2 does not work at all — it is in-scope as
"implementation until green" for the task's mandated integration test, not a
drive-by (the unmasked simp_optimize path is untouched).
