# Handoff 014 — M3.1: Void-DOF safety gate

## Task
Void-DOF safety gate (prerequisite for all SIMP FEA): before any SIMP task
calls `fea_solve_cg` on a density field, pin/filter free DOFs whose `K_ff`
diagonal is zero/near-zero (nodes attached only to void voxels) and add a
null-space / zero-pivot guard so an under-constrained or singular system throws
instead of silently returning a garbage/NaN field.

## What I did
Added the M3.1 void-DOF safety gate to the CG solve path (`fea_solve_cg`), which
is the entry point SIMP (M3.2+) will drive. No public API change, no header
change, no CMake change — the whole change lives inside the already-Eigen-gated
`core/src/fea/assembly.cpp` plus its test.

Implementation — `core/src/fea/assembly.cpp`:
- New private helper `void_dof_survivors(Kff, rf, who)` (anonymous namespace,
  next to `assemble_reduced`). It reads the diagonal of the reduced free-DOF
  matrix `K_ff` and:
  - throws `std::runtime_error` if **every** free DOF has a (near) zero diagonal
    (`dmax == 0`) — a system with no stiffness at all;
  - for each free DOF with `|diag| <= 1e-12 * dmax` (a void DOF): if it carries a
    load (`|rf[i]| > 1e-9 * max|rf|`) it throws `std::runtime_error`
    (an unsupported loaded DOF has no equilibrium — genuinely under-constrained);
    otherwise the DOF is dropped from the returned survivor list;
  - returns the indices (in `K_ff`'s reduced free-DOF numbering) of the surviving
    non-void DOFs.
  Rationale for "drop is safe": `K_ff` is symmetric positive semi-definite, so a
  zero diagonal entry forces its whole row/column to zero — the DOF is fully
  decoupled and pinning it (leaving its prescribed value, 0) changes no other
  DOF. The relative diagonal tolerance `1e-12` catches only exact structural
  voids (Empty-tagged nodes contribute nothing → exactly 0) and will not filter
  legitimately small SIMP stiffnesses (e.g. `E·ρ_min^p` sits far above `1e-12`
  relative — see Warnings).
- `fea_solve_cg` now calls the gate before the CG solve. If it throws, the CG
  diagnostics (`CgInfo`) are recorded as a structural rejection
  (`converged=false, iterations=0, residual=0`) and the exception is rethrown —
  the rejection happens before any CG iteration. If DOFs are filtered, a
  selection matrix `Q` reduces `K_ff/rf` onto the survivors (`K_gg = Q K_ff Qᵀ`,
  materialised un-chained, same pattern as the existing `P K Pᵀ` reduction to
  avoid the Eigen-5 sparse-product quirk from handoff 011), CG solves the smaller
  system, and the solution scatters back to the surviving global DOFs.
  **When nothing is filtered (a fully solid grid) the solve pointers alias the
  original `K_ff/rf` unchanged**, so solid problems — and every M2.2/M2.3/M2.4
  test — are bit-for-bit unaffected.
- `fea_solve` (the direct SimplicialLDLT path) is **untouched**: it already
  rejects void nodes via its LDLT zero-pivot check (throws `invalid_argument`).
  The task scopes the new gate to the CG path that SIMP uses.

Tests — `core/tests/unit/test_cg.cpp` (extends the existing `fea_cg` test,
Eigen-gated, self-contained CHECK harness, public API only):
- Added a namespace helper `make_cube_with_void_tail(h)`: one solid voxel + one
  Empty voxel along +x, so the four nodes on the far (a==2) face touch only the
  Empty voxel → their DOFs have an exactly-zero `K_ff` diagonal (dangling void
  nodes) while the solid cube remains a well-posed clamped problem.
- Section 6 — **filtering preserves the real solution**: Grid A (cube + void
  tail), clamp the a==0 face, load the a==1 face in −z. `fea_solve_cg` converges,
  the field is finite, the void nodes (a==2) come back exactly 0, and the solid
  nodes (a∈{0,1}) match a lone-cube solve (Grid B, 1×1×1) DOF-for-DOF to CG
  tolerance.
- Section 7 — **loaded void DOF is rejected**: Grid A but with the load on a void
  node (a==2). `fea_solve_cg` throws `std::runtime_error`, and `CgInfo` reports
  `!converged, iterations==0` (rejected before any CG iteration).
- Section 8 — **all-void grid is rejected**: a 1×1×1 all-Empty grid (no
  stiffness) throws `std::runtime_error` instead of returning a zero field.

- **Section 6 passes with or without the gate on this Eigen, by design — and I am
  flagging it rather than hiding it.** This machine's Eigen
  `DiagonalPreconditioner` already guards a zero diagonal (it uses `invdiag=1`
  instead of dividing by zero), so the unfixed CG does *not* produce NaN on a
  dangling-void grid: it converges and leaves the unloaded, decoupled void DOFs
  at 0. Section 6 is therefore a **correctness/robustness regression test** (it
  locks in that a void-tail solve equals the lone-cube solve and that the DOF
  re-mapping after filtering is correct), not a fixed-vs-unfixed discriminator on
  this Eigen. The gate's value is (a) not depending on that Eigen internal —
  older/other preconditioners that don't zero-guard would divide by zero — and
  (b) turning the two genuinely-singular cases (all-void, loaded-void) from
  "silent zero field" / "slow non-convergence throw" into an immediate,
  correctly-typed structural rejection. Sections 7 & 8 carry the discrimination.

## What I did NOT do
- **Did not detect the rigid-body (nonzero-diagonal) null space in the CG path.**
  The gate catches the *zero-diagonal* singularity named in the task (the Jacobi
  zero pivot) — decoupled void DOFs and fully-unsupported systems. A different
  singularity, where `K_ff` has all-positive diagonals but a rigid-body mode
  survives (e.g. a floating solid component with no BC), is still not detected by
  `fea_solve_cg`: CG converges to *a* solution silently, exactly as noted in the
  M2.2/M2.3 handoffs. Detecting it robustly needs a factorisation (the direct
  `fea_solve` LDLT pivot check has it) or an explicit rigid-body null-space
  basis; that is out of M3.1's scope (§8: no speculative abstraction) and no
  current or planned M3 caller under-constrains a solid component this way — SIMP
  fully clamps its load/fixture BCs. **M3.2+ must still remove rigid-body motion
  via BCs; the gate does not do that for you.**
- **Did not change `fea_solve` (direct path).** It already throws on void nodes
  via its LDLT pivot check; the task targets the CG path SIMP uses.
- **Did not filter "near-void" small-but-nonzero stiffness.** The diagonal
  tolerance is `1e-12 · dmax`, which only catches exact structural voids
  (Empty-tagged nodes contribute literally 0). SIMP's near-void voxels
  (`ρ_min > 0`) will keep a small positive stiffness and are intentionally **not**
  filtered — filtering them would delete real, if soft, structure. If SIMP later
  wants a density floor treated as void, that is an M3.2 modelling decision, not
  a solver change.
- **Did not add a per-voxel-modulus (`E·ρ^p`) assembly path.** `fea_solve_cg`
  still assembles a single uniform-material element for every solid voxel. SIMP
  needs graded per-voxel stiffness (M3.2); this gate operates on whatever `K_ff`
  the assembly produces, so it will apply unchanged once graded assembly exists.
- **Did not add a `CgInfo` field or otherwise change the public signature.** The
  structural rejection is reported by reusing the existing `CgInfo`
  (`converged=false, iterations=0`) on the throwing path.
- Did not touch `/app/`, `ARCHITECTURE.md`, `materials.json`, CI config, any
  fixture, any header, any CMake file, or any existing test/assertion/tolerance.
  The ROADMAP change is the single M3.1 box.

## Warnings for the next run
- **This is the M3.1 gate the ROADMAP says every later M3 task must go through.**
  SIMP FEA solves should call `fea_solve_cg` (not the direct `fea_solve`) so the
  void-DOF filtering applies. The box being checked satisfies the "No later M3
  task may call the solver until this box is checked" precondition.
- **The gate only removes the zero-diagonal singularity, not rigid-body modes.**
  See "What I did NOT do". If an M3 test builds a grid whose solid part floats
  (no sufficient BC), `fea_solve_cg` will *not* throw — it converges to an
  arbitrary shifted field. Constrain rigid-body motion in the BCs.
- **Diagonal tolerance is relative (`1e-12·dmax`), load tolerance is relative
  (`1e-9·max|rf|`).** Both are keyed off the exact-zero nature of structural
  voids. If a future modelling change introduces intentionally tiny but nonzero
  stiffness that should count as void, revisit these thresholds deliberately
  rather than assuming the current values cover it.
- **`fea_solve_cg` now has two throw taxonomies:** `std::invalid_argument` for
  bad BC/load indices or material (from `assemble_reduced`/`hex8_stiffness`,
  pre-assembly caller misuse), and `std::runtime_error` for solve-time failures
  (non-convergence, and now the void-gate structural rejections: all-void and
  loaded-void). The M3.1 tests catch `std::runtime_error`.
- **The `Q K_ff Qᵀ` reduction is materialised un-chained** (like the existing
  `P K Pᵀ`) to dodge the Eigen-5 `sparse*sparse` pruning template error from
  handoff 011. Keep the products in explicit `SpMat` temporaries.
- CI required by DECISIONS.md (2026-07-03): a green **CI** run (Linux, vcpkg
  OCCT/Eigen/lib3mf, `-DTOPOPT_REQUIRE_DEPS=ON`) is needed to satisfy rule 5 —
  local evidence alone does not. Open a PR and confirm the pipeline is green
  before merge. The vcpkg Eigen may not zero-guard the Jacobi diagonal the same
  way homebrew's does; the gate makes the outcome independent of that, and
  Sections 7 & 8 assert the behaviour regardless.
