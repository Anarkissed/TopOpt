# Handoff 054 — M7.mma.1: MMA updater behind SimpOptions.updater

## Task
ROADMAP M7.mma.1: add the MMA updater (Svanberg 1987), compliance objective +
volume constraint only, as a drop-in alternative to Optimality Criteria behind
`SimpOptions.updater = {OC, MMA}`; default stays OC, no fixture changes.

Ran the full local `ctest` before changing anything: 22/22 green (the OCCT- and
lib3mf-gated tests — `step_import`, `face_tag`, `mask_face`, `cli_demo`,

## What I did
Pure core (`/core/`); no `/app/`, no fixtures, no dependency added.

- **`core/include/topopt/simp.hpp`**: added `enum class SimpUpdater { OC, MMA }`
  and a `SimpUpdater updater = SimpUpdater::OC;` field on `SimpOptions` (documented
  as the design-update rule; default OC so existing callers and the locked
  Gate-V2 fixture are byte-unaffected). Reworded the `move` comment (OC/MMA).
- **`core/src/simp/simp.cpp`**: implemented the MMA updater and wired it in.
  - `mma_update(...)` (anonymous namespace) — one MMA design update for the
    minimum-compliance problem with a single volume constraint. It consumes the
    SAME inputs as `oc_update` (the physical-space compliance sensitivity, the
    volume-fraction target, the move limit), filters them into design space via
    the same symmetric filter transpose, and enforces the SAME volume constraint
    on the filtered density — only the multiplicative OC step is replaced by
    MMA's convex separable subproblem. With a single constraint (m = 1) the
    subproblem's dual is 1-D, so it is solved by the same monotone bisection
    structure `oc_update` already uses (bracket lambda, bisect until the
    approximate constraint is met). A small `MmaState` (previous two designs +
    the moving asymptotes L/U) persists across the run's iterations.
  - Constants documented in code, per the paper / the widely-used mmasub
    reference: asyinit 0.5, asyincr 1.2, asydecr 0.7, albefa 0.1, raa0 1e-5; the
    MMA move limit reuses `SimpOptions.move` (as a fraction of the x-range) so
    OC and MMA are compared at the same move.
  - Wiring: in the non-masked `simp_optimize`, the update step dispatches on
    `options.updater`; `MmaState`/`mma_iter` are owned by that overload and only
    touched on the MMA path (OC path is byte-unchanged).
  - Scope guards for M7.mma.1 (MMA is the plain compliance loop only):
    `validate_updater_options` throws `std::invalid_argument` if MMA is combined
    with a Heaviside projection schedule; the masked overload throws if
    `updater == MMA`. These are explicit rejections, not silent OC fallbacks;
    the projection/passive-region MMA paths are later tasks (M7.mma.2+/.4).
- **`core/tests/validation/test_mma.cpp`** (new) + **`core/CMakeLists.txt`**
  (register `test_mma`, Eigen-gated like the other optimizer tests).

## Design choices & why
- **MMA reproduces OC, benchmark parity test.** The test runs the two Gate-V2
  benchmark problems (cantilever 24×8×8, clamped beam 48×8×8 — identical
  geometry/BCs/loads/material/volfrac to `benchmarks.json`) through the plain
  (unprojected) SIMP loop under OC and under MMA and asserts MMA's final
  compliance is within 2% of the OC reference at the same volume fraction. The
  OC "reference" is computed in-test (the committed `benchmarks.json` is the
  PROJECTED formulation and stays OC-only — M7.mma.1 says "NO fixture changes";
  nothing in `tests/fixtures/**` is touched).
- **Filter radius 2.5 in the parity test.** I use the SAME filter radius the
  committed `benchmarks.json` locks (`rmin = 2.5`). At that radius the
  minimum-compliance design is well-regularised (no near-degenerate single-voxel
  members), so OC and MMA converge to essentially the same layout — measured
  rel = 0.381% (cantilever) and 0.103% (beam), comfortably inside the 2% band.
  (At the classic non-projected `rmin = 1.5` the two updaters still agree to
  ~1.8% at full convergence, but the tighter margin made a robust CI assertion
  fragile; 2.5 is both more faithful to the committed benchmark and robust.)
- **FD sensitivity checks.** The task's "existing FD sensitivity checks pass
  unchanged" is satisfied structurally: MMA consumes the exact same
  `simp_compliance` sensitivities OC does — only the update rule differs — so
  `test_simp`'s finite-difference gates are untouched and still green. Not
  duplicated in `test_mma`.
- **Iteration counts** are printed (informational, per the task): MMA converged
  in fewer OC-equivalent iterations than OC on both benchmarks (cantilever 72 vs
  92; beam 133 vs 150).

## Warnings for the next run
- **The MMA move limit reuses `SimpOptions.move`.** A much larger move (e.g. 0.5)
  lets MMA explore harder and can land on a *different* (even lower-compliance)
  design than OC — i.e. it stops "reproducing" OC and the OC-parity rel can
  exceed 2% in the better direction. The parity test deliberately uses move 0.2
  (matching OC) and rmin 2.5 to keep the two updaters in the same basin. If
  M7.mma.4 makes MMA the default, revisit the move-limit choice for production
  runs rather than assuming 0.2 is optimal for MMA.
- **`test_mma` runs two full ~150-iteration optimizations per benchmark** (OC and
  MMA on 24×8×8 and 48×8×8) and takes ~2.5 min locally. It is Eigen-gated and
  registered like the other optimizer validations.
- **`mma_update` asymptote state** (`xold1`/`xold2`) is only read for
  `mma_iter >= 3`, by which point both are full grid-sized; iterations 1–2 use
  the `asyinit` initialisation. If a future change resets `mma_iter` mid-run
  (e.g. per projection stage), keep that invariant.
- **`DECISIONS.md`** still carries the maintainer's uncommitted 2026-07-11
  print-time entry (not part of my commits), as noted in handoffs 045–053. I did
  not touch it; no decision entry is required for this task (no ARCHITECTURE
  change, no new dependency, an additive option with an OC default).
