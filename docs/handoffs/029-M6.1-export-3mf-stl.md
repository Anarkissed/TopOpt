## What I did
Added the M6.1 export layer: the 3MF (primary) and STL (secondary) writers plus a
3MF re-import for the round-trip, and the round-trip tests that export a
marching-cubes surface, re-import it, and assert the mesh-level Gate V3
properties + volume within 0.5%.

### STL export — always-built, pure C++/std
- `core/include/topopt/stl.hpp`: new
  `void write_stl_file(path, mesh, StlFormat format = StlFormat::Binary)`.
- `core/src/io/stl.cpp`: implementation. Binary writer mirrors the existing
  little-endian binary reader (80-byte header — deliberately NOT starting with
  "solid" so the existing auto-detector never misreads it as ASCII — uint32
  triangle count, per-triangle float32 normal + 3 vertices + zero attribute
  word). ASCII writer emits `%.17g` full-double-precision coordinates. Each
  facet normal is computed from the winding (zero vector for a degenerate
  triangle, never NaN). Vertex order and triangle winding are preserved
  verbatim, so a watertight mesh re-welds (read_stl_file welds by exact
  coordinate) watertight. Throws `StlError` if the file cannot be opened/written.

### 3MF export/import — lib3mf-gated
- `core/include/topopt/threemf.hpp` (new): `class ThreeMfError`,
  `void write_3mf_file(path, mesh)`, `TriangleMesh read_3mf_file(path)`.
- `core/src/io/threemf.cpp` (new): lib3mf implementation. `write_3mf_file` builds
  a model with one mesh object (`SetGeometry` from the mesh's vertices +
  triangle indices), references it with one build item at the identity transform
  (a mesh object is only serialized if a build item points at it), and writes via
  the "3mf" writer. `read_3mf_file` reads via the "3mf" reader and returns the
  first mesh object's vertices + triangle indices as a `TriangleMesh`. Every
  lib3mf `ELib3MFException` is wrapped in `ThreeMfError` with the path; a package
  with no mesh object throws `ThreeMfError`.

### CMake — strictly additive
- Added `find_package(lib3mf CONFIG QUIET)` to the local (DEPS=OFF) branch,
  mirroring the existing OCCT/Eigen QUIET fallbacks, so the 3MF TU + test build
  locally when lib3mf is present. CI's DEPS=ON branch already
  `find_package(lib3mf CONFIG REQUIRED)`.
- `src/io/threemf.cpp` + `lib3mf::lib3mf` link are gated on `lib3mf_FOUND`
  (same pattern as OCCT-gated `step.cpp` / Eigen-gated FEA sources).
- New always-built test `export_stl` (`test_export.cpp`); new lib3mf-gated test
  `export_3mf` (`test_3mf.cpp`). Both get `EXPORT_TMP_DIR="${CMAKE_CURRENT_BINARY_DIR}"`
  injected for their temporary files (independent of the ctest working dir).

Design choices and WHY:
- **"V3 properties hold" on a re-imported mesh = the two mesh-level V3 gates**
  (ARCHITECTURE §7: watertight + exactly one connected component). Gates 3
  (Load/Fixture retention) and 4 (min feature size) are properties of the voxel
  grid + density field, not of a bare re-imported triangle mesh, so they are not
  re-checkable from an exported file — they are asserted on optimizer output in
  `test_v3` (M3.5/M3.7), not here. Documented in both test files.
- **Round-trip input is a real marching-cubes surface** (the "printable body" the
  pipeline exports), not a hand-built triangle soup, so the test exercises actual
  export input. A solid block (single component) and a genus-1 through-hole body
  (still single component) are both round-tripped.
- **Both encodings tested.** Binary (float32) proves the 0.5% tolerance holds
  under single-precision; ASCII (`%.17g`) additionally round-trips volume to
  ~1e-9. The default (no format arg) is asserted to be binary.
- **The 3MF re-import is via lib3mf** (symmetric with the writer, one gated TU),
  not OCCT — a mesh round-trip needs no B-rep. This does not change §3's
  "STEP/STL/3MF import (OCCT)" for the *pipeline* front door; it is the
  export module's own inverse used by the round-trip test.

## Warnings for the next run
- **First CI run is the first real execution of the 3MF path.** `export_3mf` has
  never run on the author's machine (no local lib3mf). It is written to the same
  API the front-end syntax check validated, but if it fails in CI the likely
  suspects are the lib3mf writer/reader class string ("3mf") or the build-item
  requirement, not the STL path (which is fully verified locally). If CI's vcpkg
  lib3mf exposes the target under a different name than `lib3mf::lib3mf`, the link
  step is where it would surface — the existing `find_package(lib3mf CONFIG
  REQUIRED)` in the DEPS=ON branch predates this run and already assumed that
  target name.
- **STL binary is float32.** Coordinates round-trip to single precision (~1e-6
  relative), far inside the 0.5% volume tolerance for the test meshes but NOT
  bit-exact. Use the ASCII encoding when full-double fidelity matters.
- **The binary header must never start with "solid".** The auto-detector keys on
  a leading "solid" + a "facet" token; the writer's header (`"topopt binary STL
  export"`) is chosen to avoid that ambiguity. Changing the header text is fine
  as long as it does not begin with "solid".
- **`read_3mf_file` returns the FIRST mesh object only.** The writer emits exactly
  one, so the round-trip is well-defined; a multi-object 3MF authored elsewhere
  would drop all but the first. Broadening that is a future task if needed.
- **Export writes whatever mesh it is given** — it does not run check_v3 or
  enforce watertightness. The pipeline is expected to export a cleaned V3-passing
  body (M3.5 `check_v3().mesh`); export is the last, non-gating step.
