## What I did
Added the multi-variant runner: one function that takes a single job (grid +
material + BCs + loads + shared loop options) and a list of requested volume
fractions, runs the existing `simp_optimize` once per fraction, and extracts a
cleaned printable mesh from each result — returning the three meshes plus the
per-variant compliance report. This is the §5 pipeline's "SIMP loop (repeat per
requested variant) → marching cubes → mesh cleanup" fan-out, composed entirely
from already-shipped operators; no optimizer, filter, FEA, or mesh code changed.

Public API — `core/include/topopt/simp.hpp` (extended; still Eigen-free):
- `struct SimpVariant` — one optimized variant: `requested_volume_fraction` (the
  `SimpOptions.volume_fraction` used), `optimization` (the full
  `SimpOptimizeResult`: compliance, design, physical_density, achieved fraction,
  iterations, converged, history), and `mesh` (the cleaned marching-cubes
  isosurface — largest component — of that variant's physical density).
- `struct MultiVariantResult` — `variants` (one `SimpVariant` per requested
  fraction, in input order) + `std::vector<double> compliances() const`, the
  per-variant compliance report (`variants[i].optimization.compliance` in order).
- `MultiVariantResult simp_optimize_variants(grid, params, bcs, loads, options,
  volume_fractions, iso = 0.5)` — runs `simp_optimize` once per entry of
  `volume_fractions` (each overriding `options.volume_fraction`; every other
  options field shared), extracts + cleans a marching-cubes mesh at `iso` from
  each result's physical density, returns one `SimpVariant` per fraction in input
  order.

Implementation — `core/src/simp/simp.cpp` (appended to the existing Eigen-gated
TU; added `#include <utility>` for `std::move`):
- `MultiVariantResult::compliances()` — collects each variant's final compliance.
- `simp_optimize_variants` — validates `volume_fractions` is non-empty and every
  entry ∈ (0,1] up front (a clean error before any expensive solve; `simp_optimize`
  re-validates each per-variant fraction anyway), then loops: copy `options`, set
  `volume_fraction = vf`, call `simp_optimize`, then
  `keep_largest_component(marching_cubes(grid, physical_density, iso))` for the
  cleaned mesh. Preserves input order (no sorting).

Design choices and WHY:
- **Home = `simp/` (extend `simp.hpp`/`simp.cpp`), not a new module.** The runner
  is driven by `simp_optimize` (Eigen-gated), and `simp.hpp` already includes
  `voxel.hpp` (→ `mesh.hpp`), so `marching_cubes`/`keep_largest_component` are in
  scope. No new directory, no new CMake object, smallest diff. ARCHITECTURE §3 has
  no dedicated "pipeline/runner" module and §8 forbids speculative abstraction.
- **Mesh = `marching_cubes` + `keep_largest_component` (the §5 "marching cubes →
  mesh cleanup" pair), not `check_v3`.** The runner's deliverable is "three
  meshes"; those two calls produce exactly the cleaned printable body. `check_v3`
  additionally computes watertight/retention/min-feature — property-check work the
  runner doesn't need to *do*, so running it inside the runner would be dead
  computation. The V3 property suite is instead run in the *test* on each variant
  output (honoring the M3.5 mandate; see below).
- **`options.volume_fraction` is ignored; each variant uses its own fraction.**
  The runner overrides it per variant. Documented in the header; the test sets it
  to a bogus 0.99 to prove the override.
- **Store the full `SimpOptimizeResult` per variant, not just a compliance
  number.** It is what `simp_optimize` returns; the achieved volume fraction /
  history / physical_density are the honest result of each run and are needed to
  bind each variant to its own target. `compliances()` extracts the named
  "per-variant compliance report" as a one-liner over it.
- **No JSON report file.** ROADMAP M5.2 ("Job report: single JSON per run") owns
  the report *format*; M3.6's "compliance report" is the in-memory per-variant
  data. Emitting JSON here would be starting M5.

### How the test is set up and why it is trustworthy
`test_variants` (new, `tests/unit/`) builds the clamped-root/tip-loaded cantilever
`test_v3` uses (24×8×8, Fixture root face, Load tip face, all-3-DOF clamp on the
x=0 node face, total Fz=−1 spread over the x=nx nodes), then runs
`simp_optimize_variants` with fractions **[0.7, 0.5, 0.3]** (p=3, ρ_min=0.001,
filter 1.5, move 0.2, 40 iterations, change_tol 0, CG tol 1e-8) and asserts the
runner contract:
1. one variant per requested fraction, `requested_volume_fraction` preserved in
   input order;
2. each variant's **achieved** physical volume fraction matches *its own* target
   within 0.01 (the Gate V2 tolerance) — fails if the runner shares one fraction;
3. compliances strictly increase as the fraction decreases (more material →
   stiffer → lower compliance), and are finite and positive — binds the report to
   physics and catches a runner that reuses one design;
4. `compliances()` equals each variant's `optimization.compliance`;
5. per the M3.5 mandate ("Gate V3 property suite wired to run on every optimizer
   output in all future tests"), `check_v3` is run on every variant's physical
   density and the two gates the isotropic loop satisfies (watertight + single
   component) are asserted, plus the runner's own cleaned `mesh` is the same
   surface (matching triangle count);
6. argument validation: empty `volume_fractions`, a fraction > 1, and a zero
   fraction each throw `std::invalid_argument`.

## Warnings for the next run
- **`simp_optimize_variants` ignores `options.volume_fraction`** — each variant
  uses its entry of `volume_fractions`. Do not set the target via the shared
  `options` field expecting it to apply; it is overwritten per variant.
- **Each variant is an independent full `simp_optimize` run** (uniform restart at
  its own fraction), not a warm-started continuation from the previous variant.
  Cost scales linearly with the number of fractions; budget `n_fractions ×`
  a single-optimization cost. At 24×8×8/40 iters that is ~2 s each (~6 s for 3).
- **Meshes are the cleaned largest component** (`keep_largest_component`), so a
  variant's `mesh` is always a single watertight shell even when the raw
  marching-cubes output has stray islands (common at low volume fraction). If a
  later task needs the raw multi-shell mesh or the per-variant `V3Report`, call
  `marching_cubes` / `check_v3` on `variant.optimization.physical_density`
  directly — the runner intentionally stores only the printable body.
- **Retention + min-feature V3 gates still fail on these raw optimizer outputs**
  (unchanged from M3.5). A future test must not wire `check_v3(...).passes` onto a
  variant's output until M3.7 pins Load/Fixture voxels and a thin-feature length
  scale exists. Assert `gate_watertight()` + `gate_single_component()` only, as
  `test_variants` does.
- **`variants` adds ~6 s** (3 × 40-iteration 24×8×8 SIMP solves + 3 `check_v3`);
  full suite ~109 s locally, well under the M2.3 "< 5 min CI budget".
