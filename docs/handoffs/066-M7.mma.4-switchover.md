## What I did (all in `/core/`; no `/app/`, no fixtures, no ROADMAP/DECISIONS, no dep)

### 1. Masked MMA — lift the switchover rejection (`core/src/simp/simp.cpp`)
- New `mma_update_masked(...)` (anon namespace): the mask-aware analogue of
  `mma_update`. Identical moving-asymptote machinery, convex separable
  subproblem and 1-D dual bisection; it just swaps the design set
  (`grid.solid` → Active voxels) and carries the frozen pins (FrozenSolid = 1,
  FrozenVoid = 0), exactly as `oc_update_masked` does for OC. With an all-Active
  mask on an all-Interior grid it reproduces `mma_update` **bit-for-bit** (proven
  by the benchmark table below: masked == plain to the digit).
- The masked `simp_optimize` overload no longer throws for MMA. It now calls the
  shared `validate_updater_options(options)` (which rejects only MMA + a Heaviside
  projection schedule — see below), owns an `MmaState`/`mma_iter` like the plain
  overload, and dispatches the non-projected stage to `mma_update_masked` when
  `options.updater == MMA`.
- `validate_updater_options` reworded + reused by BOTH overloads: MMA + projection
  is still rejected (the projected chain is the OC-locked Gate-V2 formulation;
  MMA+projection is a separate future task), and it is now an EXPLICIT rejection
  on the masked overload too, not a silent fall-back to the OC projected updater.

### 2. Scale-invariance fix for MMA (`core/src/simp/simp.cpp`) — REQUIRED for the switchover
- Both `mma_update` and `mma_update_masked` now normalize the filtered compliance
  sensitivity `dc` to O(1) (`dc[e] /= max_e|dc[e]|`) before building the
  subproblem. **Why this is mandatory, not cosmetic:** the minimum-compliance
  topology is invariant to the load MAGNITUDE, but `dc` scales as load². The MMA
  subproblem's fixed regularizer `raa0 = 1e-5` is added to `p0/q0`; at production
  self-weight scale (`minimize_plastic` compliance is ~1e-8 — real gravity through
  `run_job.cpp` is ~1e-5·density) the un-normalized `raa0` SWAMPS `dc`, so `p0/q0`
  are driven by the regularizer instead of the objective and MMA converges to a
  load-scale-DEPENDENT, much worse design. Measured on the driver's self-weight
  bracket BEFORE the fix: MMA vs OC compliance `rel = +121% / +234% / +186%` at
  gravity 1e-6 vs `+0.6% / +0.6% / −5.4%` at gravity 1.0 (MMA's design changed
  with the load scale — a real bug). AFTER the fix: rel is IDENTICAL at 1e-6 and
  1.0 gravity (`+0.61% / +0.57% / −6.01%`) — MMA is scale-invariant like OC.
- Scaling `dc` by a positive constant leaves the MMA subproblem optimum unchanged
  (the dual multiplier absorbs it); only the `raa0` balance is fixed. The
  rmin-2.5 OC parity is preserved (`test_mma` still 0.385% / 0.103% / 0.766%).
  This is the "move-limit / convergence behavior" warning M7.mma.1's handoff
  flagged for the switchover — resolved by normalization (no GCMMA needed; MMA
  converges reliably and in FEWER iterations than OC — see the table).

### 3. Production default = MMA (`core/include/topopt/pipeline.hpp`, `core/src/simp/minimize_plastic.cpp`)
- New `MinimizePlasticOptions::updater = SimpUpdater::MMA` (the switchover). The
  driver sets `opt.updater = options.updater` per rung, overriding `simp.updater`
  (documented alongside the existing "`simp.progress`/`simp.cancel` are
  overridden" note). Set `options.updater = SimpUpdater::OC` to fall back.
- **Why the driver field, and NOT flipping the global `SimpOptions::updater`
  default:** the global default stays OC so the projection-locked **Gate V2**
  path and every direct `simp_optimize` test (`simp`, `variants`, `v3`, `mask`,
  `gate_v2`) are **byte-identical OC** — fixture discipline fully preserved, zero
  gate touched. Flipping the global default would (a) make Gate V2 THROW, since it
  sets a projection schedule and MMA+projection is rejected/out-of-scope, and
  (b) force pinning ~5 gate files to OC. The switchover's scope is the production
  driver ("the masked path, the stress path, the multi-load path"), all of which
  are driver-reachable; the projected Gate-V2 chain is explicitly out of scope
  ("do NOT change the filter"). OC remains selectable via `SimpOptions.updater =
  OC` at the `simp_optimize` API level, unchanged. (C++17 also can't cleanly give
  `MinimizePlasticOptions.simp` a per-field MMA default — designated-init DMIs are
  C++20 — so the explicit driver field is the portable expression of the same
  intent.)
