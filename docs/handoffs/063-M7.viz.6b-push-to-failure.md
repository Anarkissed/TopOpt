## The dependency gate — VERIFIED PASS (flex is alive, NOT blocked)
The push scrub renders `displacementField × pushExaggeration`. If `displacementField`
were empty/all-zero for real variants, the scrub would be silent (the failure mode to
avoid). I proved it is populated on a REAL optimize (resolution-6 cube, `minimizePlastic`),
via a throwaway diagnostic (`XCTFail` dumping the field stats, since `xcodebuild`
swallows `print`):

```
DIAGPROBE variants=4 accepted=4 dispCounts=[1029, 1029, 1029, 1029] dispNonZero=[852, 780, 714, 591]
```

- 4 accepted variants; `displacementField.count == 1029 == 3·343 == 3·7³` — correct
  per-NODE DOF sizing for a 6-voxel grid (7 nodes/side). NOT per-voxel.
- 591–852 non-zero entries per variant — the part genuinely deflects. The flex/push
  motion engine has real data. **viz.6b is not blocked.**

I then made this a PERMANENT, non-vacuous guard in the existing real-optimize
integration test (see below), so a future core regression that empties the field fails
loud instead of silently killing the animation.

## What I changed (all `/app/`, 4 files, +83/-3)
- `app/TopOptKit/Tests/TopOptKitTests/TopOptKitTests.swift` — in the real-optimize
  `testMinimizePlasticResultsFields`, assert every accepted variant's
  `displacementField` is (a) exactly `3·(nx+1)(ny+1)(nz+1)` (per-node DOF sizing) and
  (b) contains a non-zero finite component (the part deflects). Added a
  `checkedAccepted > 0` guard so the per-variant block can never pass vacuously (the
  loop is `where v.accepted`). This is the headless proof that viz.6b's motion engine
  has data, and a regression guard.
- `app/TopOptKit/Tests/TopOptFlowsTests/ResultsModelTests.swift` — two new headless
  tests hitting the acceptance directly:
  - `testPushScrubSweepMonotonicDeflectionAndYieldTrigger`: sweeps the scrub across
    `[1, multiplier]` and past it, asserting the rendered `pushFlexScale()` STRICTLY
    INCREASES at every step below yield (the part visibly flexes more), `atFailure` is
    false below the multiplier and true at/above it (the yield-threshold trigger), and
    the deflection at failure is exactly `multiplier ×` the 1× deflection (linear FEA).
  - `testPushReadoutReadsNaturallyAndFlipsAtYield`: pins the scrub readout string.
- `app/TopOptKit/Sources/TopOptFlows/ResultsModel.swift` — new pure, headlessly-testable
  `pushReadout(prediction:)`: the live scrub label "`<load> · <mult>× load`", flipping
  to "`<load> · <mult>× · YIELDS`" at/above the multiplier.
- `app/TopOptKit/Sources/TopOptFlows/ResultsScreen.swift` — the push slider's live
  readout now calls `model.pushReadout(...)` instead of the inline "`<load> of
  <failLoad>`" string, so the moving readout reads like the acceptance's own example
  ("1180 lbs · 58× load" → "1915 lbs · 94.9× · YIELDS"): live load + live multiple in
  one natural line, YIELDS prominent at failure. Pure label change — no behavior/logic
  change; the multiplier/exaggeration/marker paths are untouched.

## Why the readout change (and why it's the only UI edit)
The acceptance's "scrub should read naturally" clause gives a verbatim example format.
The 062 readout showed the live load but kept the multiple only in the panel header, so
the moving line didn't match the example. Consolidating them into one testable model
string is a minimal, in-scope readability fix. I deliberately did NOT restyle the
working panel/marker/slider (062's device-QA'd surface) — feel/pixels are maintainer
device QA per the task.

- Crossing the multiplier lights up the failure region — YES. `atFailure` flips exactly

- Motion engine alive on a real optimized part — YES (displacement field populated,
