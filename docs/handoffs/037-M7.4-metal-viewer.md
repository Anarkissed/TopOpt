### Where normals come from (a deviation, documented)
The ROADMAP parenthetical says "bridge supplies vertices/normals/face ids". The
bridge (`ImportedMesh`) supplies vertices, indices and per-triangle face ids, but
**not** normals — the core `TriangleMesh` is a welded triangle soup with no
normals (`core/include/topopt/mesh.hpp`), so the bridge has nothing to forward.
Rather than expand the fragile POD-only C++ bridge surface (see run 034's four
module/link addenda) and rebuild the xcframework, the viewer **derives** smooth
per-vertex normals from the supplied vertices+indices (area-weighted, the standard
estimate). This keeps the change Swift-only, keeps the bridge header untouched,
and makes the normal math **headlessly unit-testable** (a GPU can't run in the
macOS test slice). If the maintainer prefers OCCT's true surface normals for STEP
parts later, that is a separate bridge change.

### Testable core (pure simd, no GPU) — new files in `Sources/TopOptFlows/`
- **`ViewerMesh.swift`** — `MeshBounds` (min/max/center/bounding-sphere radius),
  `MeshGeometry.bounds(vertices:)` and `MeshGeometry.vertexNormals(vertices:
  indices:)` (area-weighted, unit-normalized, out-of-range/degenerate-safe), and
  `ViewerMesh` (positions + derived normals + `UInt32` indices + face ids +
  bounds, with an `interleaved()` `[px,py,pz,nx,ny,nz]` buffer for Metal).
- **`OrbitCamera.swift`** — a turntable camera: `target`, `distance`, `azimuth`,
  `elevation` (clamped near the poles), `fovY`, zoom limits. `frame(_:)` centres
  on and fits a `MeshBounds`; `orbit(dx:dy:)` and `zoom(_:)` (both clamped);
  right-handed `viewMatrix()` / `projectionMatrix(aspect:)` (Metal [0,1] depth)
  and a `normalMatrix()`. All simd value math, no MTKView dependency.

### Metal render layer (GPU + gestures — maintainer device QA) — `Sources/TopOptFlows/`
- **`MetalMeshView.swift`** — `MeshRenderer` (`MTKViewDelegate`: builds the
  interleaved vertex + index buffers, compiles a matcap-style shader from an
  embedded MSL source string at runtime so no `.metal` resource bundling is
  needed, draws depth-tested indexed triangles, uploads the camera matrices per
  frame) and `MetalMeshView`, a `UIViewRepresentable` (iOS) / `NSViewRepresentable`
  (macOS) over `MTKView` wiring pan→orbit and pinch/magnify→zoom. Dark stage clear
  color = `DS.Color.background`; on-demand drawing (`isPaused`+`enableSetNeedsDisplay`)
  for battery. A plain-SwiftUI fallback under `#else` keeps the module compiling
  where MetalKit is unavailable. The shading is analytic ("matcap-style" without a
  matcap texture asset): a view-space key + fill + accent-tinted fresnel rim.

### Tests (written first) — `Tests/TopOptFlowsTests/ViewerTests.swift`
20 XCTest cases, all GPU-free: bounds of a unit cube + empty; single-triangle
normal == +z; cube smooth normals are unit and point outward; out-of-range indices
don't trap; `ViewerMesh` shape/interleave/empty; camera framing centres+fits and
survives an empty mesh; orbit turns azimuth by the sensitivity; elevation clamps at
both poles; init clamps elevation; zoom clamps to both limits and ignores
non-positive factors; eye is `distance` from target; the view matrix maps eye→origin
and target→(0,0,−distance); the target projects to clip centre with w>0; the normal
matrix columns are orthonormal; and the AppModel seam retains geometry on import and
clears it on a non-watertight reject.

## What I did NOT do
- **Did NOT add normals to the bridge.** Normals are derived viewer-side from the
  bridge's vertices+indices (see "Where normals come from"). The bridge header and
  `bridge.cpp` are untouched; the vendored xcframework was rebuilt by
  `build_core.sh` only to run the tests, not changed in git.
- **Did NOT bundle a matcap texture.** The shading is an analytic matcap-style
  approximation (key + fill + fresnel rim) so no binary image asset / new resource
  is introduced. Exact look is maintainer device QA against the design.
- **Did NOT run the viewer on a GPU/device here.** The Metal draw + gestures need a
  display; they are compiled (both iOS slices + the macOS test slice) but not
  rendered. Visual fidelity, orbit/zoom *feel*, and gesture tuning are maintainer
  device QA (M7's stated /app/ standard). Only the GPU-free math is asserted.
- **Did NOT build the tools palette, face selection, force arrows, or Optimize.**
  Those are M7.5–M7.7; the workspace shows only the viewer stage, the back+chip
  chrome, and the orbit hint. The bottom tools bar / left Selections panel / top
  Optimize button from the design are deliberately absent this task.
- **Did NOT render anything for a recent project opened from Home.** Recents carry
  no file/geometry (handoff 036: in-memory demo fillers), so opening one shows the
  dark empty stage. A fresh import → Continue → workspace renders the part. Wiring
  recents to real sessions is a later (persistence) concern.
- **Did NOT change `open(_:)`/navigation.** Only `importFile`/`newTopOpt` set/clear
  the new `importedMesh`.

## Warnings for the next run
- **Run `./app/scripts/build_core.sh` before any build/test** (regenerates the
  git-ignored `vendor/` xcframework). Unchanged from M7.1–M7.3.
- **Use `-scheme TopOptKit-Package` (or `swift test`)** for the test action; the
  per-product schemes are build-only (M7.2 warning still holds).
- **`WorkspacePlaceholder` is now the real viewer** despite the legacy name (kept
  so the existing `testScreensInstantiate` and `RootView` don't need editing). If
  M7.5+ renames it, update `RootView` and that test together.
- **`OrbitCamera.orbit` gesture sign/feel is untuned.** The sensitivity (0.01
  rad/pt) and the iOS/macOS drag-axis signs are placeholders; adjust on-device.
  macOS uses `-t.y` (AppKit y-up) vs iOS `+t.y` — expect to revisit during QA.
- **Culling is off (`.setCullMode(.none)`)** in v1 so a part with inconsistent
  winding is never invisible. If M7.5 needs back-face culling for the id pass,
  confirm the core meshes wind outward first (the smooth-normal outward assumption
  is only asserted on the synthetic cube fixture, not on real imports).
- **STEP is macOS-only** (no iOS OCCT slice, M7.1). On iPad the viewer renders STL
  imports; a STEP pick still returns the core's "not available on this platform"
  toast before it reaches the viewer.
- **The `ViewerMesh` is rebuilt on a `meshID` of vertex+triangle counts.** Two
  different parts with identical counts would not re-frame; acceptable for v1
  (single import per workspace entry), but M7.5+ should key on a stronger identity
  if it supports swapping the part in place.
