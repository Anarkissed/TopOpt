## What I did
Added the job-report data model, its single-JSON serializer, and the schema
validator that defines the report's on-disk shape. Scope is the report module
(data + serialization + validation) plus the worst-case stress-margin
computation whose definition is locked in `settings/rules.json`
(`semantics_locked.margin_definition`, DECISIONS.md 2026-07-08). Nothing in FEA,
SIMP, orient, settings, mesh, voxel or IO changed behaviour; every prior test is
bit-identical. Pure C++/std — no OCCT, no Eigen, no new dependency — so it lives
in the always-built library and its test runs in every configuration (verified
in the dependency-light build too).

New public API — `core/include/topopt/report.hpp` (Eigen-free, OCCT-free):
- `struct StressMargin { double in_plane, interlayer, worst_case; }` — the three
  dimensionless safety factors of the locked `margin_definition`. `worst_case`
  is the number the M5.1 settings engine consumes.
- `StressMargin compute_stress_margin(yield_strength_mpa, z_knockdown,
  max_von_mises_stress, max_interlayer_tension)` — computes exactly:
  `in_plane = yield / vm`, `interlayer = (z_knockdown * yield) / interlayer_tension`,
  `worst_case = min(...)`. A stress of exactly 0 means that failure mode carries
  no load, so its term is `+infinity` and the min falls back to the loaded term
  (e.g. a flat-printed part under pure in-plane load has 0 interlayer tension →
  interlayer margin unbounded → worst_case = in_plane). For resin
  (`z_knockdown == 1`) the two terms scale identically. Throws `ReportError` if
  `yield` is not finite/`> 0`, `z_knockdown` ∉ (0,1], or either stress is not
  finite/`>= 0`.
- `struct VariantReport { double volume_fraction, max_stress_mpa,
  max_interlayer_tension_mpa; StressMargin margin; Vec3 orientation;
  SlicerSettings settings; }` — one variant's line. `volume_fraction` is the
  achieved physical fraction (SimpOptimizeResult.volume_fraction); "volume
  saved" is the complement `1 - volume_fraction`, emitted (derived) in the JSON.
- `struct JobReport { std::string material; std::vector<VariantReport> variants; }`.
- `std::string job_report_json(const JobReport&)` — the single JSON document
  (2-space indented). Emits `volume_saved_fraction = 1 - volume_fraction`
  per variant; non-finite doubles (unbounded margin terms) become JSON `null`;
  all output satisfies the validator.
- `void validate_job_report_json(const std::string&)` — parses and schema-checks
  the document; throws `ReportError` on malformed JSON or any schema violation.
  This function IS the concrete schema.
- `class ReportError : public std::runtime_error`.

Implementation — `core/src/settings/report.cpp` (new TU in the always-built
library). Carries a module-local minimal JSON parser for the validator (mirroring
`materials.cpp` / `settings.cpp`; ARCHITECTURE §4 "no JSON library") and a small
hand-rolled serializer. `<stdexcept>` reaches the throw site through
`report.hpp`, which includes it directly (the portability grep below prints
nothing).

**Layout note (not a conflict).** ARCHITECTURE §3's `/core/src/` module list
does not enumerate a `report/` directory, but §5's pipeline explicitly ends with
"… + JSON report" and §9 M5 lists "JSON report format" as a deliverable. To
avoid inventing a new top-level module dir not in the locked layout, the report
`.cpp` is housed in the existing §3-listed `settings/` module — the report is
tightly coupled to settings (it embeds `SlicerSettings`, and the margin it emits
is defined by `settings/rules.json`). The public header `report.hpp` sits flat in
`include/topopt/` alongside every other public header. Nothing contradicts the
locked layout or the technology table.

Design choices and WHY:
- **The report module holds data + serialization + validation only; it does NOT
  run the pipeline.** Producing `max_von_mises`, `max_interlayer_tension`, the
  chosen `orientation`, and the `SlicerSettings` requires FEA/orient/settings
  runs; wiring those into an end-to-end run is M5.3 (`minimize_plastic`
  end-to-end). Keeping report Eigen-free and OCCT-free preserves the codebase's
  always-built/dependency-light discipline (same reason settings and orient are
  Eigen-free) and lets the schema test run in every configuration.
- **"Volume saved" is derived, not stored.** `VariantReport` stores the achieved
  `volume_fraction`; the serializer emits both it and `volume_saved_fraction =
  1 - volume_fraction`, and the validator re-checks the identity (a real schema
  tooth). This makes the two fields impossible to drift.
- **Margin computation lives here because the report field lives here.** Handoff
  025 flagged M5.2 as where `worst_case_stress_margin` gets computed per the
  locked definition. `compute_stress_margin` is a pure helper over already-solved
  stress maxima + material properties; it does not solve FEA.
- **Non-finite → JSON null.** JSON has no Infinity/NaN. An unbounded margin term
  (zero stress in a mode) is emitted as `null`; the validator accepts
  number-or-null for the three margin fields. `max_stress`/`volume`/`orientation`
  are required to be finite numbers.
- **The validator is lenient on unknown extra keys** (like the settings loader),
  so M5.2b can add `min_feature_violations` to the per-variant report without
  breaking this schema.

### How the tests are set up and why they are trustworthy
`report` (new, `tests/unit/test_report.cpp`, always-built pure-C++/std, public
API only, **69 checks**):
- **`compute_stress_margin`**: numeric values with in-plane governing and with
  interlayer governing; the unbounded cases (vm=0 → in_plane ∞; interlayer
  tension=0 → interlayer ∞; both 0 → worst_case ∞); resin k=1 coincidence; and
  every validation throw (yield ≤0/inf, k=0/>1/negative, negative/NaN von Mises,
  negative/inf interlayer; k=1 allowed).
- **Serialize→validate round trip**: an FDM + a resin variant serialize to a
  document that passes the validator; targeted assertions on emitted keys and
  settings content for both families. Independent confirmation the output is
  valid JSON was done manually (Python `json.load`, throwaway).
- **Derived volume saved**: `volume_fraction 0.3` emits `volume_saved_fraction
  0.7` and validates.
- **Non-finite margin → null**: a variant with 0 interlayer tension emits
  `"interlayer": null` and still validates.
- **Empty variants** array validates; string **escaping** (quote, backslash,
  newline, tab in material name / warning) round-trips and validates.
- **Validator teeth**: a canonical document validates, and mutating exactly one
  thing is rejected — garbage JSON, missing `material`/`variants`, wrong-typed
  `material`/`variants`, a variant missing `settings`, `volume_saved_fraction`
  inconsistent with `volume_fraction`, non-integer `walls`, string
  `margin.worst_case`, out-of-range `volume_fraction`, `orientation` missing a
  component, negative `max_stress`; plus a positive control that `null` margin
  is accepted.

## Warnings for the next run
- **The report `.cpp` lives in `core/src/settings/`, not a `report/` dir.**
  ARCHITECTURE §3 does not list a report module directory; rather than invent an
  unlisted top-level module, the code is housed in the §3-listed `settings/`
  module (the report is coupled to settings and the margin is defined by
  `rules.json`). The public header is `include/topopt/report.hpp` (flat, like
  every other header). If the maintainer wants a dedicated `report/` dir, that is
  a §3 layout change (DECISIONS.md), not an agent call.
- **A margin term is `+infinity` (JSON `null`) when its stress is 0.** This is
  the correct realistic case for a flat-printed part under pure in-plane load
  (0 interlayer tension → unbounded interlayer margin → `worst_case = in_plane`).
  A report consumer must treat a `null` margin field as "unbounded", not
  "missing". `worst_case` is finite as long as at least one failure mode is
  loaded; a fully-unloaded variant emits `worst_case: null`.
- **`volume_fraction` is the ACHIEVED physical fraction; "volume saved" is its
  complement `1 - volume_fraction`.** The validator enforces the identity, so a
  caller must set `volume_fraction` to the SimpOptimizeResult value (not to
  "volume saved"), or the schema check will reject the emitted document.
- **The validator is lenient on extra keys but strict on the listed fields'
  types/ranges.** It requires `settings` to carry all eight fields (FDM + resin),
  so a resin variant emits `walls: 0 … infill_pattern: ""` (the full
  `SlicerSettings` is always serialized, per M5.1's "branch on family" note).
- **`report` adds ~0.05 s** (pure serialization/parse, no solve); full suite
  ~116 s locally, well under the M2.3 "< 5 min CI budget".
