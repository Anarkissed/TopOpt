### Placement — a new SwiftPM target `TopOptFlows`, following the M7.2 pattern
The flow is a new **`TopOptFlows`** target/product in `app/TopOptKit/Package.swift`
(depends on `TopOptKit` + `TopOptDesign`). Rationale: the M7 /app/ verification
standard is a headless `xcodebuild test`; putting the flow's logic in a package
target (not the app target, which needs a booted simulator and isn't in the test
scheme) makes it unit-testable on macOS. Same reasoning M7.1/M7.2 used. Because
`TopOptKit` is built with C++ interop, `TopOptFlows` and `TopOptFlowsTests` set
`.interoperabilityMode(.Cxx)` too (a Cxx-interop Swift module forces its
consumers to enable interop — handoff 034 addendum 3).

### The testable core — `Sources/TopOptFlows/AppModel.swift` + `FlowModels.swift`
`AppModel` (`@MainActor ObservableObject`) is the flow state machine, holding all
logic so the SwiftUI screens stay thin renderers:
- **Materials via the bridge:** `loadMaterials()` calls `TopOptKit.loadMaterials`
  on the bundled materials.json and splits by `family` ("fdm"/"resin") into
  `fdmMaterials` / `resinMaterials`, seeding each family's default selection.
  A load error toasts the diagnostic and leaves the dropdowns empty.
- **Import with core diagnostics:** `importFile(atPath:displayName:)` calls
  `TopOptKit.importMesh`. A hard core failure (unreadable/unparseable / STEP
  without OCCT) is caught and its `what()` toasted verbatim; a mesh that parses
  but is **not watertight** is rejected with a toast ("… isn't watertight …")
  and does **not** become the draft. Only a watertight import becomes
  `importedFile`.
- **Continue gating + navigation:** `canContinue` = watertight file + selected
  material; `continueToWorkspace()` records a `RecentProject`, sets the derived
  project name, and switches `screen` to `.workspace`. `newTopOpt` / `cancelImport`
  drive the sheet; `open(_:)` opens a recent (restoring process + material);
  `backHome()` returns.
- Bridge calls are **injected** (default to the real `TopOptKit`) so tests drive
  the real fixtures and force the error paths deterministically.
`FlowModels.swift`: `Screen`, `ProcessKind` (fdm/resin + design labels),
`ImportedFile`, `RecentProject`, `MaterialOption` — plain value types, no SwiftUI
/ bridge types, so the logic is headlessly testable.

### Screens (each with a `#Preview`) — `Sources/TopOptFlows/`
- **`HomeView`** — the design's HOME: header (hex logo + "TopOpt" + tagline +
  device chip) over an adaptive card grid whose first tile is "New TopOpt"
  (opens the sheet), followed by `RecentProjectCard`s.
- **`ImportSheet`** — the design's IMPORT SHEET: title/subtitle, a file row with
  a **Browse…** button, the FDM/Resin `SegmentedGlass`, the material dropdown
  (`Menu` over `currentMaterials`), and `PillButton` Cancel / Continue (Continue
  disabled until `canContinue`). Browse uses SwiftUI's **`.fileImporter`** (its
  wrapper over `UIDocumentPickerViewController` on iPadOS) limited to STEP/STL
  by extension; the pick is copied into an app-owned temp path (the
  security-scoped URL is only valid during access) and imported via the model.
- **`WorkspacePlaceholder`** — the Continue destination: the design's top-left
  Back + project/material chip over an empty stage. Deliberately minimal — the
  real workspace (viewer/tools/optimize) is M7.4–M7.7 and will replace it.
- **`RootView`** — the shell: switches Home ⇄ Workspace, presents the sheet over
  the design's dimmed+blurred scrim, shows the toast, loads materials on appear.

## What I did NOT do
- **Did NOT persist recent projects across launches.** `recentProjects` is
  in-memory for the session; the design's project cards are demo fillers.
  Cross-launch persistence (a store on disk) is deliberately out of scope for
  M7.3 (no speculative infrastructure) — a later concern, likely alongside the
  M7.4+ workspace/session model. The Home grid renders whatever recents exist
  (empty-state = just the "New TopOpt" tile).
- **Did NOT use the design's hard-coded marketing material names**
  ("PLA — Matte Black", "Tough 2000", etc.). The ROADMAP requires the dropdown
  be "populated from materials.json **via the bridge**", so the options are the
  real material names (PLA, PETG, …, Resin_Standard, Resin_Tough) split by
  `family`. The HTML list is mockup filler; the ROADMAP wins.
- **Did NOT build the real workspace.** Continue lands on `WorkspacePlaceholder`
  (Back + project/material chip only). The Metal viewer, face selection, force
  arrows and Optimize are M7.4–M7.7 and will replace it.
- **Did NOT wrap `UIDocumentPickerViewController` directly.** I used SwiftUI's
  `.fileImporter`, which *is* the document picker on iPadOS, because it compiles
  cross-platform (the package must also build for the macOS test slice) and needs
  no `UIViewControllerRepresentable`. STEP/STL are matched by filename extension
  (neither has a system UTType); the picker falls back to `.data` if a device
  lacks the declared types.
- **Did NOT surface a byte size / "solid body" count** in the file row (the
  design shows "2.1 MB · 1 solid body · watertight ✓"). The row reports what the
  core actually measured — triangle count + watertight flag; byte size/solid-body
  counting isn't exposed by the bridge and isn't needed to gate the flow.
- **Did NOT compile the SwiftUI `#Preview`s into a running app**, and did NOT do
  on-device visual QA. Previews are for maintainer side-by-side QA against the
  HTML; the token/logic *values* are what the tests verify. Exact glass/blur
  fidelity is maintainer QA (the M7.2 caveat carries over).
- **Did NOT run the OCCT/lib3mf-gated core paths differently** — no /core/ change;
  the core suite is re-confirmed 25/25 unchanged.

## Warnings for the next run
- **Run `./app/scripts/build_core.sh` before any build/test** (regenerates the
  git-ignored `vendor/` xcframework). Unchanged from M7.1.
- **The package now vends THREE products** (`TopOptKit`, `TopOptDesign`,
  `TopOptFlows`). Use `-scheme TopOptKit-Package` (or `swift test`) for the test
  action; the per-product schemes are build-only (M7.2 warning still holds).
- **`AppModel` is `@MainActor`.** Its callers (the SwiftUI views, the tests) run
  on the main actor; a background caller must hop. The M7.7 run screen will call
  `minimizePlastic` off the main thread — keep the *model mutation* on the main
  actor and only the optimizer work in the background.
- **The document-picker path (`ImportSheet.handlePick`) is not covered by the
  headless tests** — `.fileImporter` needs a UI. Tests exercise the model's
  `importFile(atPath:)` seam directly; the security-scope + temp-copy glue around
  it is maintainer device QA. Verify a real STEP/STL pick on device.
- **STEP import is macOS-only** (no iOS OCCT slice, M7.1). On iPad the picker
  offers STEP but a STEP import returns the core's "not available on this
  platform" diagnostic in a toast. Bringing STEP to iOS is the same separate
  effort M7.1 flagged.
- **Recent-projects order/dedup is minimal** (`insert(at: 0)`, no dedup, no cap).
  When persistence lands, revisit dedup-by-name and a cap.
- **`ProcessKind.resin`'s segment label is "Resin (SLA)"** but the material
  `family` string is `"resin"` — the two are intentionally different (UI label vs
