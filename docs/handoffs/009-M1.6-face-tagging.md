## What I did
Added the ability to select a B-rep face by id and tag the solid voxels flush
against it as LOAD or FIXTURE. Two pieces: the STEP importer now records which
face each tessellation triangle came from, and a new `tag_step_face` maps a
face id + grid to the voxel slab against that face.

Public API:
- `core/include/topopt/voxel.hpp`: extended `VoxelTag` with `Load = 4` and
  `Fixture = 5` alongside the existing `Empty/Interior/Surface/UserTagged`
  (M1.5 handoff explicitly left "split UserTagged into Load/Fixture" open for
  M1.6). The voxelizer still only ever emits Interior/Surface/Empty; the new
  values are only ever set by `tag_step_face`. `UserTagged` is unchanged (still
  used by the M1.5 `voxelize` test).
- `core/include/topopt/step.hpp`:
  - `StepModel` gains `std::vector<int> triangle_face`, parallel to
    `mesh.triangles`: `triangle_face[t]` is the 0-based
    `TopExp_Explorer(TopAbs_FACE)` index of the face triangle `t` was
    tessellated from — the same ordering `face_count` enumerates, so ids run
    `[0, face_count)`. This is what lets a caller name "a STEP face" after the
    per-face triangulations are welded into one mesh.
  - New `tag_step_face(VoxelGrid&, const StepModel&, int face_id, VoxelTag)`:
    tags every solid voxel whose centre is within half a voxel edge (+ numeric
    eps) of the closest point on one of face `face_id`'s triangles. Returns the
    count tagged. Throws `std::invalid_argument` if the tag is not Load/Fixture,
    the id is out of `[0, face_count)`, or `triangle_face` is not parallel to
    the mesh. The header includes `topopt/voxel.hpp` for the grid/tag types (no
    OCCT — voxel.hpp is OCCT-free, so step.hpp stays OCCT-free too).

Implementation:
- `core/src/io/step.cpp`:
  - In the tessellation weld loop, a `face_id` counter advances once per
    `TopAbs_FACE` (in the same TopExp order used to compute `face_count`); each
    welded triangle pushes its `face_id` into a parallel vector that becomes
    `model.triangle_face`. Faces with no triangulation still consume an id (they
    own zero triangles), keeping ids aligned with `face_count`.
  - `tag_step_face`: iterates the solid voxels, and for each computes the
    squared point-to-triangle distance (Ericson closest-point-on-triangle, all
    vertex/edge/face Voronoi regions) to that face's triangles, comparing
    against `(spacing/2)²`. Only solid voxels are considered (LOAD/FIXTURE mark
    material). A planar face flush with the solid boundary selects exactly the
    one-voxel-thick slab: surface-voxel centres sit `spacing/2` from the face;
    interior voxels sit `>= 1.5·spacing` away and are never within half a voxel.
    The closest-point routine is a `static` helper in step.cpp's anonymous
    namespace (used only here — no speculative public helper).

Why this shape: the operation is inherently STEP-bound (it selects a *STEP face
id*), and a `StepModel` only ever comes from the OCCT importer, so both the
per-face map and `tag_step_face` live in the OCCT-gated `io/step` translation
unit. `voxel.hpp` only gained the two tag values; no new generic geometry
surface was added.

Tests — `core/tests/unit/test_tag.cpp`, registered as ctest `face_tag`
(self-contained CHECK harness, no framework, §4; OCCT-gated like `step_import`
because it imports the STEP cube). It drives the committed `cube.step` fixture,
identifies the +Z (z==10) and −Z (z==0) faces purely from the geometry of each
face's triangles (via `triangle_face`) rather than hard-coding OCCT's face
order, then asserts the tagged sets. `STEP_FIXTURE_DIR` is injected by CMake
(same pattern as `step_import`). Fixtures unchanged.

Build:
- `core/CMakeLists.txt`: added a `test_tag` executable + `add_test(face_tag)`
  inside the existing `if(OpenCASCADE_FOUND)` block (needs OCCT to import STEP),
  with `STEP_FIXTURE_DIR` injected. CI (`-DTOPOPT_REQUIRE_DEPS=ON`) always has
  OCCT, so `face_tag` always runs in the pipeline; the OCCT-free local config
  skips it (verified below). CI's `ctest` step picks it up with no CI change.

New tests added (in `face_tag`):
- `triangle_face` well-formed: parallel to `mesh.triangles` (12 entries), every
  id in `[0, 6)`, and each of the 6 planar cube faces owns exactly 2 triangles
  — proves the importer's per-triangle face map.
- top face (z==10) tagged LOAD: count == `nx*ny` (one full cross-section), and a
  voxel is Load **iff** it is in the top layer `k == nz-1` — "exactly the
  expected slab, nothing else". Each Load voxel is still `solid()`.
- bottom face (z==0) tagged FIXTURE: count == `nx*ny`, Fixture **iff** `k == 0`,
  and Fixture is a distinct tag from Load.
- LOAD + FIXTURE on one grid: the top-LOAD and bottom-FIXTURE slabs coexist with
  their reported counts (neither overwrites the other), equal in size on a cube.
- argument validation: a non-Load/Fixture tag (Surface, Interior), a negative
  id, and `id == face_count` each throw `std::invalid_argument`.

## What I did NOT do
- Did not add per-face *area/normal* accessors, a general "tag by arbitrary
  geometry/STL face", or a public generic voxel-tag-by-triangles helper. The
  task selects a *STEP* face id; the closest-point helper is a private static in
  step.cpp (§8: no speculative abstraction).
- Did not tag the cylinder's curved face or the STEP cylinder at all. The task's
  own success criterion is the cube slab; a curved face's "slab" is not a single
  planar layer and no fixture asserts it. `tag_step_face` will still run on a
  curved face (it is pure geometry), but its behaviour there is untested here.
- Did not de-duplicate overlapping slabs or define precedence when two faces
  claim the same voxel (e.g. a shared edge voxel). `tag_step_face` writes the
  requested tag; a later call for an adjacent face would overwrite that voxel.
  For the cube's top/bottom faces the slabs are disjoint, so this does not
  arise in the test. A voxel's single tag byte holds one role at a time.
- Did not preserve the prior Interior/Surface classification once a voxel is
  tagged Load/Fixture — the tag byte is overwritten (a Load voxel reports
  `Load`, not `Surface`). It is still `solid()`. If a later stage needs both the
  boundary class and the load/fixture role at once, that is a separate field it
  can add; M1.6 only needs the role stored.
- Did not touch `/app/`, `ARCHITECTURE.md`, `materials.json`, CI config, any
  fixture, or any existing test/assertion/tolerance. `voxelize`/`step_import`
  assertions are unchanged; adding enum values and a `StepModel` field does not
  alter them.

## Warnings for the next run
- **Face ids are OCCT TopExp order, not semantic.** `triangle_face` ids match
  `TopExp_Explorer(TopAbs_FACE)`; which integer is "the top face" is not fixed
  across parts or OCCT versions. The test does NOT hard-code an id — it finds
  the +Z/−Z face from triangle geometry. Any real caller (M4 orientation, the
  app UI) must pick faces by geometry/UX, not by assuming a number.
- **`tag_step_face` needs a grid voxelized from `model.mesh`** (same origin /
  spacing / frame). Passing a grid from a different mesh will tag by stale
  coordinates. It does not (cannot) verify the grid came from that model.
- **Half-a-voxel selection is exact for planar faces flush with the boundary.**
  It relies on surface-voxel centres being `spacing/2` from the face and
  interior centres `>= 1.5·spacing`. For a face that does NOT reach the grid
  boundary, or a curved face, the selected set is whatever falls within half a
  voxel of the face triangles — reasonable, but not a clean single slab.
- **The tag overwrites the Interior/Surface class** (see "did NOT do"). Code
  that partitions the grid by Interior/Surface after tagging must account for
  Load/Fixture voxels being neither (they are still `solid()`). The existing
  M1.5 partition test runs on an untagged grid, so it is unaffected.
- `STEP_FIXTURE_DIR` is a compile-time absolute path baked in at configure time
  (same trade-off as the other tests): reconfigure if the repo is moved. A clean
  CI checkout always reconfigures.
