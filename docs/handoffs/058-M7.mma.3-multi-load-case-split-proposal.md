## Blocked

### Summary
M7.mma.3 bundles two topology-optimization capabilities that share only the
phrase "N load cases." One (weighted-sum) is a small, low-risk extension of the
CURRENT OC path; the other (worst-case / min-max) requires a new general
m-constraint MMA dual solver plus an auxiliary bound variable, carries real
convergence risk, and — if driven through `minimize_plastic` — crosses the
deliberately-deferred masked-MMA switchover boundary (M7.mma.4). Delivering both
correctly, byte-identically back-compatible, and green in one run is not
achievable without half-implementing one of them. **Requesting: re-slice
M7.mma.3 into two sequential tasks (agents cannot reorder/add ROADMAP tasks —
§8.4 / ROADMAP header — so this is a maintainer edit).**

### Proposed split
**M7.mma.3a — Weighted-sum multi-load-case (do first).**
- Objective `C(rho) = Σ_i w_i · c_i(rho)` over N load cases sharing the grid,
  material, BCs and volume constraint; each `c_i` is one FEA solve `K u_i = f_i`
  against the SAME penalized stiffness `K(rho)`, different RHS `f_i`. Gradient
  `dC/drho = Σ_i w_i · (dc_i/drho)` where each `dc_i/drho` is the ordinary
  compliance sensitivity from load case i's solve.
- **Stays on the OC path.** The summed sensitivity feeds the existing
  `oc_update` / `oc_update_masked` unchanged, so it works through
  `minimize_plastic` (which is OC-only today) with NO MMA and NO masked-MMA
  change — the interface freeze and the M7.mma.4 boundary are untouched.
- Back-compat is structural: with N = 1, w = 1 the accumulation is a single
  solve and a single sensitivity in the identical order → byte-identical to
  today's single-load path. Assertable directly (`== physical_density` /
  `== compliance` element-for-element against the current single-load call),
  mirroring the M7.disp exposure-equality test.
- Scope: a multi-load compliance/optimize path (masked + unmasked overloads or a
  `std::vector<LoadCase>` input), `MinimizePlasticOptions` gains a load-case
  list (self-weight stays the empty-list fallback, as `external_loads` does
  now), `minimize_plastic.cpp` recovers per-case stress and reports the
  worst-case margin over the cases, and the job.json schema gains a `load_cases`
  array (single-case job unchanged). Tests: back-compat equality + a weighted
  two-load behavioural check + schema round-trip.
- Estimate: this alone is ≈ one full run (touches optimize, driver, schema,
  back-compat, tests, plus the ~13-min heavy Eigen suite).

**M7.mma.3b — Worst-case / min-max multi-load-case (do second).**
- `min_rho max_i c_i(rho)` s.t. the volume constraint, via the standard MMA
  BOUND formulation: auxiliary scalar `z`, `min z` s.t. `c_i(rho) − z ≤ 0` for
  all i and `V(rho) − V* ≤ 0` → **m = N + 1 constraints** plus one extra
  variable. This is "min-max via MMA constraints" as the task words it.
- Requires a **general m-constraint MMA dual solver**. The existing code only
  has m = 1 (`mma_update`, single bisection, simp.cpp:439) and m = 2
  (`mma_update_stress`, NESTED bisection, simp.cpp:1431). Nested bisection does
  not generalise to arbitrary N; 3b needs a real dual solver (the z-stationarity
  gives `Σ_i λ_i = 1`, reducing the dual to the simplex × {μ ≥ 0} — a proper
  projected ascent / Newton, not nested 1-D bisection) and the auxiliary-`z`
  handling the current reduced MMA omits.
- **Crosses the M7.mma.4 boundary if driven through `minimize_plastic`.** The
  masked overload throws for MMA today ("the minimize_plastic/masked switchover
  is M7.mma.4", simp.cpp:1006). Min-max needs MMA, so either (i) it lifts that
  rejection on the masked path — which is M7.mma.4's job — or (ii) it ships as a
  standalone unmasked `simp_optimize_minmax` (the `simp_optimize_stress`
  precedent) tested directly, NOT wired into the driver's load-case list. That
  routing choice is a maintainer decision and should be pinned before 3b starts.
- **Convergence risk is real and precedented.** Handoff 055 (M7.mma.2) already
  documented plain-MMA failing to drive a single aggregated constraint to its
  target at CI-affordable resolution; N coupled compliance constraints in a
  min-max bound have the same non-conservative-convex-model fragility and may
  need GCMMA-style conservative approximations. 3b's test should therefore be a
  behavioural invariant ("multi-case optimum's worst-case compliance across both
  loads < the single-case optimum's"), never a pinned value.

### Why the task as one unit doesn't fit
1. Two different optimisation structures: 3a is single-objective (OC-friendly);
   3b is a constrained bound problem that hard-requires MMA + a new solver +
   an auxiliary variable. They do not share an implementation.
2. 3b's solver (general m-constraint MMA dual + `z`) is net-new and is the kind
   of piece that, per the M7.mma.2 precedent, needs its own convergence tuning
   and its own run to land green.
3. 3b entangles with M7.mma.4 (masked-MMA rejection). Deciding the routing
   (lift the rejection vs. standalone unmasked function) is a maintainer call.
4. Back-compat "single case = byte-identical" is a hard, assertable requirement
   that threads through every layer (optimize → driver → schema); it is cleanly
   satisfiable in 3a and should be locked there before 3b builds on it.

### On the "two-load fixture" and fixture discipline
The two-load test does NOT need a maintainer-authored numeric fixture: it is a
self-contained COMPARATIVE invariant built in code (like Scenario J in
`test_minimize_plastic.cpp` and the M7.mma.2 behavioural invariants) — optimise
under load-1-only, show that design is compliant under load 2, then show the
multi-case optimum's worst-case is lower. No pinned reference values, so no
`tests/fixtures/**` change and no maintainer fixture request. Flagging this so
the split tasks are worded to build the case in code (per the M7.mma.2

maintainer-generated fixture (fixture discipline, M6.3 / M7.mma.2) — but the
task as written ("optimum fails under the second load … multi-case holds both")
