New public API — `core/include/topopt/settings.hpp` (Eigen-free, OCCT-free):
- `struct SlicerSettings { std::string family; int walls, top_layers,
  bottom_layers, infill_percent; std::string infill_pattern; std::string
  print_mode; std::string warning; }` — the engine output. FDM fields are
  meaningful for `family == "fdm"`; `print_mode` for `family == "resin"`;
  `warning` is the selected band's warning ("" when none).
- `struct FdmBand`, `struct ResinBand`, `struct SizeModifier`,
  `struct FieldClamp`, `struct SettingsRules` — a faithful in-memory image of
  `rules.json`'s engine-relevant sections (bands, additive size modifiers,
  output clamps). Metadata/comment keys (`_comment`, `_rationale`, `note`,
  `semantics_locked`, `min_feature_warning`) are ignored on load.
- `SettingsError : std::runtime_error`.
- `constexpr double kSmallMaxDimensionMm = 50.0`, `kMediumMaxDimensionMm =
  150.0` — the size-class thresholds. These live only as prose in
  `rules.json` `semantics_locked.size_classes` (not as JSON numbers), so they
  are named constants here, cited to that locked text. See Warnings.
- `SettingsRules parse_settings_rules(const std::string& json_text)` /
  `SettingsRules load_settings_rules_file(const std::string& path)`.
- `SlicerSettings recommend_settings(const SettingsRules&, family, margin,
  max_part_dimension_mm)`.

Design choices and WHY (all governed by `rules.json` `semantics_locked`):
- **Band selection uses strict `<`** (`margin < band.margin_below`). The locked
  `band_boundary_rule` says a margin exactly equal to a band's `margin_below`
  belongs to the NEXT (higher) band — bands are `[previous, margin_below)`
  half-open. Strict `<` implements exactly that. The final band's
  `margin_below: null` is stored as `unbounded = true` and matches any margin,
  so a band is always selected.
- **Evaluation order = family gate → band → additive size modifier → clamps**,
  per `semantics_locked.evaluation_order`. Modifiers are added to the band's
  counts; clamps are applied afterward. `infill_pattern` is neither modified nor
  clamped (it is a name, not a count).
- **Resin returns `print_mode` only, no FDM fields, no size modifiers, no
  clamps.** `rules.json` `resin.note`: walls/top/bottom/infill are FDM concepts;
  resin parts print solid. The resin section carries no `size_modifiers`/
  `clamps`, so the engine applies none and leaves the FDM fields at 0/"".
- **The engine never encodes rule values.** All counts, thresholds, patterns,
  warnings and clamp bounds come from the parsed table. The only in-code numbers
  are the two size-class thresholds, which have no JSON home (see Warnings).
- **Input validation:** `family` must be exactly `"fdm"`/`"resin"`;
  `worst_case_stress_margin` finite and `>= 0`; `max_part_dimension_mm` finite
  and `> 0`. Throws `SettingsError` otherwise.
- **Structural parse validation:** each band array must be non-empty, ascend
  strictly by `margin_below`, and terminate in exactly one unbounded band;
  missing required numeric/string fields, a non-`[lo,hi]` clamp, or `lo > hi`
  throw. Extra metadata keys are tolerated (the table is a commented,
  human-tunable file).

### How the tests are set up and why they are trustworthy
`settings` (new, `tests/unit/test_settings.cpp`, always-built pure-C++/std,
public API only, **80 checks**). Boundary behaviour is asserted against BOTH the
committed human-seeded `rules.json` (injected path) and small synthetic tables:
- **FDM band boundaries** (thresholds 1.2/1.5/2.0/4.0) with a medium part (no
  modifier): each threshold tested at `t-ε` (lower band) and exactly `t`
  (crosses to the next band) — pinning the locked `band_boundary_rule`. Band 0
  full output + its warning; band 4 (unbounded) grid pattern; huge margin →
  band 4.
- **Size classes** (50/150 inclusive upper bounds): band 2 (well inside every
  clamp) at dim 50 (small, +5 infill), 50.001 (medium, unchanged), 150
  (medium), 150.001 (large, +1 wall +1 top) — the modifier deltas are visible
  unclamped.
- **Clamps**: small on band 0 (infill 60+5=65 → 60 ceiling); large on band 0
  (walls 6+1=7 → 6 ceiling; top 7+1=8 == ceiling, unchanged).
- **Resin**: low-margin band (solid + warning) and the 1.5 boundary → unbounded
  band (solid, no warning); FDM fields all 0/""; part size ignored.
- **Input validation**: unknown family, case-sensitivity (`"FDM"` throws),
  negative/NaN/inf margin, zero/negative/inf/NaN dimension all throw;
  `margin == 0` allowed → band 0.
- **Parsing/structure**: real table shape (5 fdm + 2 resin bands, unbounded
  terminators, clamp/modifier values); malformed JSON, missing file, missing
  `fdm`, non-unbounded terminal band, non-ascending bands, and a band missing a
  required field all throw; a well-formed synthetic table parses and evaluates
  (modifier + clamp path).

## Warnings for the next run
- **The size-class thresholds (50 mm, 150 mm) are named constants in
  `settings.hpp`, not JSON values.** `rules.json` defines them only as prose in
  `semantics_locked.size_classes` ("small <= 50; medium <= 150; large > 150"),
  which is not machine-readable, so the engine encodes them as
  `kSmallMaxDimensionMm` / `kMediumMaxDimensionMm` cited to that text. This is
  the one place the engine carries a numeric threshold. If the maintainer wants
  these tunable from the table, that is a `rules.json` schema change (add e.g.
  `size_classes: {small_max_mm, medium_max_mm}`) — a Blocked/DECISIONS item, not
  an engine change. They are treated as locked per the rules-file discipline.
- **Both band arrays MUST end with an unbounded band** (`margin_below: null`).
  The parser enforces this; a table whose last band has a finite `margin_below`
  is rejected rather than silently leaving high-margin inputs unmatched.
- **`SlicerSettings` FDM fields and `print_mode` are mutually exclusive by
  family** — a resin result has `walls==0 ... infill_pattern==""` and a non-empty
  `print_mode`; an FDM result has `print_mode==""`. A consumer (M5.2 report)
  should branch on `family`, not test individual fields for zero.
- **`infill_pattern` is not clamped or modified.** Only the four integer counts
  pass through size modifiers and clamps.
- **The module's JSON parser is module-local and lenient** (tolerates the
  table's `_comment`/`_rationale`/`note`/`semantics_locked`/`min_feature_warning`
  keys). It is NOT the strict, unknown-field-rejecting materials loader. If a
  future task needs strict validation of a settings file, that is a deliberate
  change, not a bug here.
- **`settings` adds ~0.05 s** (pure table lookup, no solve); full suite ~117 s
  locally, well under the M2.3 "< 5 min CI budget".
