# Handoff 055 — M7.viz.1: stress map scaled to the material limit (the honest heatmap)

## Task
M7.viz.1 "Stress map, scaled-to-limit (the honest heatmap). Replace any
data-range auto-scaling with a color scale keyed to the material limit (green =
comfortably below yield → red = at/above yield), shared across variants, with a
legend that states the material and the yield value it is scaled to. Consumes
existing fields only." (TRACK app / M7-VIZ block; the topmost unchecked task there.)

## What I did
Tests first (rule 3), then implementation. All changes are `/app/`-only, in
`app/TopOptKit/Sources/TopOptFlows/` + its test target — no `/core/`,
`tests/fixtures/**`, `materials.json`, `rules.json`, `ci/`, `ARCHITECTURE.md`,
`ROADMAP.md`, or `Package.swift`. No new dependency (`simd`/`SwiftUI` ship with
the OS). No C++ changed → the `<stdexcept>` portability grep is N/A (empty file set).

The M7.8 results screen already colored the variant mesh by the M7.0b per-voxel
von Mises field, but the shared scale was the **max von Mises across variants**
(data-range auto-scaling), so a given color meant a different physical stress on
every run. This task keys that scale to the **material yield** instead.

### Consuming the yield (existing fields only — interface freeze respected)
The material's `yield_strength_mpa` is already loaded by the app from
materials.json (`TopOptKit.Material.yieldStrengthMPa`, via the existing
`load_materials` bridge). No optimizer call, no new bridge field, no `/core/`
edit. The yield the app holds is the same value the core computed the report's
margins against.

- **`AppModel`**: retains the full material records in a private
  `materialsByName: [String: Material]` (previously only the names survived into
  `MaterialOption`), and exposes `yieldStrengthMPa(for: materialName) -> Double`
  (0 when unknown → relative-scale fallback).
- **`WorkspacePlaceholder`**: the `ResultsScreen` call site now passes
  `materialName: project.material` and `yieldStrengthMPa: model.yieldStrengthMPa(for:)`.

### The scale + legend (headlessly tested in `ResultsModel`)
- **`ResultsModel`** gains `materialName` + `yieldStrengthMPa` (init params,
  defaulted so existing call sites/tests are unaffected). `apply(_:)` now sets
  `stressScaleMaxMPa = yieldStrengthMPa > 0 ? yieldStrengthMPa : (data-range max)`.
  Because the scale is a constant material limit, it is identical across variants
  and unaffected by tab selection or a heavier variant streaming in. The existing
  `stressFraction(mpa:)` (divide-by-scale, clamped to [0,1]) then maps a von Mises
  value onto the ramp: **0 → blue (safe), yield → red (at/above)**. The existing
  `stressColor` ramp (blue→cyan→green→yellow→red, locked by
  `testStressColorRampEndpointsAndMidpoint`) is unchanged — only what
  `fraction == 1` *means* changed (now "at yield", not "run's data max").
- **`StressLegend`** (new value type) + `ResultsModel.stressLegend`: names the
  material and the yield the scale is keyed to, e.g. caption
  `"PLA · scaled to 55 MPa yield"`, `minLabel "0"`, `maxLabel "55 MPa (yield)"`,
  `scaledToYield` flag. In the no-material fallback the copy is honest
  (`"Relative scale (no material limit)"`).
- **`ResultsScreen`**: a `stressLegendPanel` (glass panel with the ramp bar +
  caption + 0/yield end labels) shown while the stress overlay is on. Pixels are
  device QA (the M7 /app/ standard); the legend copy is tested on the model.

### Why the data-range path still exists (deliberate, not a miss)
Two SHIPPED tests — `testStressSharedScaleAcrossVariants` and
`testStressTintsPerFlatVertexColored` — construct `ResultsModel` with **no
material** and assert the data-range max (20 / 10 MPa). Rules 1–2 forbid
modifying or weakening them. So the yield-keyed scale ACTIVATES when a yield is
supplied (production always does — `yield_strength_mpa` is a required
materials.json field), and the data-range max survives only as the no-material
fallback, which is exactly what keeps those two tests valid. In the app the
heatmap is always yield-keyed.

## Warnings for the next run
- **The data-range fallback is intentional and load-bearing for two shipped tests**
  (see "Why the data-range path still exists"). Do not "clean it up" by deleting it
  — production always supplies a yield, so the fallback only fires for the
  no-material test constructions and any legacy caller. Removing it would force
  modifying `testStressSharedScaleAcrossVariants` /
  `testStressTintsPerFlatVertexColored`, which rules 1–2 forbid.
- **Yield source is the app's material record (materials.json), keyed by
  `project.material`.** If materials haven't finished loading when a restored
  (persist-c) project opens, `yieldStrengthMPa(for:)` returns 0 and the legend shows
  the relative-scale fallback until the next construction. Rare; the material
  reloads on app appearance.
- **`stressLegend` is a computed property** (cheap) rebuilt on each access; the
  overlay reads it only while stress is on. No memoization needed.
