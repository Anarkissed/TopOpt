## What I did
Added the self-weight load generator and the Gate V4 validation. Scope is the
load-vector generator plus its test; no FEA solver, assembly, SIMP, element, or
isotropic-path behaviour changed.

Public API — `core/include/topopt/fea.hpp` (extended; still Eigen-free):
- `std::vector<NodalLoad> self_weight_loads(const VoxelGrid& grid, double
  density, double gravity, Vec3 direction = {0,0,-1})`. Every solid voxel (tag
  != Empty) carries a body force of magnitude `gravity * density *
  grid.voxel_volume()` in the unit `direction`, lumped equally (1/8) to its
  eight corner nodes and accumulated over shared nodes. Returns one `NodalLoad`
  per (node, non-zero component). `direction` is the direction the weight pulls
  (opposite the build-plate normal of ARCHITECTURE §5), default -z; it need not
  be unit length (normalised internally). Throws `std::invalid_argument` on
  `density < 0`, `gravity < 0`, or a (near) zero-length direction.

Implementation — `core/src/fea/assembly.cpp` (appended after `fea_tagged_nodes`,
in the Eigen-gated TU; no new includes needed — `<cmath>`, `<stdexcept>`,
`<vector>`, and `<array>` (via fea.hpp) are already present):
- Validates arguments, normalises `direction`, computes the per-node share
  `gravity * density * voxel_volume() / 8`, accumulates the (dhat-scaled) share
  into a per-node force vector over the eight corner nodes of every solid voxel,
  then emits one `NodalLoad` per (node, component) whose accumulated value is
  non-zero. Axis-aligned directions (e.g. the default -z) leave the other two
  components exactly 0.0, so no spurious zero loads are produced.

Design choices and WHY:
- **Home = `fea/assembly.cpp`, gated on Eigen.** The generator produces
  `NodalLoad`s from grid geometry and uses `fea_element_nodes` /
  `fea_node_count` (both defined in assembly.cpp), so it belongs beside the
  other global-FEA node/load helpers. It does not itself need Eigen, but the
  only consumer (the V4 solve) needs the solver anyway, and keeping it in the
  Eigen-gated TU matches how the rest of the global FEA API is gated. No new
  module, no new CMake object.
- **Equal 1/8 lumping IS the consistent load.** For a cubic trilinear hex under
  a uniform body force each shape function integrates to V/8, so the
  energy-consistent nodal load is V/8 per node — identical to simple lumping.
  This is why the V4 column stress comes out essentially exact (see below), and
  it is the standard Sigmund-style body-load treatment.
- **`direction` is the force direction (default -z), not the build-plate
  normal.** A load generator returning loads reads most naturally when the
  parameter is the direction the load acts. The header documents the relation to
  §5's "build-plate normal" (the weight pulls opposite it). Reorienting the part
  / choosing the normal from an orientation candidate is M4.3/M4.4; this
  function only builds the vector.
- **Accumulate then emit one entry per (node, component).** Avoids relying on
  the solver summing duplicate load entries and keeps the output deterministic
  and directly inspectable (a node's total self-weight share is one number).
- **`density`/`gravity` are caller-unit scalars, not read from a `Material`.**
  ARCHITECTURE §5 states the load as "gravity × density × voxel volume"; binding
  materials.json `density_g_cm3` and a chosen g into a solve (unit system,
  orientation) belongs to the pipeline task that wires self-weight into
  `minimize_plastic` (M5.3). No speculative coupling here.

### How the test is set up and why it is trustworthy
`gate_v4` (new, `tests/validation/test_v4.cpp`, Eigen-gated, self-contained
CHECK harness, public API only) has two parts:

Part A — load generation (exact, no solve). On a fully-solid 3×3×4 grid
(spacing 2, ρ=3, g=9.81) with per-voxel body force `fvox = g·ρ·h³`:
- default direction is -z: every entry is on the z component and negative;
- total z force equals `-fvox·36` (equilibrium hand-check: the base must carry
  the whole column weight);
- exactly one load per node on a fully-solid grid;
- corner node (0,0,0) (one voxel) carries `-fvox/8`; interior node (1,1,1)
  (eight voxels) carries `-fvox`; the axis-aligned case emits no x/y load;
- removing one voxel (set Empty) drops exactly its weight from the total;
- a non-unit `{0,0,-5}` direction gives the same result (normalisation);
- a diagonal `{1,1,1}` direction splits the same total magnitude `fvox·36`
  across x, y, z in proportion to the unit direction (1/√3 each);
- negative density, negative gravity, and a zero-length direction each throw.

Part B — Gate V4 column (the §7 V4 gate). A W=2 × W=2 × 32 column (spacing 1,
E=1000, ν=0.3, ρ=2.5, g=9.81) stands on a base plane held only in u_z (free to
contract laterally) with a minimal in-plane pin+roller to remove the remaining
rigid modes — the boundary conditions of the exact uniaxial self-weight
solution, so no constraint reaction perturbs the stress. Loaded by
`self_weight_loads` (default -z), solved with `fea_solve`, and the element
centroid axial stress `sigma_zz` (via `hex8_stress`) is compared per layer to
the analytical `-ρ·g·(H − z_c)`:
- base (peak) axial stress within 2% of the analytical value — the V4 gate;
- every layer's axial stress within 2% (the full linear profile, not just the
  peak);
- lateral normal stresses (`sigma_xx`, `sigma_yy`) are ≤ 2% of the base stress,
  confirming the state is uniaxial so the compared `sigma_zz` really is the
  axial stress;
- axial compression decreases monotonically up the column;
- the total applied self-weight equals `ρ·g·(column volume)`.

Measured: base and worst-layer relative error are both 0.0000% (the lumped load
is the consistent load, so nodal displacements — hence centroid stresses — are
essentially exact for the uniaxial column); max lateral/base stress 0.2273%.

## What I did NOT do
- **Did not wire self-weight into the SIMP loop / `minimize_plastic` / the
  multi-variant runner.** `self_weight_loads` is a standalone load generator; a
  caller passes its result to `fea_solve`/`fea_solve_cg`. The self-weight +
  auto volume-fraction-ladder `minimize_plastic` pipeline is M5.3.
- **Did not read `density_g_cm3` from a `Material` / materials.json, and did not
  pick a gravitational constant or a unit system.** `density` and `gravity` are
  caller scalars; binding a material's density and a chosen g belongs to the
  pipeline task (M5.3). The loader (M1.2) already parses `density_g_cm3`.
- **Did not use the transversely isotropic element (M4.1).** The V4 column test
  solves the isotropic path (`fea_solve`); self-weight is a load, independent of
  the constitutive model. The layer-normal / z_knockdown interaction is not
  exercised here.
- **Did not derive the direction from an orientation candidate.** `direction` is
  a caller argument (default -z); orientation candidate generation is M4.3.
- **Did not add a consistent-load or lumped-mass matrix, gravity vector on the
  `NodalLoad` struct, or any other API.** Only `self_weight_loads` was added.
- **Did not clamp the column base fully.** The V4 BC set holds only u_z on the
  base plane (plus a minimal in-plane pin+roller); a full 3-DOF base clamp would
  add a Poisson lateral constraint reaction near the base and perturb the
  uniaxial stress — the test deliberately avoids that.
- Did not touch `/app/`, `ARCHITECTURE.md`, `materials.json`, CI config, any
  fixture, or any existing test assertion/tolerance. The ROADMAP change is the
  single M4.2 box. Diff: +82 lines across `fea.hpp`, `assembly.cpp`,

## Warnings for the next run
- **`self_weight_loads` returns loads, it does not apply BCs.** A self-weight
  model with no Dirichlet constraints is a free-floating body — `fea_solve` will
  throw "singular / insufficiently supported". The caller must still fix a base
  (as the V4 test does). This is by design (the fixture face is a user/pipeline
  choice, §5).
- **`direction` is the direction the weight pulls (default -z), i.e. OPPOSITE
  the build-plate normal.** If a later task has the build-plate normal in hand,
  pass its negation, not the normal itself.
- **The V4 column error is ~0% because equal 1/8 lumping equals the consistent
  load for a cubic trilinear hex and the column stress state is uniaxial** (so
  nodal displacements are essentially exact). Do not expect the same near-exact
  agreement for a bending-dominated self-weight case; there the trilinear
  element shear-locks like the V1 beams and needs mesh refinement.
- **Loads are emitted only for non-zero accumulated components.** For an
  off-axis direction all three components appear per node; for an axis-aligned
  direction only one does. A consumer that indexes loads expecting exactly
  3×N entries must not assume that — count is (nodes touching solid) × (non-zero
  direction components).
- **`gate_v4` adds ~0.05 s** (one small direct solve); full suite ~114 s
  locally, well under the M2.3 "< 5 min CI budget".
