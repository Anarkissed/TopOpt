New public header (the API the app/pipeline sees, ARCHITECTURE §3):

  `solid_count`/`voxel_volume`/`solid_volume`/`voxel_center` helpers), and the
  free function `voxelize(mesh, resolution)`. Voxelization lives in `voxel/` per
  §3 and consumes the M1.3 `TriangleMesh`/`bounding_box` from `mesh.hpp` (the §5
  pipeline "tessellated surface → … → voxelize").

Implementation:
- `core/src/voxel/voxelize.cpp`:
  - **Grid sizing.** `resolution` = number of voxels along the mesh's longest
    bounding-box axis; voxel edge `h = max_extent / resolution`. Each axis gets
    `ceil(extent/h - 1e-9)` voxels (the epsilon stops an evenly-dividing axis
    gaining a spurious extra voxel). Voxels are cubic, as the hex-element FEA
    requires (§4). Grid origin = bbox min. For a mesh with a cubic bbox this is
    `resolution³` (cube→10³, sphere→128³).
  - **Solid fill by signed-crossing (winding) scan.** For each grid column a
    vertical +Z ray through the voxel centres collects, from every triangle
    whose XY projection contains the centre, a crossing = (z of the triangle
    plane there, sign of the triangle's projected orientation = sign of its
    normal's z). A voxel centre is inside iff the **signed sum** of crossings
    strictly above it is non-zero. Using the winding sum rather than parity is
    what makes the fill robust when the ray grazes an edge shared by two
    same-facing triangles — e.g. the diagonal of the cube's top/bottom faces
    (columns with `i+j=9` land exactly on the shared edge `x+y=10`): both
    triangles are counted, the winding doubles to ±2 but stays non-zero, so the
    voxel does not wrongly drop out. Point-in-triangle is boundary-inclusive by
    design so both sharers see the on-edge centre. Vertical triangles
    (`|projected area| < 1e-18`) are skipped (a +Z ray can't cross them).
  - **Tagging.** A solid voxel is `Surface` if any of its 6 face-neighbours is
    Empty or outside the grid, else `Interior`. `UserTagged` is never emitted by
    the voxelizer; it exists so a later stage (M1.6 LOAD/FIXTURE) can mark a
    voxel — `set_tag` + the `solid()` "non-Empty" rule let the grid store and
    round-trip that tag, which is the "user-tagged" storage M1.5 asks for.

Tests — `core/tests/unit/test_voxel.cpp`, registered as ctest `voxelize` (same

## Warnings for the next run
- **Resolution semantics:** `resolution` is voxels along the *longest* bbox axis,
  not per-axis. Cubic-bbox meshes (both fixtures) give `resolution³`; a
  non-cubic part gets fewer voxels on its short axes (same cubic `spacing`). M1.6
  and the FEA assume cubic voxels — do not switch to per-axis independent counts.
- **The cube's exact 1000 count depends on the winding fill**, specifically that
  a ray grazing the shared top/bottom-face diagonal is not miscounted. If anyone
  "simplifies" the signed-sum back to parity or makes point-in-triangle strictly
  exclusive, the diagonal columns will drop out and the exact-count test will
  (correctly) go red. See the naive-parity honesty check above.
- **Surface = touching the grid boundary counts as boundary.** For the cube,
  which fills the grid to its edges, the whole outer shell is `Surface` — that is
  correct (those voxels are on the solid's boundary). A mesh that does not reach
  the grid edge will only mark true material-boundary voxels Surface.
- `STL_FIXTURE_DIR` is a compile-time absolute path baked in at configure time
  (same trade-off as the other tests): reconfigure if the repo is moved. A clean
  CI checkout always reconfigures.
- Fill assumes a closed, consistently-wound mesh (both fixtures qualify;
  `import_stl_file` guarantees watertight). A ray grazing a *silhouette* edge —
  shared by an up-facing and a down-facing triangle — would see the two signs
  cancel; this does not occur at voxel centres for these fixtures but could for a
  pathological part aligned to the grid. Revisit if a real part voxelizes with
  holes.
