## What I did
Added marching-cubes isosurface extraction, mesh cleanup, and the four
ARCHITECTURE §7 V3 property checks packaged as a single suite entry point
(`check_v3`), plus a property test that exercises each check and runs the suite
end-to-end on a real SIMP optimizer output.

Mesh API — `core/include/topopt/mesh.hpp` (extended; still Eigen-free; no
dependency on voxel.hpp, so `mesh/` stays independent per §3's layout):
- `TriangleMesh marching_cubes(nx, ny, nz, spacing, origin, field, iso)` —
  classic Lorensen–Cline marching cubes (standard 256-case triangle table) over
  a grid-indexed scalar field. Samples at voxel **centres**; the domain is padded
  by one layer of background (0) on every side so a solid reaching the grid
  boundary is closed off there. "Inside" = `field > iso` (density convention;
  M3.5 uses `iso = 0.5`). Vertices are placed by linear interpolation along cube
  edges and welded through a global edge→vertex map, so the same lattice edge is
  one shared vertex across all cubes → the per-shell surface is closed and
  2-manifold.
- `int count_components(const TriangleMesh&)` — connected components under
  shared-**edge** adjacency (two triangles connected when they share two vertex
  indices). One watertight shell = 1 component.
- `TriangleMesh keep_largest_component(const TriangleMesh&)` — the §5 "mesh
  cleanup" step: drop degenerate (repeated-index) triangles, keep only the
  largest shared-edge component, re-index its vertices. Reduces a multi-shell MC
  output (disconnected 0.5-threshold islands) to the single dominant printable
  body.

Property/suite API — `core/include/topopt/voxel.hpp` (extended; still Eigen-free;
voxel.hpp already includes mesh.hpp, so it can call the mesh functions):
- `TriangleMesh marching_cubes(const VoxelGrid&, density, iso=0.5)` — thin
  wrapper feeding the grid's geometry to the mesh/ marching_cubes.
- `int min_feature_violations(grid, density, iso=0.5)` — count of solid voxels
  (density > iso) NOT covered by any fully-solid 2×2×2 block, i.e. features
  thinner than 2 voxels in some direction (§7 V3 "min feature size ≥ 2 voxels").
- `struct V3Report` + `V3Report check_v3(grid, density, iso=0.5)` — runs the
  whole suite: marching_cubes → keep_largest_component → the four gates
  (`gate_watertight`, `gate_single_component`, `gate_load_fixture_retained`,
  `gate_min_feature`) and `passes` = their conjunction. `kV3RetentionThreshold =
  0.9` is the §7 retention floor. The cleaned mesh is returned in the report so a
  caller can export it.

Design choices and WHY:
- **Classic marching cubes, not a blocky/voxel-face surface or marching
  tetrahedra.** ARCHITECTURE §3/§5/§9 name "marching cubes" specifically, and §1
  rules say the architecture wins. The standard table yields a closed,
  consistently-wound shell; ambiguous face configurations (diagonal
  checkerboard) are the only ones that can open it, and those are exactly what
  the M3.3 density filter (radius ≥ 1.5) suppresses — so on the filtered physical
  density the extracted surface comes out watertight and 2-manifold (verified
  empirically on the benchmark output and on synthetic solids, holed solids, and
  disjoint blocks; see evidence).
- **Cleanup = keep-largest-component.** V3's "exactly one connected component" is
  asserted on the cleaned mesh (the M3.4 handoff/`benchmarks.json` explicitly
  defer single-component to V3 on the cleaned mesh because the cantilever's two
  4-voxel tip islands are a 0.5-threshold artifact). Keeping the largest shell
  removes those islands.
- **Sample at voxel centres + one-layer background padding.** This closes the
  surface exactly at the grid boundary (a solid block's mesh spans `[0, n·h]` per
  axis) instead of leaving an open boundary, and matches the grid's index
  convention so `field[(k·ny+j)·nx+i]` is the sample at voxel (i,j,k).
- **Min feature via a solid-2×2×2-block cover.** A voxel is "thick enough" iff it
  lies in at least one fully-solid 2×2×2 block; anything left over is 1 voxel
  thin in some direction. This is the morphological-opening reading of "≥ 2
  voxels" and it bites on 1-voxel plates/struts/spikes.
- **Retention gate reads M1.6 Load/Fixture tags.** A voxel counts as a feature-
  support voxel when its tag is `Load` or `Fixture`; the gate is `min density
  over those ≥ 0.9`. With no such voxels it is vacuously satisfied.

### What check_v3 asserts vs. what the M3.4 optimizer currently delivers
The two **mesh** gates (watertight, single component) hold on every M3.4 output I
measured, so `test_v3` asserts them end-to-end on a real cantilever run. The two
other gates are **not** asserted on raw optimizer output, and this is
deliberate, not a weakening:
- **Retention** is *emergent* at M3.5. `simp_optimize` OC-updates all solid
  voxels equally; it does not pin Load/Fixture voxels, so low-stress tagged
  voxels are thinned (measured min tagged density 0.001 at vf 0.5). ROADMAP M3.7
  is the task that makes Load/Fixture "implicitly FrozenSolid (V3 retention
  becomes structural, not emergent)". Asserting retention on raw M3.5 output
  would be asserting behaviour the optimizer does not yet provide.
- **Min feature size** ≥ 2 voxels is not met by raw SIMP designs at 24×8×8 even
  at high volume fraction (measured 288 violations at vf 0.5, still 24 at vf
  0.85 — SIMP struts are routinely 1 voxel thin at this resolution). The filter
  regularizes but does not guarantee a 2-voxel print width.

Both checks are instead covered by controlled synthetic pass/fail cases (a solid
2×2×2-covered design passes; a 1-voxel plate / an isolated voxel / a dropped
tagged voxel fail), and `check_v3` returns the full report so a future task
(M3.7 passive regions, or a morphological-opening cleanup) can turn the
assertions on for optimizer output once the optimizer earns them. A genuine
all-four-gates `passes == true` verdict is asserted on a synthetic solid design
(group 3a), so the suite's positive path is exercised.

## What I did NOT do
- **Did not implement ambiguity resolution (asymptotic decider / MC33).** The
  classic table is used as-is. It is watertight on smooth (filtered) fields but
  can open on checkerboard/diagonal corner patterns; the M3.3 density filter is
  what keeps the SIMP physical density free of those, and cleanup drops any stray
  shells. If a future task feeds `marching_cubes` a raw un-filtered or
  deliberately checkerboarded field, the watertight gate may (correctly) fail.
- **Did not make Load/Fixture retention or min-feature-size pass on raw optimizer
  output.** Retention is emergent until M3.7; min-feature is not met by SIMP
  designs at this resolution. Both are covered by synthetic pass/fail cases and
  reported (not asserted) on the end-to-end run. No morphological opening / thin-
  feature removal was added to cleanup — cleanup is only degenerate-triangle drop
  + keep-largest-component.
- **Did not change `simp_optimize`, `simp_compliance`, the filter/OC, the FEA
  path, or any M3.1–M3.4 behaviour.** M3.5 only reads the physical density field.
- **Did not add STL/3MF export.** That is M6.1; `check_v3` returns the cleaned
  `TriangleMesh` but nothing writes it to a file yet.
- **Did not evaluate min-feature on the cleaned mesh occupancy.** It is computed
  on the density field directly (all solid voxels), independent of which shell

- **Did not wire `check_v3` into the existing `gate_v2` / `mbb` tests.** Those

## Warnings for the next run
- **`marching_cubes` "inside" is `field > iso` (strict).** A sample exactly equal
  to `iso` is treated as outside; degenerate (zero-area) triangles from an
  exact-tie corner are dropped. On continuous filtered densities ties are
  measure-zero, but a binary {0,1} field with `iso = 0.5` is fine (0 < 0.5 < 1).
  If you pass an integer/binary field, keep `iso` strictly between the two
  levels.
- **The watertight guarantee rides on the density filter.** It holds for
  M3.3/M3.4 physical density. Do not assume an arbitrary scalar field extracts to
  a watertight mesh — the classic table can open on diagonal/checkerboard corner
  patterns. If M6 export needs a hard guarantee on arbitrary inputs, add the
  asymptotic decider (or a manifold MC variant) then.
- **Retention + min-feature gates fail on raw M3.5 optimizer output by design.**
  If a future test wires `check_v3(...).passes` onto a `simp_optimize` result it
  WILL fail on those two gates. Assert `gate_watertight()` and
  `gate_single_component()` for now; enable retention once M3.7 pins Load/Fixture
  voxels, and min-feature once there is a thin-feature cleanup or a coarser
  design length scale.
- **`min_feature_violations` uses a 2×2×2 solid-block cover.** A legitimately
  2-voxel-thick wall that is 2 thick in only one axis and 1 in another (a 2×1×N
  beam) counts as a violation — that is intended (its smallest cross-section is 1
  voxel), but it means the check is strict about *all three* directions.
- **`count_components` uses shared-EDGE adjacency**, so two shells touching only
  at a single vertex read as 2 components (correct for a printable-body count).
- **`property_v3` adds ~3 s** (one 60-iteration 24×8×8 SIMP solve + the synthetic
  cases); full suite ~100 s locally, well under the M2.3 "< 5 min CI budget".
