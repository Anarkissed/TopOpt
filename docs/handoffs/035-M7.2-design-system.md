### Placement — a new SwiftUI package target, not app-target files
The design system is a new **`TopOptDesign`** target/product in the existing
`app/TopOptKit/Package.swift` (SwiftUI-only, no C++ interop, no core dependency).
Rationale: this is the same pattern M7.1 chose for the bridge — a local SwiftPM
target gives a **headless, machine-independent `xcodebuild test`** path (the M7
/app/ verification standard) and keeps the module reusable by the M7.3–M7.9 app
screens (they `import TopOptDesign`). Putting the views only in the app target
would leave the tokens unverifiable without a booted iOS simulator. The module
builds on every slice (macOS + iOS) because it depends on nothing but SwiftUI.

### Tokens — `Sources/TopOptDesign/DesignSystem.swift`
- **`RGBA`** value type: a color as exact 0–255 sRGB channels + 0–1 alpha (how
  the HTML writes `rgba()`/`#rrggbb`), with `hex:` and channel initializers, an
  `.opacity(_:)` re-alpha, and a `.color` → SwiftUI `Color`. Colors are stored as
  numbers precisely so tokens are verifiable **without rendering**.
- **`DS`** namespace: `DS.Color` (background, ink, ok-green, danger, the
  `#f2f2f5` text ramp at 0.55/0.45/0.40/0.35, the four accent options, the
  5-color group palette, hairline strokes, control fills, scrim), `DS.Surface`
  (the tinted glass fills: panel/sheet/dialog/bar/toast), `DS.Radius`,
  `DS.Space`, `DS.Blur` (design blur radii + saturation — retained as intent
  since SwiftUI `Material` has no numeric radius), `DS.Shadow` (sheet/panel/
  toast/accent-glow), `DS.TypeScale` (px sizes + weights + tracking — the enum is
  named `TypeScale`, not `Type`, because `DS.Type` collides with the metatype
  keyword), and `DS.Motion` (the `cubic-bezier(0.2,0.9,0.3,1)` ease, durations,
  the 2.4 s toast dwell). `.dsShadow(_:)` / `Text.dsStyle(_:)` convenience apply
  a token. Every token's doc comment cites the HTML literal it traces to.

### Views (each with a `#Preview`) — `Sources/TopOptDesign/`
- **`GlassPanel`** — the floating panel recipe (panel fill + `.ultraThinMaterial`
  backdrop + `0.11` hairline + `22` radius + panel shadow), via an internal
  `GlassSurface` modifier shared with the sheet.
- **`GlassSheet`** — the centered modal card (heavier fill, `28` radius, deep
  shadow); plus `.glassSheet(isPresented:)` presenting it over the design's
  dimmed+blurred scrim (tap-to-dismiss, `sheetIn` transition).
- **`PillButton`** — pill button with `.primary` (accent + glow) / `.secondary`
  (hairline ghost) / `.danger` (red tint) roles, optional leading SF Symbol, and
  the disabled treatment; suppresses `action` when disabled.
- **`SegmentedGlass<Value: Hashable>`** — the inset segmented control (dark
  track, lit selected segment), generic over an enum/String binding.
- **`Toast`** — the blurred pill; plus `.toast(_:)` presenting it 96 pt off the
  bottom, sliding up, auto-clearing the binding after `DS.Motion.toastDwell`.
- **`ProgressBar`** — the 5 pt determinate track + accent fill, value clamped
  0…1, `0.15 s` linear width tween.

## What I did NOT do
- **Did NOT build/run the iPad app target or import `TopOptDesign` into the app
  yet.** The module is delivered as a tested, previewable package; wiring the
  screens (Home/import/workspace/run/results) onto these views is M7.3–M7.9. The
  app target is unchanged and still depends only on the `TopOptKit` product, so
  adding the second product does not affect its build (a full iPad build remains
  maintainer device QA per M7 rules).
- **Did NOT compile the SwiftUI `#Preview`s into a running app.** Previews are
  for maintainer QA side-by-side with the HTML (the task's acceptance); they are
  stripped from normal builds. The token *values* are what the tests verify.
- **Did NOT try to reproduce the design's backdrop blur numerically.** SwiftUI's
  `Material` has no px radius, so glass uses `.ultraThinMaterial` under the dark
  tint fill; the `DS.Blur` numbers are retained as design intent + for QA, not
  fed to a blur. Exact glass look is maintainer QA.
- **Did NOT map CSS `650`/`750` font weights exactly** (SwiftUI has no such
  weights): `650 → .semibold`, `700 → .bold`, `750/800 → .heavy` (documented per
  case). Px sizes and tracking are exact and tested.
- **Did NOT add non-component tokens the design uses once** (ambient page glows,
  the stress conic-gradient legend, per-screen one-off paddings). Only the tokens
  the six required components need, plus the palette/scale the reviewer can diff.
- **Did NOT change any existing scheme file on disk.** The `TopOptKit-Package`
  scheme is SwiftPM-auto-generated; I only documented which scheme now carries
  the test action.

## Warnings for the next run
- **Use `-scheme TopOptKit-Package` (or `swift test`) to run app tests now.**
  Adding the second product (`TopOptDesign`) means the per-product `TopOptKit`
  scheme is **build-only** — `xcodebuild test -scheme TopOptKit` now errors
  "Scheme TopOptKit is not currently configured for the test action". The M7.1
  handoff's documented `-scheme TopOptKit` test command is superseded (README
  updated). `swift test` is unaffected and runs both test targets.
- **`DS.TypeScale`, not `DS.Type`.** `Type` as a nested enum name collides with
  Swift's metatype `.Type` and won't compile; the typography enum is `TypeScale`.
- **Tokens are the tested contract; the views' pixel layout is not.** The tests
  assert values, and `testViewsInstantiate` only proves the views *construct*.
  Visual fidelity (blur, exact spacing, animation feel) is maintainer QA against
  `docs/design/TopOpt.dc.html`.
- **`DS.Surface.sheet` is `rgba(30,30,36,0.80)`** — the design uses `0.78` (import/
  export sheets) and `0.82` (weight dialog); I took `0.80` as the mid and kept
  `Surface.dialog = rgba(32,32,38,0.82)` separately for the tighter dialog. If a
  screen needs the exact per-sheet alpha, use the specific surface.
- **`.glassSheet` / `.toast` presenters are conveniences**, not full navigation.
  M7.3+ may prefer to drive presentation from their own view models; the bare
  `GlassSheet` / `Toast` views compose either way.
