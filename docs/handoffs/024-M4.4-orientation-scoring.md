## What I did
Added the orientation scorer that combines the M4.3 support proxy with a
z_knockdown-penalized "tension across layer planes" term, plus the Gate V5
validation on the hook fixture. Scope is the two new scoring functions in the
existing `orient/` module and their tests; nothing in FEA, SIMP, mesh, voxel, IO
or the M4.3 orientation functions changed behaviour, and no new module or
dependency was added.

New public API — `core/include/topopt/orient.hpp` (still Eigen-free, OCCT-free):
- `double max_interlayer_tension(const VoxelGrid& grid, const
  std::vector<std::array<double,6>>& stress, const Vec3& build_dir)`. Layers are
  perpendicular to `build_dir`, so tension *across* them is the normal traction
  `n·σ·n` on a plane with normal `n = unit(build_dir)`. Returns the maximum over
  solid voxels of `max(0, n·σ·n)` — the worst inter-layer tensile stress for that
  orientation (0 if no solid voxel is in inter-layer tension). `stress` is a
  per-voxel Cauchy stress field, grid-indexed (size `grid.voxel_count()`), each
  entry Voigt `[xx,yy,zz,xy,yz,zx]` with TRUE (undoubled) shear — exactly the
  `sigma` array `hex8_stress` returns. Empty voxels are ignored. `build_dir`
  normalized internally. Throws `std::invalid_argument` on a zero-length
  `build_dir` or `stress.size() != grid.voxel_count()`.
- `struct OrientationScore { Vec3 build_dir; int support_voxels; double
  interlayer_tension; double stress_penalty; double score; }` — lower `score` is
  better.
- `std::vector<OrientationScore> score_orientations(const VoxelGrid& grid, const
  std::vector<std::array<double,6>>& stress, const std::vector<Vec3>& candidates,
  double z_knockdown, double support_weight = 1.0, double stress_weight = 1.0)`.
  Per candidate: `support_voxels = support_overhang_voxels(grid, dir)` (M4.3);
  `interlayer_tension = max_interlayer_tension(grid, stress, dir)`;
  `stress_penalty = interlayer_tension * (1/z_knockdown − 1)`. Each of the two
  terms is scaled by its maximum over `candidates` (both become dimensionless in
  [0,1]; a term whose max is 0 drops out), then combined `score = support_weight *
  support_norm + stress_weight * stress_penalty_norm`. Returns one score per
  candidate, sorted best (lowest) first; ties keep input candidate order (stable
  sort). Throws `std::invalid_argument` on empty candidates, `z_knockdown` ∉
  (0,1], a negative weight, or a stress-size mismatch.

Implementation — `core/src/orient/orient.cpp` (appended; added `<algorithm>` for
`std::stable_sort`; `<stdexcept>` already included). Pure C++/std, so it stays in
the always-built library and its unit test runs in every configuration (including
the Eigen/OCCT-absent CI-light path).

Design choices and WHY:
- **The knockdown enters in SCORING, not via a re-solve.** ARCHITECTURE §7 V5:
  "Tension across layer planes must be penalized *by the z_knockdown factor in
  scoring*." A single isotropic FEA solve in the fixed part frame gives the stress
  field once; each candidate's "across-layer" tension is the geometric projection
  `n·σ·n` of that one field, and z_knockdown is applied as a scoring penalty. This
  is exactly "max tensile stress across layer planes (knocked down)" and needs
  neither the M4.1 transversely-isotropic element nor 26 re-solves with rotated
  material axes (which would be a much larger task and is not what the spec says).
  So M4.1's TI element is intentionally NOT used here.
- **Penalty factor `(1/k − 1)` = the EXTRA inter-layer failure risk over
  isotropic.** Inter-layer strength is `k·S`, so the inverse-margin risk is
  `σ/(k·S)`; relative to the in-plane risk `σ/S` the extra risk is `σ·(1/k − 1)`.
  It **vanishes at k = 1** (resins), so for an isotropic material the stress term
  carries NO orientation preference and ranking falls back to the support proxy —
  which is precisely the fixture's mechanism requirement ("with z_knockdown set to
  1.0, the stress-alignment term must lose its preference; the support term may
  still prefer flat"). (`(1 − k)` would also vanish at k=1; I chose the
  inverse-margin form as the physically standard one.)
- **Each term normalized by its max over the candidate set** so the two
  physically-incommensurable quantities (a voxel count and a stress) combine
  without a hand-tuned unit conversion. The fixture pins only the ORDER, not the
  score scale, so a scale-free normalized combination is the robust choice. Note
  a consequence: for k < 1 the `(1/k − 1)` factor is a common positive scale that
  cancels in the normalized stress term, so the combined *ranking* is the same for
  any k ∈ (0,1); z_knockdown's discriminating role is between MATERIALS (it turns
  the stress term OFF at k=1), and the raw `stress_penalty` field the test inspects
  does respond to k. This matches "driven at least in part by the z_knockdown
  penalty" — the stress term contributes to the k<1 ranking (σ_yy ≫ σ_zz for the
  hook) and disappears for a resin.
- **Scorer takes a precomputed stress field, keeping `orient/` Eigen-free.** The
  FEA solve that produces the field needs Eigen; keeping it in the caller/test
  preserves the module's always-built, dependency-light discipline (same reason
  M4.3 kept the module geometry-only). The Gate V5 test does the solve.

### How the tests are set up and why they are trustworthy
Two test targets, matching the codebase's split (pure logic in the always-built
test, the FEA-driven gate in an Eigen-gated validation test):

`orientation` (extended, `tests/unit/test_orient.cpp`, always-built pure-C++/std,
public API only, 48 → **80 checks**). New synthetic-field blocks (no solver):
- **`max_interlayer_tension`**: uniform field σ = [0,10,1,0,0,0] on a solid box →
  build +z reads σ_zz=1, +y reads σ_yy=10, +x reads σ_xx=0; even in the sign of
  `n` (−y also 10); non-unit dir normalized; a compressive field → 0 (the
  `max(0,·)` clamp); a pure xy-shear field σ_xy=4 → 4 along the +45° xy diagonal
  and 0 along the −45° diagonal (proves the `2·n_a·n_b·τ` shear terms and the
  tension/compression sign); an Empty voxel carrying σ_yy=999 is ignored;
  zero-length dir and stress-size mismatch throw.
- **`score_orientations`**: on a solid 2×2×2 box (support 0 for all axes, M4.3),
  ranking is driven purely by the stress term; with σ = [0,10,1,0,0,0] the
  per-candidate interlayer tension is x:0, z:1, y:10, penalty = tension·(1/k−1),
  and the combined order is x < z < y (best = +x); result sorted ascending; at
  k=1 every `stress_penalty` is 0 (and, support being 0 here too, every score
  ties 0 — the term loses preference); a separate floating-shelf-on-a-pillar grid
  with a zeroed stress field shows the support term ALONE still ranks −z first;
  all argument-validation paths throw.

`gate_v5` (new, `tests/validation/test_v5.cpp`, Eigen-gated, self-contained CHECK
harness, public API only, **17 checks**) — the §7 V5 gate on the real fixture:
imports `tests/fixtures/orient/hook.stl`, voxelizes (resolution 32 → 20×32×6,
1.4375 mm voxels, 1320 solid), tags the load case by coordinate per the fixture's
`load_case` (Fixture = back-plate voxels against the x=0 wall face, y∈[8,40];
Load = tip inner-surface voxels x∈[20,27], y∈[7,15]), clamps the whole back-plate
face and applies a downward −y tip load, solves with `fea_solve_cg`, recovers the
per-voxel centroid stress field via `fea_element_nodes` + `hex8_stress`, then
scores `orientation_candidates(hook)` at PLA `z_knockdown = 0.55`. Asserts:
- the top-ranked build direction is flat (parallel to ±Z);
- `[0,0,1]` is the (tied) best score and strictly beats upright `[0,1,0]`
  (the fixture's ORDINAL-ONLY assertion; ±Z tie is acceptable);
- both terms favour flat: flat support = 0 while upright has an overhang, and
  upright's interlayer tension > flat's;
- mechanism: `stress_penalty` prefers flat at k=0.55; re-scoring at k=1.0 makes
  every `stress_penalty` exactly 0 (interlayer tension unchanged — so the change
  is z_knockdown alone), and the support proxy alone still ranks flat first.

Measured (printed by the test): best = (0,0,−1) support 0, tension 0.0837,
penalty 0.0685, score 0.1979; flat [0,0,1] identical (tied); upright [0,1,0]
support 18, tension 0.4231, penalty 0.3462, score 1.12.

## What I did NOT do
- **Did not re-solve FEA per orientation or use the M4.1 transversely-isotropic
  element.** The scoring uses one isotropic part-frame stress field and applies
  z_knockdown as a scoring penalty (§7 V5 "penalized ... in scoring"). Rotating
  the TI element's material axes to each build direction and re-solving is a much
  larger, unspecified task and is not what M4.4 asks for.
- **Did not wire scoring into the pipeline / multi-variant runner / report / a
  `minimize_plastic` driver.** `score_orientations` is a standalone function that
  a caller feeds a stress field, candidates and a material's `z_knockdown`. Binding
  materials.json `z_knockdown`, choosing the FEA load case from user tags, and
  emitting orientation into the job report are later tasks (M5.x).
- **Did not read `z_knockdown` from a `Material` / materials.json.** It is a
  function argument; the loader (M1.2) already parses the field.
- **Did not add a per-voxel stress-tensor field helper to the FEA module.** The
  V5 test builds the field from the existing public `fea_element_nodes` +
  `hex8_stress`, so no FEA/assembly change was needed (smaller diff, no risk to
  M2/M3). A `fea_stress_field` sibling of `fea_von_mises_field` could be added if a
  future task wants it, but M4.4 did not need it.
- **Did not change the M4.3 functions** (`flat_face_normals`,
  `orientation_candidates`, `support_overhang_voxels`) or their tests, the FEA/
  SIMP/mesh/voxel/IO code, `/app/`, `ARCHITECTURE.md`, `materials.json`, CI config,

## Warnings for the next run
- **The Gate V5 solve uses `fea_solve_cg`, not the direct `fea_solve`, on purpose.**
  Voxelizing the hook's curved boundary produces staircase "hinge" nodes (a node
  where solid voxels meet only diagonally) that give near-zero-energy modes; the
  strict LDLT-pivot rank check in `fea_solve` rejects the system as "singular",
  while `fea_solve_cg`'s M3.1 void-DOF safety gate handles them and converges (233
  iters, residual ≈ 1e-8 at res 32). Any future orientation/pipeline code solving
  a voxelized organic part should use the CG path for the same reason. The hook is
  a single face-connected component (checked: 1 component, 1320 voxels) — the
  hinges are internal near-zero-stiffness modes, not disconnected islands.
- **The combined ranking is z_knockdown-independent for k ∈ (0,1); the knockdown's
  role is to switch the stress term OFF at k = 1 (resins).** This is a direct
  consequence of per-term max-normalization (the `(1/k−1)` scale cancels). If a
  future task wants the *degree* of knockdown to shift the k<1 ranking, it must
  combine the raw (un-normalized) terms with an explicit unit weight instead — a
  scoring-scale change (the fixture pins only order, so that is allowed, but it is
  a deliberate design choice, not a bug here).
- **`score_orientations` normalizes each term across the passed candidate set**, so
  a candidate's score depends on the others. Scoring a single candidate in
  isolation yields score 0 for both terms (each is its own max). Always score the
  full candidate set together (as Gate V5 and the pipeline do).
- **Inter-layer tension counts tension only** (`max(0, n·σ·n)`); a purely
  compressive across-layer state scores 0 penalty, which is correct (FDM layer
  bonds fail in tension, not compression).
- **The V5 load case is tagged by coordinate** (STL has no face identity), matching
  the fixture's `load_case` description; the ranking is robust to the exact tag box
  (any real in-plane bending load makes σ_yy ≫ σ_zz for this prism), but a future
  change to the tag regions should keep the fixture and load on genuinely
  different features or the bending path degenerates.
- **`gate_v5` adds ~0.1 s** (one small CG solve on ~1320 voxels); full suite ~116 s
  locally, well under the M2.3 "< 5 min CI budget".
