## What I did
Wired the min-feature print warning through the two modules it spans: the
warning *text* is produced from `settings/rules.json` (which the `settings/`
module owns), and the warning + its violation count are carried as two new
per-variant fields in the M5.2 job report. Nothing in FEA, SIMP, orient, mesh,
voxel or IO changed behaviour; every prior test is bit-identical. All-new code
is pure C++/std (no OCCT, no Eigen), so it stays in the always-built library and
its tests run in every configuration (verified in the dependency-light build).

**Why it spans settings + report:** the warning message lives in
`settings/rules.json`'s `min_feature_warning` section (a `template` with a
`{count}` placeholder + a `threshold`), which handoffs 025/026 explicitly
deferred to M5.2b. The `settings/` module already parses `rules.json` and
already emits human-facing warning strings (`SlicerSettings.warning`), so the
template + threshold parse there and the formatter is a sibling of
`recommend_settings`. The report module then just stores + serializes +
validates the resulting count and string — it stays pure data/serialization/
validation and gains no rules dependency.

### settings module (`core/include/topopt/settings.hpp`, `core/src/settings/settings.cpp`)
- New `struct MinFeatureWarning { std::string message_template; int threshold = 1; }`
  and a `MinFeatureWarning min_feature_warning;` field on `SettingsRules` — a
  faithful in-memory image of the `rules.json` `min_feature_warning` section.
- `parse_settings_rules` now parses that section. It is **optional**: absent →
  defaults (empty template, threshold 1), so tables predating M5.2b (and the
  existing synthetic tables in `test_settings`) still parse. When present,
  `template` is a required string and `threshold`, if given, an integer;
  a non-object section or a non-string template throws `SettingsError`.
- New `std::string min_feature_warning_text(const SettingsRules&, int violations)`:
  returns "" when `violations < threshold` (no warning) or no template is
  configured; otherwise the template with every `{count}` occurrence replaced by
  `violations`. Throws `SettingsError` if `violations` is negative. With the
  committed table (`threshold: 1`) this is exactly the task's "warning when > 0".

### report module (`core/include/topopt/report.hpp`, `core/src/settings/report.cpp`)
- `VariantReport` gains `int min_feature_violations = 0;` and
  `std::string min_feature_warning;`.
- `job_report_json` always emits both per variant (`"min_feature_violations"` as
  an integer, `"min_feature_warning"` as an escaped string).
- `validate_job_report_json` schema-checks them. To keep the pre-M5.2b canonical
  test documents valid, the fields are **optional** (like the module's existing
  leniency on unknown keys), but when present they have teeth:
  `min_feature_violations` must be a non-negative integer, `min_feature_warning`
  must be a string, and a **non-empty warning requires a positive violation
  count** (no warning without violations). The emitter always writes both, so
  every real report carries them; the leniency only preserves older documents.

Design choices and WHY:
- **Report-only, no gating.** `check_v3` already computes
  `min_feature_violations` (V3Report gate 4); this task surfaces that number and
  a warning into the report and touches no geometry, FEA, or the V3 gate itself
  (ARCHITECTURE §2: settings are heuristic/advisory).
- **Warning text comes from the human-tunable template, not hardcoded.** The
  exact wording + threshold are maintainer-owned in `rules.json`; the engine only
  substitutes `{count}`. The ROADMAP's example wording is illustrative — the
  committed template is the source of truth.
- **Validator tooth is threshold-independent.** The report validator does not
  know the rules threshold, so it enforces only the always-true invariant
  "a warning implies violations > 0", not "violations > 0 implies a warning"
  (a maintainer could raise the threshold to suppress low counts). The
  threshold→warning behaviour is tested in `settings` against the real table.

## What I did NOT do
- **Did not call `check_v3` from the report or run the pipeline.**
  `min_feature_violations` is a plain `int` field on `VariantReport`; producing
  it (from `check_v3(grid, density).min_feature_violations`) and calling
  `min_feature_warning_text` to fill `min_feature_warning` is the M5.3 pipeline
  driver's job. There is no glue that runs the V3 suite and assembles a variant
  report (that would be speculative pipeline wiring, same boundary handoff 026
  drew for the rest of the report).
- **Did not gate or modify geometry.** The warning is report-only; the V3
  gate 4 (`gate_min_feature`) and every optimizer output are untouched.
- **Did not make the new report fields required in the schema.** They are
  optional-but-checked so the pre-M5.2b canonical documents in `test_report`
  (which I did not modify) still validate; the emitter always writes them.
- **Did not enforce "violations > 0 ⟹ warning present" in the report
  validator.** The report validator has no threshold; that direction is
  threshold-dependent and is exercised in the `settings` test against the real
  table instead.
- **Did not change `rules.json`**, its values, or its structure. I only parse
  the `min_feature_warning` section that was already committed there.
- **Did not add `{count}` alternatives** (e.g. `{n}`, printf-style) — only the
  literal token the committed template uses. Did not add `\u` escape handling to
  the settings parser (the template doesn't use it), matching M5.1.
- Did not touch `/app/`, `ARCHITECTURE.md`, `materials.json`, CI config, any
  fixture, or any existing test assertion/tolerance. The ROADMAP change is the
  single M5.2b box.

## Warnings for the next run
- **The report validator is lenient on the two new fields' *absence* but strict
  on their *content*.** A document may omit `min_feature_violations` /
  `min_feature_warning` (older shape) and still validate, but if present they
  must be a non-negative integer / a string, and a non-empty warning requires a
  positive count. `job_report_json` always emits both, so any report this module
  produces carries them.
- **"Warning when > 0" is really "warning when violations ≥ threshold", and the
  committed threshold is 1.** If the maintainer raises `min_feature_warning.
  threshold` in `rules.json`, low positive counts will produce no warning — by
  design. The report validator will not flag that (it only rejects a warning
  attached to a *zero* count).
- **The formatter lives in `settings/`, not `report/`.** `min_feature_warning_text`
  takes a `SettingsRules` and is a sibling of `recommend_settings`; the report
  module never reads `rules.json`. The M5.3 driver should call the settings
  formatter and drop its result into `VariantReport.min_feature_warning` (and set
  `min_feature_violations` from `check_v3`).
- **`min_feature_violations` counts solid voxels not covered by any solid 2×2×2
  block** (voxel.hpp `min_feature_violations`), i.e. features thinner than 2
  voxels — a voxel-resolution proxy for printability, not a physical mm width.
  The warning text says "~reliably printable width at this resolution" for that
  reason.
- **Both new tests add < 0.1 s** (pure parse/format/serialize, no solve); full
  suite ~117 s locally, well under the M2.3 "< 5 min CI budget".
