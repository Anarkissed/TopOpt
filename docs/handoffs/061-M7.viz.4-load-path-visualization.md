# Handoff 061 — M7.viz.4: load-path visualization

## Task
M7.viz.4 (TRACK app / M7-VIZ block): show how force travels from the loaded region
to the anchors, revealing WHY the optimizer kept material where it did. v1 may be a
principal-stress-direction / stress-flow overlay derived from the stress data the app
already receives — a VISUALIZATION pass, not a new physics computation, not an
optimizer call. If a coherent load path needs a full directional stress TENSOR that
the bridge does not expose, EITHER (a) derive a v1 approximation from the displacement
field + scalar stress already available (state the approximation + limits), OR (b)
STOP with a Blocked handoff requesting the tensor from the core track. Reduced-motion:
static presentation available. Consumes existing fields only; `/app/` only. Device QA
for legibility; xcodebuild tests for any pure-data derivation. Do NOT check the
ROADMAP box.

## Decision: option (a), not Blocked — and why it is honest
The bridge exposes only the von Mises **scalar** (M7.0b `vonMisesField`) and the
per-node displacement **vectors** (M7.disp `displacementField`) — there is **no
directional stress tensor**. A Block would have been defensible, but a genuinely
honest v1 is derivable from what already exists, so option (a) is the right call:

1. **Strain is the symmetric gradient of the already-solved displacement**,
   ε = ½(∇u + ∇uᵀ), computed as the trilinear hex-element gradient at each voxel
   centre. This is pure kinematics — a spatial derivative of an existing field, no new
   solve, no material model — the same category of post-processing as deriving vertex
   normals from positions or the von Mises magnitude from the solved field. It is NOT a
   new physics computation and NOT an optimizer call.
2. **Principal-stress axes coincide with principal-strain axes for isotropic
   material** (σ and ε share eigenvectors under isotropic stiffness), so the
   eigenvectors of ε (a small Jacobi solver) give the principal-stress directions. The
   dominant one (largest |eigenvalue|) is the local load-path axis; the sign of its
   eigenvalue is tension (+) vs compression (−).

No `/core/` change, no bridge change — the interface freeze holds.

### Approximation & limits (recorded on-screen and in code)
- Directions are a **finite-difference estimate** of the gradient, not an analytic
  tensor field.
- **Isotropy assumed.** The core also models transverse isotropy (M4.1 z_knockdown),
  for which principal-stress = principal-strain is an approximation on the layer-normal
  axis. v1 treats the field as isotropic.
- It is a **direction field** — one dominant axis per sampled voxel — not integrated
  streamlines. Reads as the standard principal-stress "hedgehog"/trajectory overlay,
  not curved flow tubes.
- Shows only the **single dominant axis** per voxel, not separate tension and
  compression trajectory families (the sign is captured in the data as
  `LoadPathGlyph.isTensile` for a future pass).
- Colour is the von Mises magnitude on the **shared M7.viz.1 scale**, so a hot path
  reads red exactly as the heatmap does.

A true directional stress tensor from `/core/` would remove the isotropy caveat; that
is a future core field (the disp→viz.3 pattern), deliberately NOT added here. The
on-screen key states "Estimated from the deflection field" so the caveat travels with
the overlay.

### Pure, headlessly-tested derivation — `LoadPath.swift` (new)
- **`SymmetricEigen.decompose`** — cyclic Jacobi eigensolver for a symmetric 3×3
  (Double internally), returns `(value, unit vector)` pairs sorted by descending
  eigenvalue.
- **`LoadPathField.strainTensor`** — ε at a voxel centre from the 8 corner nodal
  displacements (trilinear element-centre gradient, per-axis mean forward difference).
- **`LoadPathField.glyph` / `build`** — one dominant-principal-direction `LoadPathGlyph`
  per sampled printed voxel; a cube-root stride keeps the glyph count near a budget
  (default 260) for phone legibility; the stress field gates sampling to printed
  material (void nodes carry zero displacement → zero strain → skipped anyway).
- **`LoadPathCopy`** — the user-facing key strings, kept beside the derivation so the
  honesty copy is asserted headlessly.

### Model wiring — `ResultsModel.swift`
- `DisplacementField.node(a,b,c)` — exact node accessor the strain gradient needs.
- `loadPathOn` toggle, `hasLoadPath` (needs a displacement field, same as flex),
  `toggleLoadPath`, cached `selectedLoadPath`, and `loadPathSegments(for:)` — the
  pos+rgba line buffer (two verts/glyph, coloured on the shared stress scale).
- Flex and load-path are **mutually exclusive** (flex moves vertices; load path draws
  at rest), enforced in `toggleFlex`/`toggleLoadPath`.

### Rendering — `MetalMeshView.swift` (device QA)
- Load-path line buffer + `setLoadPath`/`clearLoadPath`; a draw pass reusing the
  existing line/ground pipeline under the **mesh's** model·view·proj (so segments lock
  to the part), depth-tested against the part, no depth write. Cleared on mesh change so
  a stale variant's glyphs can't leak. Threaded `loadPathSegments` through
  `MeshViewInputs` + both platform inits + the coordinator (upload on change, clear when
  off).

### UI — `ResultsScreen.swift` (device QA)
- "Load path" toggle beside Flex/Stress (shown only when `hasLoadPath`). Top-leading key
  panel (shares the flex slot — mutually exclusive — so it never collides with the
  top-centre stress legend). Excluded from the history morph; forces the fully-formed
  final mesh while active. Static overlay ⇒ reduced-motion-safe by construction.
