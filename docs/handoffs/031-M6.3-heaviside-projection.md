## What I did
Implemented the locked projected SIMP formulation and moved Gate V2 to it.

### Projection functions — `core/include/topopt/simp.hpp`, `core/src/simp/simp.cpp`
- `heaviside_project(rho_tilde, beta, eta=0.5)` — the Wang/Lazarov/Sigmund
  (2011) tanh threshold exactly as `formulation_locked.projection` states it.
- `heaviside_project_derivative(...)` — beta * sech^2(beta*(rt-eta)) / (tanh
  (beta*eta) + tanh(beta*(1-eta))).
- Both throw `std::invalid_argument` for beta <= 0 / non-finite or eta outside
  (0, 1).
- `struct ProjectionStage { beta, move, iterations }` and
  `heaviside_continuation_schedule()` returning the fixture-locked schedule:
  beta {1,2,4,8,16,32}, 50 iterations per stage, move 0.2/0.2/0.2/0.2/0.1/0.05.
- `SimpOptions` gains `projection` (vector of stages; EMPTY = disabled =
  default) and `projection_eta` (default 0.5).

### Staged loop — both `simp_optimize` overloads (`src/simp/simp.cpp`)
The loop now iterates over a stage plan. Empty schedule -> a single unprojected
stage using `options.move` / `options.max_iterations`: the pre-M6.3 loop, same
operations in the same order (the untouched simp/mbb/property_v3/variants/
passive_regions/minimize_plastic tests confirm behavior). Non-empty -> per
stage: analysis density = heaviside_project(filter(x), stage beta, eta), OC via
the new projected update, per-stage move limit, change_tol ends a stage early
(re-converging, then continuation proceeds); `converged` reports the last
stage. Final state: `physical_density` = projected filtered design at the LAST
stage's beta, `compliance` from one final solve on it, `volume_fraction` = the
achieved PROJECTED fraction. The masked overload mirrors this with projection
applied to Active voxels before the FrozenSolid->1 / FrozenVoid->0 pins.

### Projected OC update — `oc_update_projected` / `oc_update_masked_projected`
(anonymous namespace; the public M3.3 `oc_update` is untouched). Same bisection
structure, with the three locked differences (benchmarks.json
`formulation_locked` / `oc_guard_locked`):
1. Sensitivities chain the projection derivative BEFORE the filter transpose:
   dc/dx = H^T((dc/drho_bar * drho_bar/drho_tilde)/Hs); the volume sensitivity
   is H^T(drho_bar/drho_tilde / Hs).
2. The volume constraint is enforced on the PROJECTED candidate density.
3. The elementwise OC ratio -dc/(lambda*dv) uses UNCLAMPED dv; a non-finite
   ratio (0/0 — whole filter neighbourhood has zero projection derivative at
   high beta) HOLDS the voxel (Be = 1). dv is never clamped at a floor — the
   fixture documents the measured catastrophic failure of that variant.

  this bite — the FD failures already demonstrate sensitivity.
- **dv-clamp bite** (the fixture's documented catastrophic variant: floor dv
  at 1e-12, drop the 0/0-hold): behaviorally **INERT** in this C++
  implementation — gate_v2 output bit-identical, 72/72. Analysis: with
  dproj = beta/(cosh^2(beta*(rt-eta)) * denom), the derivative's minimum on

  0/0 arises anywhere on the locked schedule, and a 1e-12 floor binds only

  their box bounds. The guard is nevertheless implemented VERBATIM per the
  locked spec (it protects sharper schedules and other derivative numerics
  where dproj can underflow to exact 0). Reported honestly: this bite does

## Warnings for the next run
- **gate_v2 is now a long test:** ~209 s in this container (~3-4x slower than
  the maintainer's Mac per fea_cg/beam timings, so expect roughly 50-60 s
  there; it was 12.9 s in run 030). The locked schedule is 300 iterations per
  benchmark vs the old 60, plus sharper (higher-contrast) systems for the CG
  solver. Local full suite here: ~10.5 min. Comfortably inside CI's 150-min
  budget.
- **Compute-cost data point for the ROADMAP's "measure on-device":** the
  per-iteration cost of a projected iteration is essentially the unprojected
  cost (projection itself is negligible next to the CG solve). Total cost
  therefore scales as (300 locked iterations) / (the unprojected iteration
  budget being replaced): ~5x vs the old 60-iteration gate, ~10x vs the demo
  job's 30-iteration SIMP budget. The ROADMAP's "1.5-3x" reads as a
  100-200-iteration unprojected baseline; on-device measurement (M7) should
  budget with the 300-iteration schedule in mind.
- **Projection is opt-in everywhere.** Nothing in production sets
  `SimpOptions.projection` yet. `minimize_plastic` forwards `options.simp`
  verbatim, so `MinimizePlasticOptions.simp.projection =
  heaviside_continuation_schedule()` is the entire adoption path for the app —
  whether/when to flip it (and the CLI job schema's stance, which is
  maintainer-owned per DECISIONS 2026-07-09) is a maintainer decision.
- **The cantilever limit-cycle phase matched the reference** (stage-32
  pre-update sample ~91.5, final post-update analysis ~40.3). If a future
  solver/ordering change flips the phase, the final-compliance BAND still
  gates it; per the fixture's limit_cycle_note, extend the final stage by 50
  iterations and re-evaluate before suspecting the physics.
- **Stage-end sampling convention:** the reference's `stage_end_compliance`
  for stage k is the last PRE-update analysis of that stage =
  `history[50k-1].compliance`; `final_analysis_compliance` is the post-loop
  extra solve = `result.compliance`. Do not "fix" the off-by-one.
- **The 0/0-hold OC guard is dormant on the locked schedule** (see the
  dv-clamp bite note): at beta <= 32 the tanh-form derivative never reaches
  exact 0, so the hold never fires today. It is locked
  (benchmarks.json oc_guard_locked) and load-bearing for sharper betas /
  different derivative numerics — do not simplify it away, and never clamp dv.
- **`SimpOptions.move` / `max_iterations` are ignored but still validated**
  when a projection schedule is set (the per-stage values govern) — a caller
  must still pass valid values for them (defaults are fine).
- **`converged` semantics with a schedule:** each stage may stop early on
  change_tol; the flag reports whether the LAST stage did. change_tol = 0
  reproduces the reference's exactly-50-per-stage trajectory and is what
  gate_v2 uses.
