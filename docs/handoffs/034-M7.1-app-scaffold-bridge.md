### Architecture
- **`TopOptBridge`** (C++): an interoperability-clean facade over the core.
  `Sources/TopOptBridge/include/TopOptBridge.hpp` exposes ONLY std::string /
  std::vector / POD structs — never a core, OCCT or Eigen type — so the Swift
  C++ importer parses it cleanly (ARCHITECTURE §3: OCCT is an implementation
  detail, not in the public surface). All heavy includes live in `bridge.cpp`,
  which catches core exceptions and returns them as `BridgeError` (nothing
  throws across the language boundary).
- **`TopOptKit`** (Swift, C++ interop): the idiomatic wrapper the app/tests
  call. Converts bridge results to Swift value types, turns `BridgeError` into
  thrown `TopOptError`, and bridges the M7.0a progress callback (Swift closure →
  `@convention(c)` trampoline via an opaque context box) and cancellation
  (closure returns `false` → sets a bool the bridge mirrors into the driver's
  atomic).
- Both are a **local SwiftPM package** (`app/TopOptKit/Package.swift`) so the
  C++-interop build settings and `xcodebuild test` are reliable and machine-
  independent. The **Xcode project** `app/TopOpt.xcodeproj` (iPad SwiftUI app
  target "TopOpt") references the package as a local dependency.

### Core built into the app via CMake — `app/scripts/build_core.sh`
Runs CMake to build the core as a static library (`libtopopt.a`, macOS host
slice) and vendors it plus the OCCT headers/libs under
`app/TopOptKit/vendor/` via package-relative symlinks, so `Package.swift`
carries no machine-specific absolute paths. OCCT is dynamically linked (LGPL
2.1, ARCHITECTURE §4/§10). lib3mf is not needed for M7.1 (STL export is pure
C++; 3MF export is M7.9). `vendor/` and the CMake build tree are git-ignored
(`app/.gitignore`) — they are regenerated, never committed. iOS device slice =
maintainer step (documented in build_core.sh + app/README.md).

### Wrapper surface (all of the M7.1 list) — `Sources/TopOptKit/TopOptKit.swift`
`loadMaterials(path:)`, `importMesh(path:linearDeflectionMM:)` (STL + STEP by
extension, STEP carries per-triangle face ids for M7.5), `voxelize(meshPath:
resolution:)`, `tagStepFace(stepPath:faceID:asFixture:resolution:)`,
`minimizePlastic(... progress:)` (M5.3 driver with M7.0a progress +
cancellation; imports → voxelizes → tags the min-x boundary as the Fixture
mounting face → clamps its nodes → walks the self-weight ladder), `exportSTL
(mesh:to:)`, and `smoke(materialsPath:meshPath:)` — the M7.1 smoke summary
(material count + imported triangle count) shared by the app screen and the
tests.

### App shell — `app/TopOpt/`
`TopOptApp.swift` (@main App) + `ContentView.swift` (the M7.1 bridge smoke
screen: core version + material count + imported-mesh triangle count via
`TopOptKit.smoke`). Bundles `Resources/sample_cube.stl` (a freshly-authored
cube, not a core fixture) and the canonical `core/src/materials/materials.json`
(referenced, not duplicated) as app resources. Minimal `Assets.xcassets`. The
project uses a file-system–synchronized group for `TopOpt/`, a
`XCLocalSwiftPackageReference` to `TopOptKit`, and a shared `TopOpt` scheme.
This is throwaway smoke UI — the design-system screens are M7.2–M7.9.

### Tests (written first) — `Tests/TopOptKitTests/TopOptKitTests.swift`
16 XCTest cases exercising the whole wrapper against the committed /core/
fixtures: core version; materials (count 12, PLA E/family, resin family, sorted
order, missing-file throw); STL import (cube → 12 tris / 8 verts / watertight,

## Warnings for the next run
- **Run `./app/scripts/build_core.sh` before any `swift test` / `xcodebuild
  test` / app build** (and after any /core/ change). It regenerates
  `app/TopOptKit/vendor/` (the static lib + OCCT symlinks). Without it the
  package fails to link (`library not found for -ltopopt`).
- **`Package.swift` links OCCT with `.unsafeFlags`.** That is allowed only
  because TopOptKit is a *local* package; do not turn it into a versioned/remote
  dependency without replacing the unsafeFlags (e.g. with a binaryTarget
  xcframework).
- **The bridge header is the interop contract — keep it POD-only.** Do not add a
  core/OCCT/Eigen type (or a raw C++ template) to `TopOptBridge.hpp`, or the
  Swift C++ importer will start dragging heavy headers (and their macros) into
  the Swift module. New capabilities go as new POD-in/POD-out functions.
- **Cancellation latency:** a cancel raised from the progress closure is honored
  at the *start of the next* OC iteration (the bridge relays progress first,
  then mirrors the flag into the driver's atomic). The `bool*` is read on the
  optimizing thread; the tests flip it from within the progress callback (same
  thread) to stay race-free. A UI Cancel button that flips it from the main
  thread while the optimizer runs on a background queue is a benign data race on
  a single bool — M7.7 should use an atomic/actor-guarded flag if it wants
  strict correctness.
- **iOS slice is the real remaining work for a device build:** OCCT (and, for
  M7.9, lib3mf) built for `arm64-apple-ios`, fed to CMake with an iOS toolchain,
  with `vendor/` repointed or an `.xcframework` packaged. Until then the iPad
  target links only against the macOS core slice (i.e. it does not link on iOS).
- **`run_minimize_plastic` uses resolution as given** and clamps the whole min-x
  node plane. On a part whose min-x grid face has no solid voxels there would be
  no Fixture voxels (V3 retention vacuous); the cube fixture fills its grid, so
  the tests are unaffected, but the M7.5/M7.6 real tagging path must set
  Fixture/Load groups explicitly (the app already has `tagStepFace`).
- **Benign linker warnings:** `ld: warning: ... built for newer version 26.0`
  come from Homebrew OCCT being built for macOS 26 while the package deployment
  target is lower; harmless, filtered from the pasted evidence.

## Addendum (same run): TopOptBridge module build fix

**Symptom (maintainer's Mac):** opening `app/TopOpt.xcodeproj` (⌘B) and/or
`xcodebuild test` on the TopOptKit package failed with
```
Clang dependency scanning failure: While building module 'TopOptBridge' ...
  .../TopOptBridge/include/TopOptBridge.hpp:15:10: fatal error: 'cstdint' file not found
fatal error: could not build module 'TopOptBridge'
Unable to resolve module dependency: 'TopOptBridge'
```
even though the same package built green in the original run's environment.

**Root cause (confirmed, not assumed).** SwiftPM auto-generated the module map
for the C++ facade target with **no language requirement**:
```
module TopOptBridge {
    umbrella "…/Sources/TopOptBridge/include"
    export *
}
```
Nothing in that module map states the module is C++. `TopOptBridge.hpp`
correctly includes `<cstdint>`/`<string>`/`<vector>` directly (the header is
not the problem), but with no `requires cplusplus` a clean Clang dependency
scan is free to build the module in **C** mode, where the C++-only `<cstdint>`
is not on the include path → "file not found". Whether the scan picks C or C++
depends on the exact Xcode/clang invocation, so it built on the original
machine and failed on the maintainer's — a target-configuration problem, not a
code problem, exactly as diagnosed.

**Fix (smallest, build-environment-independent).** Added an explicit module map
that pins the module to C++ for every consumer and every build system:
`app/TopOptKit/Sources/TopOptBridge/include/module.modulemap`
```
module TopOptBridge {
    umbrella "."
    requires cplusplus
    export *
}
```
`requires cplusplus` forces the module to be built as C++ in all contexts
(Xcode building the app target, `xcodebuild test`, and `swift build`/`swift
test`). SwiftPM now hands this map to the compiler instead of auto-generating
one; the Swift consumers already import it in C++ mode
(`interoperabilityMode(.Cxx)` on TopOptKit + TopOptKitTests →
`-cxx-interoperability-mode=default`), which satisfies the requirement. No
change to the header (kept POD-only), `Package.swift`, the wrapper, or any
core/fixture file. This is the only file added by the addendum.

**Verified here (both toolchain entry points, clean module cache):**
`swift build -v` shows the compiler invoked with
`-Xcc -fmodule-map-file=.../Sources/TopOptBridge/include/module.modulemap`
(the source map, not a `.build/`-generated one) and `-cxx-interoperability-mode
=default`. Clean `xcodebuild test -derivedDataPath <fresh>` and `swift test`
both green:
```
Executed 16 tests, with 0 failures (0 unexpected) in 0.285 (0.290) seconds
** TEST SUCCEEDED **
```
The original `cstdint` failure could NOT be reproduced in this environment
(this Xcode scans the module as C++ by default), so the fix is applied on the
strength of the confirmed root cause rather than a local red→green flip.
