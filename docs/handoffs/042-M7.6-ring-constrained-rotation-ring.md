The M7.6-ring feature is the constrained aiming affordance MOD-F1 D4 defers to
"shell v2": a load's direction is normally the Gravity/Push/Pull snap; the ring
lets the user aim a **custom** direction by rotating about ONE fixed axis (a
single degree of freedom), snapping to 15° detents, with a haptic tick on each
detent crossing, and — to reach any direction on the sphere without a freehand
2-DOF drag (the exact failure MOD-F1 P1 rejects) — a **second** orthogonal ring
that appears only after the first commits.

### Tested core (headless — the M7 /app/ verification standard)

**`Sources/TopOptFlows/RotationRing.swift` (new)** — two pure value types:
- `RotationRing`: rotates a unit `base` about a fixed `axis` by one scalar
  `angle`. Init projects `axis` onto the plane ⟂ base (and falls back to an
  arbitrary perpendicular if the supplied axis is parallel/degenerate) so the
  rotation is a genuine single DOF and never NaNs. Exposes `liveDirection`
  (unsnapped drag preview) and `snappedDirection` (committed), plus the static
  detent math used by the gesture layer: `nearestDetentAngle`, `detentIndex`,
  `didCrossDetent` (the haptic-tick edge), and `angleForDrag(center:location:)`
  (a screen drag → the one angle about the ring centre).
- `RingAiming`: the two-stage sequence. `first` is always present; `second` is
  nil until `commitFirst()`, which snaps ring 1 and reveals ring 2 about an axis
  `cross(first.axis, committedDir)` — orthogonal to both, i.e. a new DOF that
  tilts out of ring 1's plane. `commitFirst()` is a no-op once on ring 2 (only
  two rings ever exist). `committedDirection` composes the two rotations.

**`Sources/TopOptFlows/ForceModel.swift` (modified, additive)** — so a committed
ring actually aims a load (otherwise the ring is decorative):
- New `customAim: [UUID: SIMD3<Float>]` — a per-load world-space custom
  direction. `setCustomDirection` (load-only, rejects a degenerate vector),
  `customDirection(for:)`, `clearCustomDirection`.
- `resolvedDirection(for:groupNormal:)` — the custom aim if set, else the snap
  direction. `loadForceVectorNewtons` now routes through it (so M7.7 / the solver
  traction path gets the aimed direction). Existing callers pass no custom aim, so
  behaviour is unchanged for them.
- `setDirection` (choosing a snap preset) clears the custom aim — "the last
  constrained affordance chosen wins" (D4). `sync`/`clearKind` prune
  `customAim` alongside `kinds`. `panelKindLabel` shows "· custom" for an
  aimed load.

### Device-QA UI (compiles on all three slices; pixels + haptics are maintainer QA)
- **`Sources/TopOptFlows/RotationRingGizmo.swift` (new)** — a SwiftUI overlay
  drawing the ring (+ the dimmed flattened ring 2 once on the second stage) with
  24 detent ticks at the arrow base, a drag constrained to `angleForDrag` (its
  distance from centre ignored — never a 2-DOF vector), a light haptic
  `impactOccurred()` on each `didCrossDetent` edge (guarded `#if canImport(UIKit)`
  so macOS builds the same view without it), and Set/Cancel controls.
- **`Sources/TopOptFlows/WorkspacePlaceholder.swift` (modified)** — added an
  "Aim" pill to the active load's snap row (highlights when a custom aim is set;
  snaps de-highlight while custom is active), presents `RotationRingGizmo` at the
  active load's projected centroid, and the begin/update/commit/cancel handlers
  (Cancel restores the pre-aim direction). The in-scene arrow now renders
  `resolvedDirection` so it reflects the aim live.

## What I did NOT do
- **Did NOT verify any pixel/haptic on-device.** The ring drawing, the detent
  ticks, the drag feel, the light-impact haptic on each detent, the ring-2 reveal,
  and the arrow live-tracking the aim are compiled + logic-tested but visually
  unverified (device QA, the M7 /app/ gate).
- **The custom aim is stored in WORLD space, not model space.** Like the
  gravity/push/pull snaps it is resolved at the current camera/settle state, but
  unlike them (which recompute from the world normal each frame) a stored world
  vector does NOT re-rotate if the user later re-settles gravity ("Change"). Re-
  aiming after a re-settle is a v2 refinement; the snap directions remain
  settle-tracking. Noted so a later run stores model-space if that matters.
- **Did NOT map the screen drag through the ring's true projected plane.** The
  gizmo maps the pointer's screen angle about the ring centre directly to the ring
  rotation angle (a usable constrained control); an exact ellipse-plane projection
  is a device-QA feel refinement.
- **Did NOT plumb the aimed direction into an actual solve.** `loadForceVectorNewtons`
  now returns the aimed vector, but running the optimizer with it (traction on a
  background queue) is still M7.7. `LoadCaseTagger` is untouched.
- **Did NOT persist a custom aim across imports/launches** (per-workspace `@State`,
  same as all M7.5/M7.6 selection state).
- **Did NOT change the snap model (`LoadDirection`)** — it stays the 3 presets;
  the ring is a separate override channel, so no existing `.load(direction:…)`
  construction or test changed.
