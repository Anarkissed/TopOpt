# Handoff 062 — M7.viz.6: Failure-load prediction ("push it till it breaks")

## Task
M7.viz.6 — show how much load the part carries before it yields, WHERE it fails
first, and an optional "push" scrub that drives the existing flex animation toward
failure; a pure derivation from already-computed data (peak von Mises + material
yield + applied load), labelled as a solid-print estimate with an infill-adjusted
value when an infill % is set.

## What I did
Pure derivation, no new physics / no solve / no optimizer call (linear FEA ⇒ stress
scales linearly with load):
`multiplier = yield / peak_von_mises`, `failure_load = multiplier × applied_load`,
`failure_location = the M7.viz.2 hot spot` (reused: same peak point / position / index).

Key design choices + WHY:
- **Applied-load magnitude is app data, not a core field** — it's the user-declared
  load case (`ForceModel.totalLoadKg`, kgf). Threading it workspace → ResultsScreen →
  ResultsModel is pure app plumbing, so the interface-freeze "STOP if the magnitude
  isn't available to the results screen" did NOT trigger (it IS available). It is 0 for
  STL/self-weight runs (no declared load), which suppresses the prediction entirely.
- **Units** follow the existing `WeightUnit` kg/lbs toggle (the app has no Newton
  toggle; kgf is the native store).
- **Push scrub is proportional + bounded**: on-screen exaggeration =
  `maxExaggeration × pushFactor/multiplier`, so at failure the deflection reaches the
  legible max and a lightly-loaded part honestly barely moves at 1× — the "feels dead"
  case this feature answers. It is a static scrub (position → deflection), hence
  reduced-motion-safe with no loop.
- **Honesty**: the number is always labelled a solid-print estimate; when infill < 100%
  is set, an infill-adjusted value is shown using the SAME `f^1.5` family as the
  M7.infill-margin ladder (mirrored in `FailureLoad.infillKnockdown`, pinned by a test)
  so the two surfaces agree. No invented curve.

## What I did NOT do
- Did NOT check the ROADMAP box (maintainer checks at merge, per the concurrent-mode
  rule).
- Did NOT run `xcodebuild test` on a device or the iOS simulator, and there is no PR/CI
  URL — I built the macOS core slice locally and ran `swift test` on macOS. The
  animation *feel*, marker placement/legibility, card layout, and the red "Yields
  here" transition are deferred to maintainer device QA (the `/app/` standard).
- Did NOT add a Newton unit — matched the existing kg/lbs toggle only.
- Multi-load-group handling uses the SUM of load-group weights as the single applied
  scalar (matches `ForceModel.totalLoadKg`); I did not model per-group directional
  resultants (linear scaling makes the total the intuitive "load the user dialed in",
  and single-load is the common case).
- Did NOT change the displayed hot-spot/margin values or the stress field — the
  knockdown affects only the failure-load estimate, exactly like core keeps the
  displayed margin solid.

## Warnings for the next run
- **The `f^1.5` infill knockdown now lives in TWO places** — core
  `minimize_plastic.cpp infill_margin_knockdown()` and app
  `ResultsModel.FailureLoad.infillKnockdown`. If the maintainer tunes the core exponent
  (the seed comment invites raising p toward 2 / adding a wall-count floor), update
  `FailureLoad.knockdownExponent` too, or the two honesty surfaces disagree.
  `testInfillKnockdownMatchesCoreCurve` pins the app copy to the current curve and will
  fail if only core moves.
- The failure prediction is suppressed (nil) for STL/self-weight runs (no declared
  load) — that is intended, not a bug. The "Failure" toggle simply does not appear.
- `pushFlexScale()` is guarded on `hasFlex` as well as the prediction, so it returns 0
  when there's no displacement field even though `failurePrediction` is non-nil — the
  number+marker still show; only the scrub is withheld.
- The two pre-existing build warnings (`PlaybackVideo.swift`, `RunModel.swift`,
  MainActor/Sendable) are not mine; this change introduces no new warnings.
