## What I did
- Added public API `core/include/topopt/materials.hpp`: a `Material` struct
  (the six locked §6 fields), a `MaterialError` exception, a
  `MaterialLibrary = std::map<std::string, Material>` alias, and two entry
  points — `parse_materials(std::string json_text)` and
  `load_materials_file(std::string path)`.
- Implemented both in `core/src/materials/materials.cpp`, added to the
  `topopt` library target.
- **No JSON dependency.** ARCHITECTURE §4 locks the dependency set to
  OCCT / Eigen / lib3mf — none provides a JSON parser, and §8 forbids new
  deps. materials.json is small and flat, so the .cpp contains a purpose-built
  recursive-descent JSON parser (objects/arrays/strings/numbers/bool/null,
  standard string escapes incl. `\uXXXX`→UTF-8) that *rejects* malformed input
  rather than repairing it. It is kept internal (anonymous namespace inside the
  materials TU) — no speculative public "json" module (§8).
- Validation enforced exactly the §6 rules and nothing more (so it will not
  reject the human-seeded numeric values later):
  - top level must be an object of `{ name: material_object }`;
  - each material must have **exactly** the six fields — missing → reject,
    unknown → reject, duplicated key → reject;
  - numeric fields must be JSON numbers; `family` must be a string and one of
    `"fdm"` / `"resin"`;
  - `z_knockdown` ∈ (0, 1];
  - `family == "resin"` ⇒ `z_knockdown == 1.0`.
  I deliberately did **not** add extra physical-range checks (e.g. poisson
  bounds, positive modulus) — §6 does not mandate them, and inventing bounds
  risks rejecting the maintainer's future seeded values.
- Tests: `core/tests/unit/test_materials.cpp`, registered as ctest
  `materials_loader`. Self-contained harness (no test framework — that would be
  a new dependency); a `CHECK` macro counts failures and main() returns
  non-zero on any. Exercises the public API only.
- `core/CMakeLists.txt`: added `materials.cpp` to the library and the test
  executable + `add_test`.
- CI (`.github/workflows/ci.yml`): flipped the ctest step from
  `--no-tests=ignore` to `--no-tests=error`. Handoffs 001/002 explicitly
  assigned this flip to "the first task that adds a real test" (M1.2). It is a
  strictly stronger gate (a zero-tests build now fails), never a weakening.
  `ci/ci.yml` is a symlink to this file and follows automatically (verified).
- Checked the M1.2 box in ROADMAP.md — the only ROADMAP edit.

## Warnings for the next run
- The JSON parser is intentionally minimal and **materials-scoped**, living in
  an anonymous namespace inside `materials.cpp`. It is not a general public
  JSON facility. If a later task (e.g. M5.2 job report, M6.2 CLI) needs JSON,
  decide deliberately whether to promote/replace it — do not assume it is a
  supported shared API.
- `--no-tests=error` is now in CI. Every future configure must register at
  least one test or the Test step fails. Adding a module without wiring its
  test into CMake will now go red (that is the point).
- The parser accepts standard JSON only (no comments, no trailing commas). If
  the human-authored `materials.json` is hand-edited with a trailing comma it
  will be rejected with a byte-offset error message — that is intended strict
  behavior, not a bug.
- `z_knockdown == 1.0` and the resin equality check compare doubles to exactly
  1.0. JSON `1.0` / `1` both parse (via `std::stod`) to exactly `1.0`, so the
  comparison is safe for the literal the schema expects; do not "fix" it with a
  tolerance without reason.
- Local `TOPOPT_REQUIRE_DEPS=OFF` still proves nothing about dependency
  resolution — only CI does. This task links nothing new, so risk is low.
