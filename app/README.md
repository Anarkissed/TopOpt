# TopOpt iPad app (`/app/`)

The SwiftUI + Metal iPad shell over the platform-agnostic C++ core in `/core/`
(ARCHITECTURE.md §3/§4). It is deliberately thin: all geometry, FEA, SIMP and
export logic lives in the core; the app is a front end over the **TopOptKit**
Swift/C++ bridge.

> `/app/` is **not** covered by the Linux CI that gates `/core/`. Its
> verification standard (ROADMAP M7 rules) is `xcodebuild test` on the
> maintainer's Mac plus device QA.

## Layout

```
app/
  TopOpt.xcodeproj/        # iPad SwiftUI app target "TopOpt" (references the package below)
  TopOpt/                  # app sources (thin shell)
    TopOptApp.swift
    ContentView.swift      # M7.3 flow host (embeds TopOptFlows.RootView)
    Assets.xcassets/
    Resources/sample_cube.stl
  TopOptKit/               # Swift package: the bridge + Swift wrapper + tests
    Package.swift
    Sources/
      TopOptBridge/        # interop-clean C++ facade over the core (bridge.cpp)
        include/TopOptBridge.hpp
      TopOptKit/           # idiomatic Swift wrapper (the app-facing API)
      TopOptDesign/        # M7.2 SwiftUI design system: tokens + reusable glass views
      TopOptFlows/         # M7.3 home + import flow: AppModel + Home/Import/Workspace screens
    Tests/TopOptKitTests/     # headless macOS unit tests (xcodebuild test)
    Tests/TopOptDesignTests/  # headless design-token tests (M7.2)
    Tests/TopOptFlowsTests/   # headless flow-logic tests (M7.3 AppModel)
    vendor/                # generated by scripts/build_core.sh (git-ignored)
  scripts/build_core.sh    # CMake build of the core static lib + vendor step
```

## Building the core into the app (CMake)

The core is compiled into a static library by CMake and vendored under the
package. Run once (and after any `/core/` change) before building or testing:

```sh
./app/scripts/build_core.sh
```

This produces `app/TopOptKit/vendor/TopOptCore.xcframework` — a multi-platform
static-library xcframework with three slices, so Xcode links the right one per
platform (a single static `.a` cannot: iOS device and simulator are distinct
platforms):

| Slice | Arch | OpenCASCADE | Notes |
|-------|------|-------------|-------|
| `macos-arm64` | arm64 | yes | full pipeline incl. STEP import; runs the tests |
| `ios-arm64_x86_64-simulator` | arm64 + x86_64 | no | OCCT-free |
| `ios-arm64` (device) | arm64 | no | OCCT-free |

OpenCASCADE is not available for iOS (Homebrew ships macOS only), so the iOS
slices are **OCCT-free**: STEP import and STEP face-tagging are macOS-only and
return a clear "not available on this platform" error on iOS. Everything else —
materials.json, STL import/export, voxelize, FEA, SIMP, `minimize_plastic` —
works on all platforms (Eigen is header-only and compiled into every slice).
OpenCASCADE is dynamically linked on macOS only (LGPL 2.1, ARCHITECTURE §4/§10);
lib3mf is not required for M7.1 (3MF export is M7.9 — STL export is pure C++).
The script vendors only package-relative paths, so `Package.swift` carries no
machine-specific absolute paths.

## Testing the bridge (the M7 verification standard)

```sh
./app/scripts/build_core.sh
cd app/TopOptKit
xcodebuild test -scheme TopOptKit-Package -destination 'platform=macOS'
# or, equivalently, for a quicker local loop:
swift test
```

The `TopOptKitTests` target exercises the whole wrapper surface headlessly
against the committed `/core/` fixtures: load materials.json, import STL/STEP,
voxelize, tag STEP faces, run `minimize_plastic` (with M7.0a progress +
cancellation), and STL export round-trip. The `TopOptDesignTests` target pins
the M7.2 design tokens to the values in `docs/design/TopOpt.dc.html`. The
`TopOptFlowsTests` target drives the M7.3 flow logic (`AppModel`): material
dropdown split by family, STL/STEP import with core diagnostics surfaced on
failure, Continue gating, recent-projects and navigation.

> **Scheme note:** the package vends three products (`TopOptKit`,
> `TopOptDesign`, `TopOptFlows`), so SwiftPM's auto-generated
> **`TopOptKit-Package`** scheme is the one wired for the *test* action (it
> builds all targets and runs all three test targets). The per-product schemes
> are build-only. `swift test` runs every test target regardless of scheme.

## Building / running the iPad app (device QA)

Open `app/TopOpt.xcodeproj` in Xcode and run the **TopOpt** scheme on an iPad
simulator or device. The app depends on the local `TopOptKit` package; run
`build_core.sh` first so the package can link the core xcframework. The app
target has C++ interoperability enabled (`SWIFT_OBJC_INTEROP_MODE = objcxx`),
required because TopOptKit is built with C++ interop.

For a device run, select your development Team under Signing & Capabilities
(the app uses automatic signing). STEP import is unavailable on device/simulator
(no iOS OpenCASCADE) — import STL parts there; STEP works in the macOS tests.
Bringing STEP to iOS would need an `arm64-apple-ios` build of OpenCASCADE fed to
`build_core.sh`'s iOS slices (currently OCCT-disabled) — a separate effort.
